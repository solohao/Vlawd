# 03 · 语音交互运行时
 
## 核心定位
 
Voice Interaction Runtime 不是一个简单的 ASR + TTS 插件。
 
它是 AI 光标的耳机协作中枢，负责：
 
```text
听见用户
识别打断
理解短命令
组织语音反馈
控制 TTS 播放
绑定任务状态
协调光标执行
```
 
如果没有这一层，AI 光标会退化成“语音鼠标”。
 
---
 
## 目标交互形态
 
AI 光标需要的不是普通 voice chatbot，而是：
 
> **任务型准全双工耳机协作系统。**
 
核心要求：
 
```text
用户可以随时说话
AI 正在播报时可以被打断
AI 必须快速停止 TTS
用户短命令必须低延迟生效
AI 只汇报关键任务状态
```
 
---
 
## 模块结构
 
```text
Audio I/O
↓
VAD / Interrupt Detector
↓
ASR / Command Recognizer
↓
Command Router
↓
Interaction Policy
↓
Task Runtime / Action Scheduler
↓
TTS Controller
↓
Headset Output
```
 
---
 
## Audio I/O
 
负责：
 
- 麦克风输入
- 耳机输出
- 音频设备选择
- 回声抑制
- 降噪
- 音量控制
- 播放中断
 
桌面 MVP 可以直接使用本机音频 API。
 
不建议第一版就引入 WebRTC，因为本地链路是：
 
```text
本机耳机 / 麦克风
→ 本机桌面 App
→ 本机执行器
```
 
---
 
## VAD / Interrupt Detector
 
负责判断用户是否在说话，以及是否打断 AI。
 
关键场景：
 
```text
AI 正在说：“我找到了 5 个结果……”
用户说：“等下。”
系统立即停止 TTS
```
 
VAD 不只是录音分段工具，它直接影响“员工是否听话”。
 
必须本地处理的抢占词：
 
```text
停
暂停
等下
不是这个
错了
继续
我来
取消
```
 
这些不能等云端 LLM 返回。
 
---
 
## ASR / Command Recognizer
 
语音识别应分两条路径。
 
### 快速命令路径
 
本地识别，低延迟：
 
```text
点 3
右键
往下
继续
停
打开第二个
```
 
### 复杂任务路径
 
完整转写后交给 LLM：
 
```text
帮我把这些岗位按和我简历的匹配度排序，然后把前三个放到右侧工作区。
```
 
---
 
## Command Router
 
把语音输入路由到不同处理器：
 
```text
interrupt_command
cursor_command
workspace_command
task_goal
confirmation_response
conversation_feedback
```
 
例如：
 
- “停” → interrupt_command
- “点 3” → cursor_command
- “这个也加进去” → workspace_command
- “帮我找远程岗位” → task_goal
- “确认” → confirmation_response
- “嗯” → conversation_feedback
 
---
 
## Interaction Policy
 
这是 Voice Interaction Runtime 最关键的模块。
 
它决定：
 
- AI 什么时候说话
- AI 说多长
- AI 是否需要播报
- AI 是否只在屏幕显示
- AI 是否必须请求确认
- 用户是否正在专注，不应打扰
 
### 必须播报
 
```text
task_started
phase_changed
uncertain_target
risk_confirmation_required
blocked
error_recovered
task_completed
```
 
### 可静默
 
```text
mouse_moved
safe_click_done
scroll_done
normal_waiting
```
 
### 播报风格
 
短、确定、任务化：
 
```text
我打开第二个。
这里需要登录，我暂停。
已保存到求职工作区。
```
 
避免长篇解释。
 
---
 
## TTS Controller
 
TTS 不只是把文本转语音，还要支持：
 
- 短句播报
- 流式播放
- 可中断
- 队列管理
- 优先级
- 多任务前缀
- 情绪 / 语气标签
 
优先级建议：
 
```text
user_interrupt_ack
> risk_confirmation
> blocked/error
> task_completed
> progress_update
> low_priority_summary
```
 
---
 
## 任务型准全双工状态机
 
```text
idle
listening
thinking
speaking
executing
interrupted
waiting_confirmation
```
 
核心转移：
 
```text
speaking + user_speech_detected → interrupted
interrupted → stop_tts → listen
executing + stop_command → pause_executor
waiting_confirmation + confirm → resume_execution
```
 
---
 
## 与 Task Runtime 的绑定
 
每一句语音都应绑定任务上下文：
 
```json
{
  "utterance_id": "utt_001",
  "task_id": "task_job_search",
  "workspace_id": "workspace_1",
  "role": "user",
  "text": "先看第二个",
  "intent": "cursor_command"
}
```
 
AI 的语音反馈也要留痕：
 
```json
{
  "utterance_id": "utt_002",
  "task_id": "task_job_search",
  "role": "assistant",
  "text": "好的，我打开第二个。",
  "trigger": "command_ack"
}
```

## 语音技术路线
 
语音选型不应从“哪个 voice agent 框架最酷”开始，而应从产品本质开始：
 
```text
耳机协作
低延迟打断
任务状态汇报
本地隐私
低成本高频交互
屏幕执行联动
```
 
推荐路线是：
 
> **本地优先的 Hybrid Voice Desktop Agent。**
 
---
 
## 能力分层
 
```text
P0：按住说话 + 本地短反馈
P1：任务型准全双工
P2：多端实时语音入口
P3：端到端全双工 / 多模态语音模型
```
 
### P0：按住说话 + 本地短反馈
 
目标是最快验证“耳机指挥 + 光标执行 + 语音汇报”的爽感。
 
建议能力：
 
- Push-to-talk / 全局快捷键
- 本地 ASR
- 本地 TTS
- 本地识别抢占词
- 复杂任务交给 LLM
- 系统鼠标键盘执行
- 动作日志
 
适合技术：
 
- ASR：SenseVoice、faster-whisper、whisper.cpp、FunASR
- TTS：Supertonic 3、Kokoro、Piper
- 音频：本机系统音频 API
 
这一阶段不需要 LiveKit，也不需要 WebRTC。
 
### P1：任务型准全双工
 
新增能力：
 
- VAD 持续监听
- AI 播报可被打断
- TTS 可中断播放
- 流式 ASR 可选
- 语音确认
- 任务阶段播报
- 本地抢占词优先级
 
参考技术：
 
- Silero VAD
- WebRTC VAD
- 本地 ASR 流式能力
- 可中断 TTS 播放队列
 
### P2：多端实时语音入口
 
当产品需要手机、网页、远程控制电脑时，再考虑 LiveKit / WebRTC。
 
适合场景：
 
- 用户不在电脑前，通过手机耳机指挥电脑
- Web 端语音入口
- 多人或多 Agent 房间
- 企业 IM / 通话系统集成
- 云端 Agent Worker
 
LiveKit Agents 的价值：
 
```text
Room / Job / Worker 模型
WebRTC 音频流
VAD
打断
流式 TTS
Agent 加入通话房间
```
 
但 LiveKit 不应成为本地桌面 MVP 的架构中心。
 
### P3：端到端全双工模型
 
PersonaPlex、RealtimeModel、Qwen-Omni 类模型代表未来方向。
 
优势：
 
- 同时听和说
- 处理打断
- 理解反馈语
- 更自然的接话
- 声音和情绪更统一
 
代价：
 
- 模型重
- GPU 要求高
- 本地部署成本高
- 云服务依赖强
- 不利于独立开发者第一版控制成本
 
因此它适合高端模式、企业部署、本地高性能设备和未来硬件成熟后替换部分 pipeline。
 
---
 
## ASR / TTS 选型
 
### ASR
 
AI 光标 ASR 应优先考虑：
 
- 中文识别
- 低延迟
- 本地运行
- 快捷命令准确率
- 可流式或准流式
 
候选：
 
- SenseVoice：中文体验值得优先验证
- faster-whisper：生态成熟
- whisper.cpp：本地部署友好
- FunASR：中文与工程生态强
 
建议不要把完整自然语言任务和短命令混在同一条链路里处理。
 
### TTS
 
候选：
 
- **Supertonic 3**：约 99M 参数，ONNX Runtime，本地推理，多语言，适合短状态播报和本地隐私。
- **Kokoro / Piper / edge-tts**：适合原型期或兜底。
- **Higgs Audio v3 / SGLang-Omni**：适合后期高质量、流式、可控 TTS。
- **OmniVoice**：适合高级多语言和音色克隆，不建议作为 MVP 默认依赖。

## 推荐技术栈组合
 
### 原型版
 
```text
Desktop Shell：Tauri / Electron
ASR：SenseVoice or faster-whisper
TTS：Supertonic 3 / Kokoro / Piper
VAD：Silero / WebRTC VAD
LLM：云端强模型
Execution：mouse/keyboard + browser extension/CDP
Storage：SQLite
```
 
### 可用版
 
```text
本地快速命令识别
可中断 TTS 播放队列
任务型准全双工状态机
屏幕编号 overlay
单真实光标调度
多 AI 工作区
动作日志与工作流模板
```
 
### 后期高级版
 
```text
LiveKit / WebRTC 多端入口
Higgs / OmniVoice / PersonaPlex 高级语音
端到端全双工模型
手机远程控制桌面
多角色 AI 员工音色
```
 
---
 
## 一句话总结
 
**Voice Interaction Runtime 是 AI 光标从“语音鼠标”升级为“耳机里的 AI 员工”的关键：它管理听、说、打断、汇报、确认和任务上下文绑定；技术路线应先本地、低延迟、可打断、任务化，再逐步接入 LiveKit 和端到端全双工模型。**
