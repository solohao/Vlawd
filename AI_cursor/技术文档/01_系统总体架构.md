# 01 · 系统总体架构
 
## 架构结论
 
AI 光标适合采用：
 
```text
Local-first Desktop App
+ Browser-first Execution
+ Structured Action Protocol
+ Event-sourced Trace
+ Hybrid Local/Cloud Model
+ Voice Interaction Runtime
+ Personalization Layer
+ AI-maintainable Modular Architecture
```
 
对应含义：
 
- **本地优先桌面端**：处理麦克风、耳机、快捷键、鼠标键盘、窗口、文件和日志。
- **浏览器优先执行**：第一阶段打透网页、标签、表单、搜索和多窗口操作。
- **结构化动作协议**：AI 输出动作 JSON，而不是直接输出自然语言或裸坐标。
- **事件日志驱动**：所有操作留痕，未来可回放、复用和沉淀工作流。
- **本地 / 云端混合模型**：本地处理高频简单命令，云端处理复杂规划。
- **耳机协作运行时**：本地处理打断、短命令、TTS 状态播报和任务型准全双工。
- **可监督自配置层**：把用户习惯用语、偏好规则和安全边界转成可确认、可撤销、可留痕的本地配置。
- **面向 AI 可维护架构**：用 DDD bounded context、ports/adapters、稳定协议和插件合同，降低未来 AI / 人类修改代码时的误伤概率。
 
---
 
## 一句话技术定义
 
**AI 光标的技术系统是一套以耳机语音 / 快捷键为入口、以 Voice Interaction Runtime 为协作中枢、以本地快脑 + 云端强脑 + 工作流记忆为推理分层、以屏幕感知为上下文、以结构化动作协议为核心、以鼠标键盘 / 浏览器 / 窗口执行器为手、以任务日志和安全策略为边界的桌面 Agent 执行架构。**

## 核心技术目标
 
### 1. 可见执行
 
AI 不只输出文本，而要驱动真实桌面动作：
 
- 移动光标
- 点击
- 右键
- 双击
- 拖拽
- 滚动
- 输入
- 快捷键
- 窗口 / 标签操作
 
### 2. 实时交互
 
高频指令必须低延迟：
 
- 停止
- 暂停
- 继续
- 点编号
- 滚动
- 返回
 
耳机反馈必须可中断：
 
- AI 播报时用户可打断
- 停止指令本地抢占
- 高风险动作语音确认
- 关键进度短句汇报
 
### 3. 可监督安全
 
系统必须支持：
 
- 动作前目标高亮
- 高风险确认
- 可暂停 / 继续 / 取消
- 执行日志
- 回退和错误恢复
 
### 4. 低成本运行
 
高频、低风险、可验证动作本地化，复杂推理才调用云端强模型。
 
```text
本地快脑：ASR / TTS / VAD / 快捷命令 / 元素定位 / 动作校验 / 工作流执行
云端强脑：复杂规划 / 长上下文理解 / 写作 / 困难失败恢复
工作流记忆：成功路径沉淀后，下次尽量本地复用
```
 
### 5. 工作流沉淀
 
每一次执行都要留痕，为未来从“临时指挥”升级为“可复用工作流”打基础。

## 分层架构
 
### 1. Interface Layer
 
面向用户的入口层：
 
- 全局快捷键
- 语音入口
- 耳机输入 / 输出
- 悬浮状态窗
- 屏幕标号 overlay
- 任务面板
- 执行日志面板
- 设置与偏好面板
 
职责：
 
- 接收指令
- 展示 AI 将要操作的目标
- 显示执行状态
- 允许用户暂停、继续、纠错和接管
 
### 2. Voice Interaction Runtime
 
负责实时耳机协作：
 
- VAD
- ASR
- 快速命令识别
- TTS
- 音频设备管理
- 指令流式分段
- TTS 可中断播放
- 任务对讲机策略
 
关键要求：
 
```text
停 / 暂停 / 继续 / 点编号
```
 
这类指令不能完全依赖云端模型。
 
### 3. Context & Observation Layer
 
负责理解当前电脑状态：
 
```text
屏幕截图
OCR
当前窗口
鼠标位置
DOM
Accessibility Tree
浏览器标签
剪贴板
文件上下文
```
 
输出不是原始截图，而是可供模型、规则和执行器使用的候选目标列表。
 
### 4. Planning Layer
 
负责：
 
- 理解用户意图
- 选择下一步
- 拆解长任务
- 判断是否需要用户确认
- 根据观察结果修正计划
 
它可以调用强模型，但必须被动作协议约束。
 
### 5. Action Layer
 
负责把计划变成动作：
 
```text
mouse.move
mouse.click
mouse.drag
mouse.scroll
keyboard.type
keyboard.hotkey
browser.open_tab
browser.group_tabs
window.arrange
clipboard.copy
```
 
所有动作都要带：
 
- 目标
- 参数
- 风险等级
- 预期结果
- 是否需要用户确认
 
### 6. Execution Layer
 
负责真实执行：
 
- 系统鼠标事件
- 系统键盘事件
- 浏览器扩展 API
- DevTools Protocol
- 操作系统窗口 API
- 剪贴板 API
- 文件系统 API
 
执行器必须支持：
 
```text
pause
cancel
rollback_if_possible
handoff_to_user
```
 
### 7. Trace & Workflow Layer
 
负责事件记录和流程沉淀。
 
每一步都记录：
 
```text
用户指令
观察状态
AI 决策
动作参数
执行结果
用户纠错
时间戳
风险等级
```
 
未来可以从日志生成工作流模板。
 
### 8. Personalization Layer
 
负责让 AI 光标逐步学习用户习惯，但不允许静默自我改造。
 
它维护：
 
```text
Preference
PhraseAlias
BehaviorRule
WorkflowShortcut
SafetyPolicyOverride
WorkspacePreset
InteractionStyle
```
 
所有配置修改都应走：
 
```text
preference.propose_update
→ risk_classification
→ user_confirmation
→ preference.apply_update
→ trace
```
 
### 9. Architecture Governance Layer
 
负责保证长期扩展不侵蚀核心：
 
```text
bounded contexts
→ public module APIs
→ versioned contracts
→ ports/adapters
→ plugin contracts
→ ADR
→ AI-readable module README
```
 
核心约束：
 
```text
domain 不依赖 adapters
UI 不持有业务规则
模型不直接调用 executor
插件不直接写核心状态
偏好规则不绕过 Safety Policy
```

## 本地与云端分工原则
 
### 本地处理
 
- 唤醒 / 快捷键
- 音频采集
- 本地 ASR
- TTS
- 快速命令识别
- 鼠标键盘执行
- 屏幕截图
- OCR 可选
- 日志存储
- 简单规则引擎
- 偏好和工作流匹配
- 动作结果快速验证
 
### 云端或强模型处理
 
- 复杂任务拆解
- 长上下文推理
- 写作和总结
- 跨网站任务规划
- 失败恢复建议
- 生成新工作流模板
 
分工原则：
 
```text
高频、低风险、低智能需求 → 本地
低频、高价值、高推理需求 → 强模型
```

## 为什么不是直接训练光标模型
 
直接训练模型输出坐标存在问题：
 
- 坐标随分辨率变化
- 页面布局变化即失效
- 训练数据难收集
- 用户语言目标通常是语义对象
- 可解释性和安全性差
 
更合理路径：
 
```text
先定义动作协议
→ 使用现有模型生成动作
→ 执行器完成坐标映射
→ 收集轨迹数据
→ 后期训练目标识别 / 动作选择模型
```
 
---
 
## 一句话总结
 
**AI 光标的架构核心是“本地执行器 + 屏幕观察 + 动作协议 + 安全策略 + 耳机协作 + 任务留痕”，模型只是其中的决策模块，不应让模型直接裸奔控制电脑。**
