# AI 光标 · 产品技术文档体系总览
 
## 文档体系定位
 
这套文档不再按“讨论发生顺序”组织，而按一个产品从战略到落地的内在逻辑组织。
 
核心对象是：
 
> **耳机协作式 AI 光标：用户在耳机里指挥一个可见、可控、可纠正的 AI 桌面员工，AI 在屏幕上替用户操作电脑，并把成功过程沉淀为工作流。**
 
它不是语音鼠标、不是聊天机器人、不是黑箱 Agent，也不是重做操作系统。
 
它是：
 
```text
耳机任务对讲机
+ 可见桌面执行层
+ 多工作区任务系统
+ 本地快脑 / 云端强脑 / 工作流记忆
+ 可监督自配置能力
```

## 文档重构原则
 
### 原则一：按问题域组织，而不是按讨论顺序组织
 
旧文档的问题是每次讨论产生一个文件，容易出现：
 
- 产品定义、交互、安全、情绪价值交叉重复
- 技术架构、工程架构、模型路线互相穿插
- 新读者不知道先看哪一份
- 后续 AI 修改文档时容易继续堆叠
 
新结构按问题域拆分：
 
```text
战略定位
→ 用户场景
→ 核心体验
→ 协同与个性化
→ 系统架构
→ 执行闭环
→ 语音运行时
→ 模型分层
→ 调度系统
→ 工程治理
→ 验证商业
```
 
### 原则二：产品文档讲“用户价值”，技术文档讲“系统实现”
 
同一概念在两处的职责不同：
 
| 概念 | 产品文档关注 | 技术文档关注 |
|---|---|---|
| 耳机协作 | 为什么带来心理隔离感 | VAD、ASR、TTS、Interrupt、状态机 |
| 可见执行 | 为什么用户敢放手 | Observation、ActionProtocol、Executor |
| 多工作区 | 为什么像多个 AI 员工 | Workspace、TaskRun、CursorLease |
| 自配置 | 为什么形成长期留存 | PreferenceProposal、RiskPolicy、ConfigStore |
| 本地模型 | 为什么降低等待感 | L0-L4 推理分层和路由策略 |
 
### 原则三：保留核心内容，重建章节关系
 
本次重构不是否定旧内容，而是把已讨论出的内容重新归位：
 
```text
产品定义 / 历史演化 → 产品战略与定位
核心场景 / MVP → 目标用户场景与 MVP
交互安全 / 耳机协作 → 核心体验与安全原则
多工作区 / 可监督自配置 → 工作区协同与个性化
运行时架构 / 动作协议 / 感知留痕 → 系统闭环与执行闭环
Voice Runtime / 语音选型 → 语音交互运行时
本地模型路线 → 模型分层与本地推理
AI 可维护架构 → 工程架构与 AI 可维护性
```

## 一句话总结
 
**这套文档现在应被当成一份完整的产品技术蓝图：先说明为什么 AI 光标值得做，再定义第一阶段做什么，随后说明如何安全、低延迟、可监督地实现，最后用工程架构和验证计划保证它能长期迭代。**
