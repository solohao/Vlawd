# 01 · 产品战略与定位
 
## 一句话定义
 
**AI 光标是一个耳机协作式 AI 桌面员工：用户通过耳机或快捷键委托、监督和纠正，AI 在屏幕上可见地移动光标、操作网页、填写表单、整理资料，并把成功过程沉淀为可复用工作流。**
 
更短的对外表达：
 
> 用语音指挥一个可见、可控、可纠正的 AI 光标替你操作电脑。
 
---
 
## 产品本体
 
AI 光标的本体不是聊天框、不是浏览器插件、不是语音输入法，而是覆盖在桌面环境上的执行层和协作通道。
 
```text
用户意图
↓
耳机语音 / 快捷键 / 鼠标指令
↓
AI 理解屏幕与任务
↓
光标 / 键盘 / 浏览器 / 窗口执行
↓
AI 耳机汇报，用户监督、确认和纠正
↓
任务日志与工作流沉淀
```
 
它的核心输出不是一段回答，而是：
 
```text
电脑上的可见动作
+ 执行结果
+ 可复用流程
```
 
---
 
## 解决的核心问题
 
用户每天有大量低价值但不得不做的电脑操作：
 
- 点网页
- 切标签
- 开窗口
- 滚动查找
- 填表
- 复制粘贴
- 搜索资料
- 整理文件
- 生成草稿
- 多页面对照
 
这些任务的共同特点是：
 
```text
不难
但烦
不值得投入创造力
但又不能完全交给黑箱自动化
```
 
AI 光标要解决的是：
 
> **让用户不用亲手做琐碎电脑操作，同时仍然保留监督感和控制权。**
 
---
 
## 用户心理价值
 
传统效率工具强调“更快”，AI 光标还必须强调“更不痛苦”。
 
用户抗拒的不是结果，而是过程：
 
```text
我知道怎么做
但我不想开始点
不想打开一堆页面
不想复制粘贴
不想填表
不想重复查找
```
 
AI 光标把过程变成：
 
```text
我说一句
它去做
我看着
不对我纠正
危险时它问我
```
 
这带来的不是简单“省几秒”，而是：
 
- **老板感**：我在指挥一个员工，而不是亲自点网页。
- **解压感**：烦人任务被 AI 代劳，心理阻力下降。
- **陪伴感**：任务过程中 AI 持续汇报进度，用户知道事情在推进。
- **隔离感**：用户不用亲自进入痛苦任务，只需监督关键节点。
 
---
 
## 历史机会：不是旧式 GUI Agent 复活
 
AI 光标看起来像早期网页 Agent / GUI Agent 的回潮，但本质上是一次螺旋上升。
 
```text
早期：AI 直接看网页、点坐标
↓
中期：行业退回 MCP / API / IDE / CLI 等结构化工具环境
↓
现在：模型能力、视觉理解、动作协议、语音交互和可监督执行成熟后，重新回到桌面
```
 
早期网页 Agent 失败或显得鸡肋，主要因为：
 
- 模型理解弱
- 观察层只靠截图或简陋 DOM
- 动作空间粗糙，常输出裸坐标
- 点完以后缺少验证和恢复
- 用户监督弱，要么全自动黑箱，要么用户一步步手动
- 每一步都调用大模型，慢且贵
 
行业随后退回结构化环境：
 
```text
MCP / API / 搜索工具 / IDE / CLI / 浏览器插件
```
 
这些环境可靠，是因为状态可读、动作可定义、结果可验证、错误可回滚。
 
现在重新回到桌面的机会在于：
 
```text
Screenshot
+ OCR
+ DOM
+ Accessibility Tree
+ Window State
+ Action Protocol
+ Safety Policy
+ Trace
+ Voice Interaction Runtime
```
 
AI 不再只是“看图猜坐标”，而是把桌面包装成一个可观察、可执行、可监督的工具环境。
 
---
 
## 新旧两代“AI 操作网页”的差异
 
| 维度 | 早期 Web Agent | AI 光标 |
|---|---|---|
| 定位 | AI 模仿人点网页 | AI 使用桌面作为可监督工具环境 |
| 感知 | 截图 / DOM 单点能力 | 截图 + OCR + DOM + Accessibility + 窗口状态 |
| 动作 | 坐标点击 | 结构化动作协议 + 执行器 |
| 交互 | 用户等待结果 | 用户耳机协作、随时纠正 |
| 安全 | 黑箱执行 | 可见执行、高风险确认 |
| 成本 | 每步依赖大模型 | 本地高频命令 + 云端复杂推理 |
| 沉淀 | 一次性操作 | 任务日志 → 工作流 |
 
---
 
## 产品边界
 
### 它是
 
- 语音控制的 AI 光标
- 耳机协作式 AI 桌面员工
- 可监督的桌面执行层
- 个人版 RPA / Agent 过渡形态
- 电脑重复操作代劳工具
- 任务执行日志与工作流生成器
 
### 它不是
 
- 不替代操作系统
- 不替代浏览器
- 不做纯聊天机器人
- 不做单纯语音输入法
- 不承诺完全无人值守
- 不做企业级复杂 RPA 编排平台
- 不在第一阶段覆盖所有桌面软件
 
---
 
## 与同类概念的区别
 
| 类型 | 核心承诺 | 主要问题 | AI 光标差异 |
|---|---|---|---|
| AI 操作系统 | 重构系统入口 | 平台难度高，巨头优势大 | 不替代系统，只做执行层 |
| AI 浏览器 | 在浏览器里帮用户浏览和操作 | 易被浏览器厂商竞争，桌面外能力弱 | 浏览器优先，但不止浏览器 |
| 语音输入法 | 把声音变成文字 | 价值停留在输入 | 把意图变成电脑动作 |
| 全自动 Agent | 黑箱完成任务 | 用户不信任、易失控 | 可见、可喊停、可纠正 |
| RPA | 预配置流程自动执行 | 配置门槛高、不灵活 | 语音即时指挥，流程从使用中沉淀 |

## 战略定位
 
不要把 AI 光标定位成：
 
```text
AI 自动操作网页
```
 
这会让用户联想到早期失败的 GUI Agent。
 
更准确的定位是：
 
> **耳机协作式 AI 桌面员工：用户通过耳机委托、监督和纠正，AI 在屏幕上可见地操作电脑，并把成功流程沉淀为工作流。**
 
第一阶段的战略选择：
 
```text
不争 AI OS
不争浏览器入口
不做万能 Agent
先做现有桌面上的可监督执行层
用真实电脑苦活代劳体验建立价值
```
 
---
 
## 一句话总结
 
**AI 光标的本质是“可监督电脑代劳”：它不是追求一上来全自动完成一切，而是让用户以最低摩擦指挥 AI 在电脑上可见地做事，并逐步把成功操作沉淀成工作流。**
