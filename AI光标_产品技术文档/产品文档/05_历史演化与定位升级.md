# 05 · 历史演化与定位升级

## 核心判断

AI 光标看起来像早期“AI 操作网页 / GUI Agent”的回潮，但它不是旧概念的简单复活。

更准确地说，这是一次螺旋上升：

```text
早期：AI 直接看网页、点坐标
↓
中期：行业退回 MCP / API / IDE / CLI 等结构化工具环境
↓
现在：模型能力、视觉理解、动作协议、语音交互和可监督执行成熟后，重新回到桌面与网页
```

第一阶段的问题是：愿景对，但能力和闭环不够。

现在的机会是：用组合式架构把早期愿景重新做成可用产品。

---

## 早期网页 Agent 为什么显得鸡肋

早期 AI 操作网页通常是：

```text
模型看截图 / DOM
↓
猜下一步点击哪里
↓
执行坐标点击
↓
失败后卡住或继续猜
```

它的问题不在“让 AI 操作网页”这个方向本身，而在缺少完整闭环。

### 1. 模型理解弱

早期模型难以稳定理解：

- 页面结构
- 弹窗
- 多步骤流程
- 表单语义
- 当前状态和目标之间的关系

### 2. 观察能力弱

当时通常依赖截图、OCR 或简陋 DOM。

但真实网页很复杂：

- 动态加载
- iframe
- Shadow DOM
- Canvas
- 弹窗遮挡
- 登录态变化
- 响应式布局

AI 不知道自己“看见”的状态是否完整。

### 3. 动作空间粗糙

很多方案直接让模型输出坐标。

```text
click(x=842, y=317)
```

这对窗口大小、分辨率、页面布局极其敏感，也缺少可解释性。

### 4. 反馈闭环弱

点完之后，系统不一定知道：

- 是否点中了
- 页面是否变化
- 是否出现错误
- 是否需要等待
- 是否应该重试或询问用户

### 5. 用户监督弱

早期产品常常在两个极端之间摇摆：

```text
完全让 AI 自己跑
或用户一步步手动点
```

缺少“AI 做、用户看、用户随时纠正”的中间态。

### 6. 成本和延迟高

如果每一步截图理解、规划、执行都调用大模型，体验会慢且贵。

---

## 为什么行业中期退回结构化环境

随着早期 GUI Agent 暴露问题，行业自然收缩到更可靠的环境：

```text
MCP / API / 搜索工具 / IDE / CLI / 浏览器插件
```

这些环境的共同特点是：

- 状态可读
- 动作可定义
- 结果可验证
- 错误可回滚
- 用户监督成本低

Cursor 和 Claude Code 的成功说明：

> **AI 更容易先在结构化、高价值、反馈明确的工作环境中落地。**

VSCode 和命令行不是因为“更酷”，而是因为它们天然给 AI 提供了稳定的工具调用环境。

---

## 现在为什么可以重新回到桌面

现在与早期最大的不同，不是“又有人想让 AI 点网页”，而是底层条件变了。

### 1. 模型能力增强

现在模型更擅长：

- 多步规划
- 工具调用
- 视觉理解
- 长上下文
- 结构化输出
- 错误恢复
- 按约束生成动作

### 2. 观察层可以多源融合

现在可以组合：

```text
Screenshot
+ OCR
+ DOM
+ Accessibility Tree
+ Browser Extension
+ Window State
+ Clipboard
```

AI 不再只能“看图猜坐标”。

### 3. 动作层可以协议化

AI 可以输出：

```json
{
  "action": "mouse.click",
  "target": "element_3",
  "safety": "safe",
  "expected_result": "打开岗位详情页"
}
```

执行器再负责坐标映射、目标高亮、执行和反馈。

### 4. 语音交互可以本地低延迟

ASR、VAD、TTS、本地抢占词、小模型命令识别逐渐成熟，使得：

```text
停 / 继续 / 点 3 / 不是这个
```

这类高频命令可以本地快速响应。

### 5. 用户监督可以成为产品核心

新形态不是黑箱 Agent，而是：

```text
AI 可见执行
用户耳机监督
用户随时打断
高风险动作确认
成功流程留痕
```

这解决了早期用户不信任的问题。

## 新旧两代“AI 操作网页”的根本差异

| 维度 | 早期 Web Agent | AI 光标 |
|---|---|---|
| 定位 | AI 模仿人点网页 | AI 使用桌面作为可监督工具环境 |
| 感知 | 截图/DOM 单点能力 | 截图 + OCR + DOM + Accessibility + 窗口状态 |
| 动作 | 坐标点击 | 结构化动作协议 + 执行器 |
| 交互 | 用户等待结果 | 用户耳机协作、随时纠正 |
| 安全 | 黑箱执行 | 可见执行、高风险确认 |
| 成本 | 每步依赖大模型 | 本地高频命令 + 云端复杂推理 |
| 沉淀 | 一次性操作 | 任务日志 → 工作流 |

---

## 新本质：桌面版工具调用环境

早期 AI 操作网页是：

> **AI 模仿人类操作界面。**

AI 光标应该是：

> **把现有桌面包装成 AI 可观察、可执行、用户可监督的工具环境。**

这意味着产品建设重点不是“让 AI 更像人类鼠标操作员”，而是建立一套桌面 Agent 基础设施：

```text
Observation
→ Action Protocol
→ Executor
→ Voice Interaction
→ Safety Policy
→ Trace
→ Workflow
```

---

## 对产品定位的影响

不要把 AI 光标定位成：

> AI 自动操作网页。

这会让用户联想到早期失败的 GUI Agent。

更准确的定位是：

> **耳机协作式 AI 桌面员工：用户通过耳机委托、监督和纠正，AI 在屏幕上可见地操作电脑，并把成功流程沉淀为工作流。**

---

## 一句话总结

**AI 光标不是旧式网页 Agent 复活，而是把早期 GUI Agent 的直觉、MCP/API/IDE/CLI 的结构化经验、现代多模态模型能力和耳机协作交互组合在一起，形成新的可监督桌面执行层。**
