# 09 · 本地模型与低延迟推理路线

## 核心问题

云端大模型的问题不是不能用，而是它不适合控制 AI 光标的每一个高频动作。

真实交互里，用户要的是：

```text
说“停”立即停
说“点第二个”立即点
AI 播报时能立刻打断
页面变化后能快速判断是否成功
常见任务不用每一步都重新思考
```

如果每一步都走云端大模型：

```text
截图 → 上传 → 大模型思考 → 返回 → 执行
```

体验会被：

- 网络延迟
- 模型排队
- thinking 时间
- token 成本
- 隐私顾虑

共同拖慢。

因此 AI 光标的模型路线应是：

> **本地快脑 + 云端强脑 + 工作流记忆。**

---

## 结论

本地模型可以承担大量高频、低延迟、低风险、可训练的任务。

但第一阶段不应幻想“一个本地通用大模型完全替代云端强模型”。

更现实的路线是：

```text
本地模型：实时交互、短命令、元素定位、动作校验、常见流程
云端模型：复杂理解、长上下文推理、陌生任务规划、困难失败恢复
工作流：把成功路径沉淀后，下次尽量本地执行
```

长期目标不是“永远依赖云端”，而是：

```text
第一次复杂任务可能需要云端
第二次相似任务部分本地
第 N 次常见任务几乎全本地
```

---

## 本地模型适合做什么

### 1. 实时语音层

必须本地优先：

```text
VAD
打断检测
停止词
短命令识别
ASR
TTS
```

原因：

- 延迟要求最高
- 用户对卡顿最敏感
- 隐私敏感
- 高频调用

示例：

```text
“停”
“继续”
“点 3”
“不是这个”
“往下”
“确认”
```

这些不应等云端大模型。

---

### 2. 屏幕元素识别与排序

本地模型可以做：

```text
OCR
元素检测
按钮/输入框/链接分类
可点击目标排序
屏幕编号推荐
```

目标不是完美理解页面，而是把屏幕转成候选元素：

```json
[
  {"id": "element_1", "type": "button", "text": "提交", "risk": "high"},
  {"id": "element_2", "type": "link", "text": "详情", "risk": "safe"}
]
```

然后强模型或规则只需要在候选中选择。

---

### 3. 快速动作选择

对于常见短命令，本地模型/规则可以直接生成动作：

```text
用户：“点第二个”
↓
本地解析：target = element_2
↓
ActionProtocol: mouse.click(element_2)
```

不需要大模型规划。

适合本地化的动作：

- 点击编号
- 滚动
- 返回
- 切标签
- 聚焦输入框
- 复制
- 粘贴
- 暂停任务
- 接管光标

---

### 4. 动作结果验证

执行后，本地模型可以快速判断：

```text
页面是否变化
目标是否出现
是否仍在加载
是否弹出登录
是否出现错误提示
```

示例：

```json
{
  "action_id": "step_001",
  "expected": "打开岗位详情页",
  "local_verdict": "success | uncertain | failed | login_required"
}
```

只有 `uncertain` 或复杂失败才交给云端。

---

### 5. 常见网站与工作流执行

成功流程沉淀后，本地可以执行：

```text
打开网站
读取 DOM
套用工作流模板
填表
筛选
保存
生成结构化记录
```

这类能力更像“个人 RPA + AI 纠错”，不需要每次重新让云端思考。

---

### 6. Personalization 与偏好匹配

本地必须处理：

```text
习惯用语匹配
工作流快捷语
网站默认规则
安全偏好
交互风格
```

例如：

```text
用户：“按老规矩”
↓
本地命中 WorkflowShortcut
↓
加载对应工作流
↓
按 SafetyPolicy 执行
```

---

## 云端模型适合做什么

云端强模型仍然有价值，但应从“每步控制器”降级为“复杂顾问”。

适合云端：

- 陌生任务的首次规划
- 长网页/多文档理解
- 复杂语义判断
- 写作、总结、改写
- 跨网站任务拆解
- 失败恢复策略
- 生成新工作流模板
- 解释为什么某个动作有风险

不适合云端：

- 停止
- 打断
- 点击编号
- 普通滚动
- TTS 短播报
- 每步动作校验
- 本地偏好匹配

---

## 推荐模型分层

```text
L0：规则与状态机
L1：本地小模型
L2：本地 VLM / 小 LLM
L3：云端强模型
L4：沉淀后的工作流
```

### L0：规则与状态机

最快、最可靠。

用于：

- stop
- pause
- resume
- confirm
- cursor lease
- safety blocking
- preference lookup

### L1：本地小模型

用于：

- 短命令分类
- 意图路由
- 元素类型判断
- 简单动作选择
- 打断语义识别

### L2：本地 VLM / 小 LLM

用于：

- 屏幕摘要
- 候选元素排序
- 简单页面理解
- 短计划生成
- 结果验证

### L3：云端强模型

用于：

- 复杂规划
- 长上下文
- 创造性输出
- 复杂失败恢复

### L4：工作流

最长期、最便宜。

用于：

- 复用成功路径
- 用户习惯流程
- 常见网站自动化
- 低成本重复执行

## 运行时决策策略

每个请求先判断是否可以本地完成：

```text
UserCommand
↓
LocalCommandRecognizer
↓
if high_confidence && low_risk:
  local_action
else if known_workflow:
  workflow_execute
else if local_small_model_confident:
  local_plan
else:
  cloud_plan
```

动作后验证：

```text
ActionExecuted
↓
LocalVerifier
↓
success → continue
uncertain → ask_user or cloud_recover
failed → local_retry or cloud_recover
```

## 对云端 thinking 的处理

如果必须调用云端模型，不能让用户干等。

应采用：

```text
本地先响应
云端后台思考
屏幕显示/耳机播报状态
云端返回后再执行复杂步骤
```

示例：

```text
用户：帮我分析这几个岗位哪个适合我。
本地 AI：好的，我先把页面和简历资料整理出来。
本地执行：收集网页文本、截图、DOM、简历片段。
云端后台：做匹配度分析。
AI：分析完成，第二个最匹配，我可以打开详情。
```

这样用户感知到的是“员工正在工作”，不是“系统卡住了”。

---

## 本地模型不是单一大模型

不要把本地化理解成：

```text
在用户电脑上跑一个巨大通用模型，替代所有云端能力
```

更好的本地化是：

```text
多个小模型 + 规则 + 工作流 + 缓存 + 状态机
```

组合示例：

```text
VAD：判断是否说话
ASR：转写用户短句
Command Classifier：识别“点 3”
Element Ranker：找到 element_3
Safety Rule：判断是否安全
Executor：执行点击
Verifier：判断页面是否打开
TTS：播报“我打开第三个”
```

这条链路可以完全本地。

---

## 训练数据来源

AI 光标天然会产生本地训练数据：

```text
用户语音
屏幕状态
候选元素
AI 动作
用户纠错
执行结果
成功工作流
```

但必须注意：

- 默认本地保存
- 用户可删除
- 上传训练必须显式授权
- 隐私字段脱敏
- 高风险动作不能用来自动放宽安全策略

---

## MVP 建议

第一版本地模型优先级：

```text
1. VAD + stop/pause/continue 本地抢占
2. 本地 ASR/TTS
3. 短命令分类器
4. 屏幕元素编号与本地目标选择
5. 动作结果本地 verifier
6. 工作流模板执行器
```

暂不追求：

- 完全本地通用 Agent
- 本地 70B 大模型
- 本地复杂写作
- 本地全站语义规划

---

## 一句话总结

**云端强模型不应控制 AI 光标的每一步；AI 光标应采用“本地快脑 + 云端强脑 + 工作流记忆”的分层推理架构，让停止、打断、短命令、元素定位、动作校验和常见流程尽量本地完成，云端只处理少数复杂规划和困难恢复。**
