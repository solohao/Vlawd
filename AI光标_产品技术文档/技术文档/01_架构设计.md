# 01 · 架构设计

## 架构结论

AI 光标适合采用：

```text
Local-first Desktop App
+ Browser-first Execution
+ Structured Action Protocol
+ Event-sourced Trace
+ Hybrid Local/Cloud Model
```

对应含义：

- **本地优先桌面端**：处理麦克风、耳机、快捷键、鼠标键盘、窗口、文件和日志。
- **浏览器优先执行**：第一阶段打透网页、标签、表单、搜索和多窗口操作。
- **结构化动作协议**：AI 输出动作 JSON，而不是直接输出自然语言或裸坐标。
- **事件日志驱动**：所有操作留痕，未来可回放、复用和沉淀工作流。
- **本地/云端混合模型**：本地处理高频简单命令，云端处理复杂规划。

## 分层架构

### 1. Interface Layer

面向用户的入口层：

- 全局快捷键
- 语音入口
- 耳机输入/输出
- 悬浮状态窗
- 屏幕标号 overlay
- 任务面板
- 执行日志面板

职责：

- 接收指令
- 展示 AI 将要操作的目标
- 显示执行状态
- 允许用户暂停、继续、纠错和接管

### 2. Interaction Runtime

负责实时交互：

- VAD
- ASR
- 快速命令识别
- TTS
- 音频设备管理
- 指令流式分段

关键要求：

```text
停 / 暂停 / 继续 / 点编号
```

这类指令不能完全依赖云端模型。

### 3. Context & Observation Layer

负责理解当前电脑状态：

```text
屏幕截图
OCR
当前窗口
鼠标位置
DOM
Accessibility Tree
浏览器标签
剪贴板
文件上下文
```

输出不是原始截图，而应转成可供模型使用的候选目标列表。

### 4. Planning Layer

负责：

- 理解用户意图
- 选择下一步
- 拆解长任务
- 判断是否需要用户确认
- 根据观察结果修正计划

它可以调用强模型，但必须被动作协议约束。

### 5. Action Layer

负责把计划变成动作：

```text
mouse.move
mouse.click
mouse.drag
mouse.scroll
keyboard.type
keyboard.hotkey
browser.open_tab
browser.group_tabs
window.arrange
clipboard.copy
```

所有动作都要带：

- 目标
- 参数
- 风险等级
- 预期结果
- 是否需要用户确认

### 6. Execution Layer

负责真实执行：

- 系统鼠标事件
- 系统键盘事件
- 浏览器扩展 API
- DevTools Protocol
- 操作系统窗口 API
- 剪贴板 API
- 文件系统 API

执行器必须支持干预：

```text
pause
cancel
rollback_if_possible
handoff_to_user
```

### 7. Trace & Workflow Layer

负责事件记录和流程沉淀。

每一步都记录：

```text
用户指令
观察状态
AI 决策
动作参数
执行结果
用户纠错
时间戳
风险等级
```

未来可以从日志生成工作流模板。

## 事件流

```text
UserCommandReceived
→ ContextObserved
→ IntentClassified
→ PlanGenerated
→ ActionProposed
→ SafetyChecked
→ ActionExecuted
→ ResultObserved
→ StepRecorded
→ NextActionSelected / TaskFinished / UserConfirmationRequired
```

---

## 多任务架构

多任务不能是多个聊天上下文混在一起，而应是任务对象：

```text
TaskRun
 ├─ goal
 ├─ status
 ├─ assigned_window
 ├─ assigned_tabs
 ├─ context_snapshot
 ├─ steps
 ├─ pending_confirmation
 └─ artifacts
```

任务状态：

```text
queued
running
paused
waiting_user
failed
completed
cancelled
```

第一阶段建议先做任务队列，再做真正并行。

## 本地与云端分工

### 本地处理

- 唤醒/快捷键
- 音频采集
- 本地 ASR
- TTS
- 快速命令识别
- 鼠标键盘执行
- 屏幕截图
- OCR 可选
- 日志存储
- 简单规则引擎

### 云端或强模型处理

- 复杂任务拆解
- 网页结构理解
- 长上下文推理
- 写作和总结
- 失败恢复建议

### 分工原则

```text
高频、低风险、低智能需求 → 本地
低频、高价值、高推理需求 → 强模型
```

## 为什么不是直接训练光标模型

直接训练模型输出坐标存在问题：

- 坐标随分辨率变化
- 页面布局变化即失效
- 训练数据难收集
- 用户语言目标通常是语义对象
- 可解释性和安全性差

更合理路径：

```text
先定义动作协议
→ 使用现有模型生成动作
→ 执行器完成坐标映射
→ 收集轨迹数据
→ 后期训练目标识别/动作选择模型
```

---

## 一句话总结

**AI 光标的架构核心是“本地执行器 + 屏幕观察 + 动作协议 + 安全策略 + 任务留痕”，模型只是其中的决策模块，不应让模型直接裸奔控制电脑。**
