# 04 · 多 AI 工作区与光标调度

## 核心架构

多 AI 工作区不等于多个系统光标。

推荐架构：

```text
User Cursor
Real Cursor Executor
Execution Scheduler
Task Runtime A / Virtual Cursor A
Task Runtime B / Virtual Cursor B
Task Runtime C / Virtual Cursor C
Workspace Context Store
Trace Store
```

真实鼠标键盘由 `Real Cursor Executor` 独占控制，多个任务通过 `Execution Scheduler` 申请执行窗口。

---

## 对象模型

### Workspace

```json
{
  "workspace_id": "workspace_job_search",
  "name": "求职分析",
  "type": "browser_and_files",
  "assigned_tabs": ["tab_1", "tab_2"],
  "assigned_files": ["resume.pdf"],
  "drop_zone_enabled": true,
  "status": "active"
}
```

### VirtualCursor

```json
{
  "virtual_cursor_id": "vcursor_1",
  "task_id": "task_1",
  "workspace_id": "workspace_job_search",
  "visual_state": "reading | planning | waiting | requesting_control | executing",
  "last_target": "element_3"
}
```

### CursorLease

真实光标控制权应建模为租约：

```json
{
  "lease_id": "lease_001",
  "owner": "task_1",
  "resource": "real_cursor",
  "expires_in_ms": 15000,
  "scope": "browser_window_1",
  "interruptible": true,
  "granted_at": "2026-06-16T00:00:00Z"
}
```

任务拿到租约后才能执行真实前台鼠标键盘动作。

---

## 调度规则

### 可以并行执行的动作

不需要真实光标租约：

```text
llm.plan
document.summarize
browser.read_dom
file.parse
clipboard.read_when_authorized
workflow.generate_draft
task.wait_user
```

### 必须独占执行的动作

需要真实光标租约：

```text
mouse.move_to
mouse.click
mouse.drag
mouse.scroll
keyboard.type
keyboard.hotkey
window.focus
window.drag
foreground_app.interact
```

### 必须用户确认的租约

以下请求必须询问用户：

```text
接管当前用户正在操作的窗口
执行提交/发送/删除/付款等高风险动作
超过指定时长的连续前台操作
切换用户当前焦点
操作非任务绑定窗口
```

---

## 执行调度器

`Execution Scheduler` 的职责：

1. 接收多个任务的执行请求
2. 判断动作是否需要真实光标
3. 根据优先级排序
4. 申请或授予 `CursorLease`
5. 执行期间监听用户中断
6. 租约到期或任务完成后释放
7. 写入任务日志

建议优先级：

```text
user_interrupt
> safety_confirmation
> active_user_command
> foreground_task
> scheduled_task
> background_task
```

---

## 中断机制

用户中断必须抢占所有 AI：

```text
stop_all
pause_task(task_id)
release_cursor_lease
handoff_to_user
```

当用户移动鼠标或输入键盘时，系统可以进入保护模式：

```text
检测到用户手动操作
↓
暂停 AI 前台执行
↓
释放真实光标租约
↓
询问是否继续
```

---

## 工作区输入与拖拽

每个工作区应有 `DropZone`：

```json
{
  "event": "workspace.input_added",
  "workspace_id": "workspace_job_search",
  "input_type": "url | file | text | screenshot",
  "source": "user_drag",
  "trace_id": "trace_123"
}
```

拖入资料后：

1. 写入 workspace context
2. 触发任务更新
3. 生成可追踪输入节点
4. 允许未来工作流复用该输入类型

---

## 沙盒并行

未来如果需要真正多 AI 同时“操作界面”，应使用隔离环境：

```text
Browser Context A
Browser Context B
Headless Browser C
Virtual Desktop D
Remote Sandbox E
```

这些环境可以有自己的虚拟坐标和执行器，不影响用户主桌面的真实光标。

主桌面只用于：

- 需要用户监督的动作
- 需要登录态的网页
- 需要用户拖拽/确认的任务
- 高风险动作前的可见执行

---

## 状态同步

多个工作区必须避免上下文串扰。

每个任务绑定：

```text
workspace_id
browser_context_id
allowed_windows
allowed_tabs
allowed_files
current_goal
trace_scope
```

AI 不应在未授权情况下跨工作区读取或操作上下文。

---

## MVP 实现建议

第一阶段实现：

```text
Task Queue
Workspace Card
Virtual Cursor State
Single Real Cursor Executor
CursorLease
Stop All
Pause Task
DropZone
Trace per Workspace
```

暂不实现：

- 真多桌面并行
- 多个真实系统指针
- 跨应用复杂并发操作
- 未确认的后台提交

---

## 一句话总结

**多 AI 并行的关键不是突破操作系统只能有一个真实光标的限制，而是把真实光标做成可租用的独占执行资源，让多个 AI 工作区并行准备、排队执行、随时被用户中断。**
