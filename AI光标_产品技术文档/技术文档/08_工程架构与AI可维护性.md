# 08 · 工程架构与 AI 可维护性

## 文档定位

本文只回答一个问题：

> **代码仓库应该如何组织，才能让人类和 AI 编程助手都容易理解、修改、扩展，并且不会因为版本迭代频繁重构。**

它不重复描述运行时链路。

运行时链路见：

```text
01_系统运行时架构.md
```

## 核心目标

AI 光标的代码架构必须同时服务两类维护者：

```text
人类开发者
+ AI 编程助手
```

目标不是把第一版写得最复杂，而是让未来迭代时：

- AI 容易看懂代码边界
- 修改一个功能不容易误伤其他模块
- 新能力可以通过扩展点接入
- 核心协议长期稳定
- 版本升级不需要频繁重构
- 每个模块有清晰输入、输出、状态和责任

一句话：

> **用稳定领域模型和显式边界降低未来 AI/人类共同维护的认知成本。**

---

## 总体架构原则

建议采用：

```text
Modular Monolith first
+ Domain-Driven Design
+ Hexagonal Architecture
+ Event-driven Trace
+ Stable Protocol Contracts
+ Plugin-based Capability Extension
+ AI-readable Codebase Conventions
```

不建议第一阶段直接拆微服务。

原因：

- 产品领域还在快速变化
- 桌面端需要强本地协同
- 过早微服务会增加部署、调试、版本兼容成本
- AI 修改跨服务代码更容易遗漏边界

第一阶段应做“模块化单体”：

```text
一个应用
多个边界清晰的领域模块
模块之间只通过显式接口、事件和协议通信
```

等领域模型稳定后，再按 bounded context 拆服务。

---

## 领域驱动设计：核心域与支撑域

AI 光标不是普通桌面自动化工具，它的核心域是：

> **可监督的桌面任务执行。**

### Core Domain

必须重点保护：

```text
Task Runtime
Action Protocol
Execution Scheduler
Safety Policy
Trace & Workflow
Voice Interaction Runtime
Personalization Layer
```

这些模块定义产品差异化，不应被 UI 框架、模型供应商、浏览器实现污染。

### Supporting Domains

提供能力，但可替换：

```text
Observation
Model Gateway
Browser Adapter
Desktop Adapter
Audio Adapter
Storage Adapter
```

### Generic Domains

尽量使用现成库：

```text
日志
配置文件
数据库迁移
音频编码
HTTP 客户端
窗口枚举
```

## 建议的 Bounded Context

### 1. Task Context

负责任务生命周期。

核心对象：

```text
TaskRun
TaskGoal
TaskStep
TaskStatus
TaskArtifact
TaskWorkspaceBinding
```

不负责：

- 鼠标怎么点
- TTS 怎么播
- 模型怎么调用

### 2. Interaction Context

负责用户与 AI 的实时协作。

核心对象：

```text
Utterance
InteractionIntent
InterruptSignal
ConfirmationRequest
SpokenFeedback
```

不负责：

- 任务规划
- 动作执行
- 偏好持久化

### 3. Observation Context

负责把电脑状态转成 AI 可用上下文。

核心对象：

```text
ScreenSnapshot
ElementCandidate
WindowState
BrowserState
ObservationFrame
```

不负责：

- 决定下一步做什么
- 执行动作

### 4. Planning Context

负责从目标和观察生成计划。

核心对象：

```text
Plan
PlanStep
ActionProposal
Uncertainty
RecoveryPlan
```

不直接调用系统鼠标键盘。

### 5. Execution Context

负责把动作协议变成真实操作。

核心对象：

```text
ActionCommand
ActionResult
CursorLease
ExecutorCapability
ExecutionReceipt
```

不负责理解用户意图。

### 6. Safety Context

负责权限、风险、确认和阻断。

核心对象：

```text
SafetyRule
RiskLevel
PolicyDecision
PermissionScope
ConfirmationPolicy
```

不依赖具体 UI 或具体模型。

### 7. Trace Context

负责事件溯源和工作流沉淀。

核心对象：

```text
DomainEvent
TraceSession
TraceNode
WorkflowTemplate
ReplayPlan
```

它是调试、复盘、工作流生成和 AI 自我分析的基础。

### 8. Personalization Context

负责偏好、习惯用语和自配置。

核心对象：

```text
Preference
PhraseAlias
BehaviorRule
WorkflowShortcut
PreferenceProposal
```

所有修改都必须走 propose → confirm → apply。

## Clean / Hexagonal Architecture

每个 bounded context 内部建议分三层：

```text
domain/
  纯领域对象、规则、状态机

application/
  用例编排、命令处理、事务边界

adapters/
  UI、数据库、模型、浏览器、系统 API
```

依赖方向：

```text
adapters → application → domain
```

禁止：

```text
domain → adapters
domain → database
domain → model_provider
domain → browser_api
```

这样未来换模型、换浏览器执行器、换数据库，不会动核心领域逻辑。

    task/

      README.md

    interaction/
      domain/
      application/
      ports/
      adapters/
      tests/
      README.md

    observation/
    planning/
    execution/
    safety/
    trace/
    personalization/
    workspace/

  shared/
    kernel/
      ids.ts
      result.ts

      domainEvent.ts

      actionProtocol.ts

      fixtures/

    executors/

规则：

- `contracts` 只放跨模块协议，不放业务流程
- 每个 domain 必须有 README 说明职责、入口、事件和禁止事项

## 面向 AI 修改的代码组织

AI 编程助手最怕三件事：

```text
隐式依赖
跨文件副作用
同名概念混乱
```

因此建议引入 **AI-maintainability conventions**。

### 1. 每个模块必须有 Module README

模板：

```md
# Module: execution

## Responsibility
把 ActionCommand 转成真实系统操作。

## Owns
- CursorLease
- ExecutorCapability
- ActionResult

## Does not own
- 用户意图理解
- 安全策略判定
- TTS 播报

## Public API
- requestCursorLease()
- executeAction()
- releaseCursorLease()

## Emits events
- CursorLeaseGranted
- ActionExecuted
- ActionFailed

## Invariants
- 没有 CursorLease 不得执行前台鼠标键盘动作
- 用户中断优先级最高
```

AI 修改代码前先读 README，能显著减少越界修改。

### 2. 每个模块只暴露一个 public API

```text
domains/execution/index.ts
```

其他模块只能从 `index.ts` import。

禁止跨模块深层 import：

```text
domains/execution/domain/cursorLease.ts
```

### 3. 用显式端口隔离外部能力

例如：

```ts
interface MousePort {
  moveTo(target: ScreenPoint): Promise<void>
  click(button: MouseButton): Promise<void>
}
```

Execution 依赖 `MousePort`，不依赖具体系统库。

### 4. 命名必须领域化

避免：

```text
manager
helper
utils
processor
service
```

优先：

```text
CursorLeaseAllocator
ActionSafetyEvaluator
PreferenceProposalPolicy
TaskStateMachine
ObservationFrameBuilder
```

## 稳定协议优先于实现

AI 光标最应该稳定的是协议，而不是内部实现。

长期稳定协议：

```text
Action Protocol
Observation Frame
Domain Event Schema
Preference Proposal Protocol
Cursor Lease Protocol
Safety Policy Decision
Workflow Template Schema
```

实现可以换：

- mouse executor
- browser adapter
- ASR/TTS provider
- LLM provider
- storage backend
- UI framework

协议不轻易换。

## 事件驱动与 Event Sourcing

所有关键动作都应生成领域事件。

示例：

```text
UserUtteranceReceived
InteractionIntentClassified
TaskStarted
ObservationCaptured
ActionProposed
PolicyDecisionMade
CursorLeaseGranted
ActionExecuted
UserInterrupted
PreferenceProposalCreated
WorkflowTemplateCreated
```

事件有三种用途：

1. UI 更新
2. 调试和回放
3. 未来工作流生成

建议：

```text
命令改变状态
事件记录事实
查询读取投影
```

不要让 UI 到处读取内部对象，而应读取 projection：

```text
TaskPanelProjection
WorkspaceProjection
TraceTimelineProjection
PreferencePanelProjection
```

## 版本演进策略

### 1. 协议版本化

所有跨模块协议必须带版本：

```json
{
  "schema": "action_command",
  "version": "1.0",
  "action": "mouse.click"
}
```

### 2. 只新增，不破坏

优先新增字段，避免删除字段。

```text
v1: expected_result
v1.1: expected_result + verification_hint
```

### 3. 用 adapter 兼容旧版本

```text
ActionCommandV1Adapter
PreferenceProposalV1Adapter
WorkflowTemplateV1Adapter
```

### 4. Migration 与领域事件分开

数据库迁移只处理存储结构。

领域事件迁移处理事件 schema。

## 避免未来重构的关键边界

### 不让 UI 持有业务规则

UI 只能发命令、读 projection。

错误：

```text
点击确认按钮 → UI 判断是否可以提交
```

正确：

```text
点击确认按钮 → application command → SafetyPolicy → decision → projection
```

### 不让模型直接调用执行器

错误：

```text
LLM output → mouse.click
```

正确：

```text
LLM output
→ ActionProposal
→ SafetyPolicy
→ ExecutionScheduler
→ Executor
```

### 不让偏好层绕过安全层

错误：

```text
用户偏好：以后自动提交
→ 直接提交
```

正确：

```text
用户偏好
→ SafetyPolicy 合并
→ 高风险仍需确认
```

### 不让插件共享全局状态

插件只能通过 ports 访问能力，不能直接读写核心状态库。

## 面向 AI 的测试策略

测试不只是防 bug，也是给 AI 修改代码时提供护栏。

### 1. Domain invariant tests

例如：

```text
没有 CursorLease 不得执行 mouse.click
blocked 动作不得进入执行队列
PreferenceProposal 未确认不得 apply
用户 interrupt 必须释放 CursorLease
```

### 2. Contract tests

每个插件必须跑合同测试：

```text
ExecutorPluginContract
ObserverPluginContract
ModelProviderContract
AudioProviderContract
```

### 3. Golden trace tests

保存典型任务 trace：

```text
用户说“打开第一个”
→ observation
→ action proposal
→ safety decision
→ action executed
```

每次改架构后确保事件序列仍合理。

### 4. Simulation tests

用模拟桌面环境测试：

- 页面变化
- 弹窗
- 用户打断
- 光标租约冲突
- 配置规则冲突

## AI 修改代码的推荐流程

为未来 AI 编程助手内置工作流：

```text
1. 读取目标模块 README
2. 读取 public API
3. 搜索相关 DomainEvent
4. 修改 domain/application
5. 不跨模块深层 import
6. 补充 invariant / contract / golden trace 测试
7. 更新模块 README 或 ADR
```

如果 AI 需要跨 bounded context 修改，必须生成 ADR。

## MVP 到成熟版的演进路径

### P0：模块化单体

```text
domains/
shared/contracts/
adapters/
ui/
```

先把边界划清。

### P1：插件化执行器和观察器

```text
executor plugins
observer plugins
model provider plugins
```

支持替换底层能力。

### P2：事件溯源与工作流

```text
domain events
trace store
workflow template
projection
```

支持复盘、回放、生成工作流。

### P3：多工作区和个人化

```text
workspace context
cursor lease scheduler
personalization rules
preference proposals
```

支持长期用户习惯。

### P4：可拆服务

当上下文稳定后，可以拆：

```text
model service
trace service
workflow service
plugin host
```

但核心桌面端仍保留本地执行闭环。

---

## 架构红线

为了避免版本迭代后大重构，必须禁止：

```text
模型直接操作系统 API
UI 直接修改领域状态
插件直接读写核心数据库
偏好规则绕过安全策略
工作区共享隐式全局上下文
未版本化的跨模块 JSON
无 README 的大型模块
无事件留痕的关键状态变化
```

---

## 一句话总结

**AI 光标最适合先做模块化单体，用 DDD 划清领域边界，用 Hexagonal Architecture 隔离外部能力，用事件溯源沉淀可回放轨迹，用稳定协议和插件机制承载未来扩展，并用 AI-readable conventions 让人类和 AI 都能安全修改代码。**
