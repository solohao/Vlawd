# 06 · 语音技术选型路线

## 核心原则

AI 光标的语音技术选型不应从“哪个 voice agent 框架最酷”开始，而应从产品本质开始：

```text
耳机协作
低延迟打断
任务状态汇报
本地隐私
低成本高频交互
屏幕执行联动
```

因此推荐路线是：

> **本地优先的 Hybrid Voice Desktop Agent。**

本地负责高频交互，云端或大模型负责复杂推理。

---

## 能力分层

```text
P0：按住说话 + 本地短反馈
P1：任务型准全双工
P2：多端实时语音入口
P3：端到端全双工 / 多模态语音模型
```

---

## P0：按住说话 + 本地短反馈

目标是最快验证“耳机指挥 + 光标执行 + 语音汇报”的爽感。

建议能力：

- Push-to-talk / 全局快捷键
- 本地 ASR
- 本地 TTS
- 本地识别抢占词
- 复杂任务交给 LLM
- 系统鼠标键盘执行
- 动作日志

适合技术：

- ASR：SenseVoice、faster-whisper、whisper.cpp、FunASR
- TTS：Supertonic 3、Kokoro、Piper
- 音频：本机系统音频 API

这一阶段不需要 LiveKit，也不需要 WebRTC。

---

## P1：任务型准全双工

目标是让 AI 像耳机里的员工一样工作。

新增能力：

- VAD 持续监听
- AI 播报可被打断
- TTS 可中断播放
- 流式 ASR 可选
- 语音确认
- 任务阶段播报
- 本地抢占词优先级

关键不是让 AI 闲聊得像真人，而是：

```text
AI 正在汇报时，用户可以马上插话纠正
```

参考技术：

- Silero VAD
- WebRTC VAD
- 本地 ASR 流式能力
- 可中断 TTS 播放队列

---

## P2：多端实时语音入口

当产品需要手机、网页、远程控制电脑时，再考虑 LiveKit / WebRTC。

适合场景：

- 用户不在电脑前，通过手机耳机指挥电脑
- Web 端语音入口
- 多人或多 Agent 房间
- 企业 IM/通话系统集成
- 云端 Agent Worker

LiveKit Agents 的价值：

```text
Room / Job / Worker 模型
WebRTC 音频流
VAD
打断
流式 TTS
Agent 加入通话房间
```

但 LiveKit 不应成为本地桌面 MVP 的架构中心。

它适合后期作为远程语音接入层。

---

## P3：端到端全双工模型

PersonaPlex、RealtimeModel、Qwen-Omni 类模型代表未来方向。

优势：

- 同时听和说
- 处理打断
- 理解反馈语
- 更自然的接话
- 声音和情绪更统一

代价：

- 模型重
- GPU 要求高
- 本地部署成本高
- 云服务依赖强
- 不利于独立开发者第一版控制成本

因此它适合：

- 高端模式
- 企业部署
- 本地高性能设备
- 未来硬件成熟后替换部分 pipeline

---

## TTS 选型

### Supertonic 3

适合作为默认本地 TTS 候选。

特点：

- 约 99M 参数
- ONNX Runtime
- 本地推理
- 多端示例丰富
- 支持多语言
- 更重视复杂文本朗读

适合 AI 光标：

```text
短状态播报
任务确认
本地隐私
低成本高频反馈
```

### Higgs Audio v3 / SGLang-Omni

适合作为后期高质量、流式、可控 TTS 方向。

特点：

- 面向对话场景
- 支持流式合成
- 支持情绪、风格、韵律、音效控制标记
- 多 stage 推理框架
- 更适合服务端 GPU 部署

适合后期：

```text
高质量耳机协作声音
多 AI 员工不同语气
情绪化状态反馈
云端/局域网 TTS 服务
```

### OmniVoice

适合高级多语言和音色克隆。

特点：

- 600+ 语种
- 零样本克隆
- Voice Design
- 更偏高能力模型路线

不建议作为 MVP 默认依赖。

### Kokoro / Piper / edge-tts

适合原型期或兜底。

---

## ASR 选型

AI 光标 ASR 应优先考虑：

- 中文识别
- 低延迟
- 本地运行
- 快捷命令准确率
- 可流式或准流式

候选：

- SenseVoice：中文体验值得优先验证
- faster-whisper：生态成熟
- whisper.cpp：本地部署友好
- FunASR：中文与工程生态强

建议不要把完整自然语言任务和短命令混在同一条链路里处理。

```text
短命令：本地快速识别
长任务：完整 ASR 后交给 LLM
```

---

## Voice Agent 框架如何看待

很多 voice agent 框架默认目标是：

```text
用户说话 → AI 语音回复
```

AI 光标的目标是：

```text
用户说话 → AI 操作电脑 → AI 必要时语音反馈
```

所以不能让 voice framework 成为核心架构。

核心仍然应是：

```text
Task Runtime
Action Protocol
Execution Scheduler
Observation Layer
Trace Store
Safety Policy
Voice Interaction Runtime
```

---

## 推荐技术栈组合

### 原型版

```text
Desktop Shell：Tauri / Electron
ASR：SenseVoice or faster-whisper
TTS：Supertonic 3 / Kokoro / Piper
VAD：Silero / WebRTC VAD
LLM：云端强模型
Execution：mouse/keyboard + browser extension/CDP
Storage：SQLite
```

### 可用版

```text
本地快速命令识别
可中断 TTS 播放队列
任务型准全双工状态机
屏幕编号 overlay
单真实光标调度
多 AI 工作区
动作日志与工作流模板
```

### 后期高级版

```text
LiveKit / WebRTC 多端入口
Higgs / OmniVoice / PersonaPlex 高级语音
端到端全双工模型
手机远程控制桌面
多角色 AI 员工音色
```

---

## 一句话总结

**AI 光标的语音路线应先本地、低延迟、可打断、任务化；LiveKit 用于后期多端实时入口，PersonaPlex/RealtimeModel 用于未来自然全双工，Supertonic 3/Higgs/OmniVoice 分别服务本地默认、高质量流式和高级多语种音色。**
