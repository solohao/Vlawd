## 目标

Dogfood 制度用于解决两个问题：

1. 开发激情被模型选择、调试和重复维护消耗；
2. 界面手感、布局和视觉细节无限抢占核心闭环。

运行循环：

```text
真实使用
→ 自动记录 Session、失败和纠正
→ 每周统一归类
→ 选择最大阻塞
→ 只修改一个问题
→ 历史任务回放
→ 录制短演示
→ 继续真实使用
```

---

## 单一 Golden Path（DOGFOOD.GOLDEN）

任何周期只允许一个完整场景。

当前 Golden Path：

> **戴着耳机告诉 Vlawd 一个真实研究目标；它可见地整理资料；用户途中插话纠正；结果保存为以后可以继续的 Session / Project Draft。**

一个 Golden Path 必须写清：

- 用户是谁；
- 起始状态；
- 输入目标；
- 允许动作；
- 禁止动作；
- 可见结果；
- 成功标准；
- 停止条件。

如果新需求不能提高当前场景的成功率、安全性、恢复能力或复用价值，则不进入当前周期。

---

## 使用与开发分离

每次真实使用后只记录问题，不立即切换到代码修复。

建议：

```text
使用时：
  只作为用户
  完成真实目标
  标记问题和感受

固定开发窗口：
  汇总多个 Session
  根据证据选择一个问题
  开始修改
```

这样避免一天内不断在“产品用户、设计师、工程师、模型评测者”之间切换。

## 两周周期（DOGFOOD.CYCLE）

规划本身也有预算：

- 每周最多一次规划；
- 每次最多 45 分钟；
- 周期中不改变主线；
- 只有周期结束时允许调整下一目标；
- 下一步已经可以用一句验收标准描述时，立即停止继续规划。

### Day 1：选择唯一问题

- 查看真实 Session；
- 选择最高优先级问题；
- 写一句自然语言验收标准；
- 冻结模型、布局和技术选择。

### Day 2-6：最小实现

- 只修改实现当前验收所需内容；
- 优先复用现有接口；
- 不顺便重构；
- 新想法进入 Idea Vault。

### Day 7-10：真实 Dogfood

- 使用真实任务；
- 至少运行多个不同输入；
- 保存成功和失败 Session；
- 不因第一次失败立即换模型。

### Day 11-12：修最大缺陷

- 只修阻止验收的问题；
- 更新必要测试和回放；
- 不进入纯视觉打磨。

### Day 13：验证

- 同一 Golden Path 连续成功至少三次；
- 运行安全、类型和回放检查；
- 记录指标和已知限制。

### Day 14：视频和复盘

- 录制 30–90 秒真实演示；
- 决定继续、缩小或停止；
- 下一周期只保留一句目标。

---

## 可视化检查点（DOGFOOD.DEMO）

视频不是宣传片，而是可重复运行的验收证据。

### 必须显示

1. 起始状态；
2. 用户输入；
3. 产品产生的可见变化；
4. 最终结果；
5. 本周期新增能力。

### 规则

- 30–90 秒；
- 使用真实版本；
- 只允许简单裁剪等待时间；
- 不制作复杂动画、字幕或宣传音乐；
- 录制准备超过 30 分钟则停止；
- 文件按 `日期-周期-场景` 保存；
- 失败保留为 Session，不进入正式演示。

### 价值

- 防止“代码写了很多但体验没有改变”；
- 形成连续产品成长记录；
- 为以后外部展示提供材料；
- 让开发成果产生可感知反馈；
- 帮助对比退化。

---

## 问题优先级（DOGFOOD.PRIORITY）

| 优先级 | 类型 | 处理方式 |
|--------|------|---------|
| P0 | 安全、隐私、数据丢失 | 立即停止其他开发 |
| P1 | Golden Path 无法完成 | 当前周期优先 |
| P2 | 失败后无法恢复 | 当前或下一周期 |
| P3 | 同一摩擦反复出现 | 满足三次证据后处理 |
| P4 | 偶发不顺或多一步操作 | 记录 |
| P5 | 颜色、间距、动画、图标偏好 | Polish Later |

排序：

```text
安全
> 任务成功
> 可恢复
> 可理解
> 反复摩擦
> 视觉精致
```

## UI 调优止损（DOGFOOD.POLISH）

### UI 成熟等级

```text
U0：日志或最简控件可操作
U1：可以完整完成任务
U2：状态清楚，不容易误操作
U3：视觉统一、动画和细节精致
```

当前只要求 U2。核心闭环稳定前禁止以 U3 为主要周期目标。

### 三次证据规则

一个 UI 问题只有满足以下任一条件才进入开发：

- 阻止任务完成；
- 容易导致危险确认；
- 造成明显状态误解；
- 在独立真实 Session 中出现至少三次。

“感觉不够完美”不是证据。

### 时间预算

```text
60% 核心能力与可靠性
20% 测试、回放和日志
10% 可理解性与必要 UI
10% 复盘和机动
```

纯视觉调整上限为周期总时间的 10%。只有当前假设就是“画布是否可理解”时，UI 才能成为主要投入。

### 视觉冻结

基础颜色、字体、间距和组件至少冻结三个周期：

- 不反复重做主题；
- 不移动整体导航结构；
- 不制作复杂动画；
- 不同时支持多套布局；
- 只优化当前主要窗口尺寸；
- 优先使用已有组件。

### 30 分钟止损

纯视觉问题 30 分钟内没有解决就记录并回到主线。安全、可访问性和误操作问题不属于纯视觉问题。

### UI 验收问题

- 我是否知道 AI 正在做什么；
- 我是否知道为什么暂停；
- 我能否暂停、取消和接管；
- 我能否找到结果；
- 第二天能否继续任务。

五项成立即可通过当前阶段。

## Idea Vault（DOGFOOD.VAULT）

每个新想法最多记录：

```text
一句话是什么
解决谁的问题
为什么与愿景相关
最小验证是什么
什么条件满足后才允许启动
```

记录时间不超过几分钟，不在当前周期继续扩展商业和架构计划。

新想法只有在当前周期结束、并且比现有最高优先级问题更重要时，才可能进入下一周期。

---

## Definition of Done

一个周期完成必须同时满足：

- 自然语言验收标准通过；
- Golden Path 连续成功至少三次；
- 无 P0；
- 适用 lint、typecheck 和测试通过；
- 指标可记录；
- 已知限制明确；
- 30–90 秒演示完成；
- 下一周期目标只有一个。

完成不要求：

- 所有界面精致；
- 所有模型都评测；
- 架构永久最优；
- 全部功能具备；
- 对外发布。

## 自助迭代成熟路线

### Stage A：人工选问题

Vlawd 自动保存 Session，创始人每周选择一个最高优先级问题。

### Stage B：自动聚类失败

系统按音频、抢占、规划、执行、权限、恢复、结果质量和成本聚类。

### Stage C：历史 Session 回放

每次修改后自动运行 Golden Tasks，比较成功率、延迟、成本和安全。

### Stage D：提出候选改进

系统根据失败聚类给出三个候选，创始人只批准一个。系统不能自动发布高风险改动。

最终目标：

```text
真实使用
→ 自动收集证据
→ 提出最值得修的候选
→ 人选择一个
→ AI 辅助实现
→ 历史回放验证
→ 人验收并录制演示
```

## 外部反馈边界

证据优先级：

```text
创始人反复真实使用
> 客观 Session 指标
> 历史任务回放
> 少量用户真实操作
> 用户口头意见
> 陌生人评论
```

外部用户贡献问题证据，但不直接决定解决方案。市场化前仍需要少量真实用户验证，不能永久用 Dogfood 代替市场证据。

---

## 一句话总结

**Dogfood 制度把无限调优变成一个受控循环：每两周只修一个有真实证据的问题，以短视频证明体验改变，并把纯视觉优化限制在明确预算内。**
