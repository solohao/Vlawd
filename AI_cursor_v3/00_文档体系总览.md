## 长期愿景

> **构建一个可执行、可验证、可演化的 Agentic Web。**

更完整的定义：

> 让人通过自然交互调度本地与云端 Agent 完成现实任务，并把每次执行沉淀为可验证、可复用、可持续进化的能力与项目。

```text
今天的互联网：
人阅读页面、点击界面、手工完成操作

目标中的 Agentic Web：
人表达目标
→ Agent 发现知识、能力与服务
→ 在明确权限内执行
→ 人可以监督、打断和纠正
→ 结果形成 Evidence
→ 成功经验沉淀为 Capsule / Living Project
→ 其他人和 Agent 可以复用、验证、Fork 与改进
```

---

## 为什么升级为 V3

V2 的核心对象是一个本地全双工 AI 桌面产品。V3 保留这一产品主线，但长期范围已经发生架构级变化：

```text
V2：
全双工语音
→ 可见桌面执行
→ Session
→ 个人工作流

V3：
Vlawd 本地 Runtime
+ Session / Evidence
+ Capsule 可移植能力协议
+ Post is Project 协作平台
+ Module Spec 代码垂直项目
+ 高校二维知识论坛垂直项目
+ 服务与商业能力网络
```

这不是给 V2 继续增加功能，而是把 Vlawd 放入一个更大的项目群和长期网络架构中，因此采用主版本升级。

V3 当前是**规划与验证版本**，不复制 V2 代码。现有实现仍位于：

```text
AI_cursor_v2/app
```

只有当 V3 的真实验证要求证明现有代码边界需要破坏性调整时，才通过独立 ADR 决定代码迁移或目录升级。

详见 [V2_to_V3_变更说明](./变更记录/V2_to_V3_变更说明.md)。

**Vlawd 是 Agentic Web 的全双工个人终端；Session 记录真实协作；Capsule 让能力可移植；Post is Project 让问题、方法和能力能够持续演化；不同垂直项目负责验证这套网络是否真正产生价值。**

## 文档管理规则

### 版本规则

```text
主版本：
产品本体、系统边界或项目群结构发生架构级变化

次版本：
模块内目标、接口、阶段或验证标准发生变化

修订：
不改变决策含义的文字、链接和例子修正
```

### 模块编号规则

```text
格式：前缀.后缀

VISION.*   长期愿景与类别定义
PORT.*     项目群成员与定位
ARCH.*     分层架构与边界
ART.*      Session / Capsule / Project / Evidence 对象
UX.*       交互、画布、论坛和任务工作区
VALID.*    MVP 验证路径
DOGFOOD.* 创始人自用迭代制度
EVAL.*     模型、成本和回放评测
MOAT.*     护城河与生态积累
```

所有编号必须在 [01_项目群模块注册表](./01_项目群模块注册表.md) 中注册。

### 阶段管理规则

```text
完整文档描述长期目标。
阶段文档只决定当前允许实现什么。

新增想法：
  先登记到 Not Now / Idea Vault
  不直接改变当前阶段

进入开发：
  必须有明确用户、Golden Path、验收指标和停止条件

阶段升级：
  必须满足证据门，不因概念完整或界面完成而升级
```

### 规划与实现分离

```text
V3 文档：
  管理项目群愿景、边界、验证顺序和共同协议

V2 app：
  当前可运行代码基线

未来实现：
  优先在现有代码中完成 Vlawd Golden Path
  不因文档主版本升级而复制工程目录
```

## 当前唯一执行焦点

V3 的长期愿景允许多条路径，但当前默认主线是：

> **先通过 Vlawd 自我 Dogfood，证明“全双工讨论 / 研究 → 可见执行 → 可恢复 Session → 可继续项目”的个人闭环。**

第一阶段不建设：

- 通用 Agent 网络协议；
- 开放社区和传统 Feed；
- 完整 Post is Project 平台；
- 通用共享显卡池；
- 自研耳机硬件；
- 商家支付与交易网络；
- 大规模 Capsule Marketplace；
- 多模型自由选择中心。

---

## 一句话总结

**V3 不是把所有想法一次实现，而是为 Agentic Web 项目群建立清晰边界：先用 Vlawd、Module Spec 和高校知识论坛三个垂直场景验证真实需求，再从共同摩擦中提取 Capsule 协议和平台。**
