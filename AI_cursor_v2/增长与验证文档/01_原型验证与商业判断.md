# 01 · 原型验证与商业判断
 
## 核心问题
 
AI 光标方向是否值得做，不取决于概念是否像 AI OS 或 AI 浏览器，而取决于：
 
> **用户是否愿意每天让它替自己做多次烦人的电脑操作。**
 
第一阶段要验证的不是技术上能否万能，而是：
 
```text
这个体验是否足够爽
用户是否觉得它真的替我干活了
用户是否愿意为“少亲手做电脑苦活”付费
```
 
---
 
## 需要验证的假设（VALID.HYPO）
 
### 假设 1：可见执行比黑箱自动化更容易获得信任
 
验证方式：
 
- 让用户看 AI 光标真实移动、点击、拖拽
- 高风险动作前暂停确认
- 观察用户是否更愿意继续让它做下一步
 
### 假设 2：语音纠错能降低失败挫败感
 
验证方式：
 
- 让 AI 遇到不确定目标
- 用户说“不是这个，点右边那个”
- 观察是否能自然恢复任务
 
### 假设 3：用户会为“代劳感”付费
 
验证方式：
 
- 找 3-5 个重复任务
- 让用户体验前后对比
- 询问是否愿意每月付费
 
### 假设 4：浏览器优先足以覆盖早期高频任务
 
验证方式：
 
- 记录用户每天想交给 AI 的电脑任务
- 分类浏览器、文件、聊天、文档、系统设置
- 如果浏览器任务占比高，先浏览器优先是正确的
 
### 假设 5：低成本架构能支撑高频交互
 
验证方式：
 
- 高频命令本地识别
- 复杂规划才调用大模型
- 统计每小时成本和延迟
 
### 假设 6：耳机协作反馈能形成心理隔离感
 
验证方式：
 
- 用户通过耳机委托 AI 执行烦人网页任务
- AI 只在关键节点短句汇报
- 用户可随时打断、纠正、确认
- 对比“只有鼠标执行无语音反馈”和“耳机任务对讲机反馈”的心理负担差异
 
### 假设 7：可监督自配置能提升长期留存
 
验证方式：
 
- 允许用户用自然语言创建习惯用语
- 所有配置变更以卡片确认
- 观察用户是否会复用短语和“按老规矩”类表达
- 观察用户是否愿意维护自己的 AI 偏好面板
 
---
 
## 原型验证任务（VALID.DEMO）
 
建议先做 7 个强 Demo。
 
### Demo 1：网页搜索代劳
 
```text
用户说：帮我搜索这个主题，打开前 5 个结果。
AI 光标：打开浏览器、搜索、打开标签页、逐个切换。
```
 
验证点：
 
- 光标动作是否有真实代劳感
- 多标签操作是否流畅
 
### Demo 2：屏幕标号点击
 
```text
系统给页面元素编号。
用户说：点 3，右键 5，往下滚一点。
```
 
验证点：
 
- 语音低层控制是否顺滑
- 用户是否觉得安全
 
### Demo 3：表单半自动填写
 
```text
用户说：把我的资料填进这个表单，不确定的问我。
```
 
验证点：
 
- 用户是否愿意让 AI 接触真实表单
- 高风险提交确认是否足够安心
 
### Demo 4：标签组整理
 
```text
用户说：把这些求职相关网页放到一个标签组。
```
 
验证点：
 
- AI 对浏览器操作是否像熟练员工
- 标签 / 窗口管理是否有明确价值
 
### Demo 5：任务留痕回放
 
```text
执行完成后展示：
你刚才让我完成了 8 步操作，可保存为工作流。
```
 
验证点：
 
- 用户是否理解“从指挥中生成工作流”
- 是否愿意重复使用同类流程
 
### Demo 6：耳机任务对讲机
 
```text
AI 光标执行网页任务时，耳机持续短句汇报：
“我打开第二个。”
“这里需要登录，我暂停。”
“我准备提交，需要你确认。”
用户可随时说“停”“不是这个”“继续”。
```
 
验证点：
 
- 耳机反馈是否显著增强“员工在替我做事”的感觉
- 打断响应是否足够快
- 用户是否更愿意放手让 AI 继续执行
 
### Demo 7：自然语言训练 AI
 
```text
用户说：
“以后我说‘投这个’，就是保存岗位、分析匹配度、生成定制简历。”
 
AI 显示配置卡片：
短语：投这个
动作：保存岗位 → 分析匹配度 → 生成定制简历
范围：求职工作区
提交前：仍需确认
```
 
验证点：
 
- 用户是否理解 AI 学到了什么
- 配置确认是否足够安全
- 下次短语触发是否能明显降低交互成本

## 商业价值判断（VALID.BIZ）
 
### 低价值形态
 
```text
语音控制鼠标
```
 
如果只停留在这一层，容易像玩具。
 
### 中价值形态
 
```text
语音指挥 AI 光标操作网页和窗口
```
 
如果体验流畅，有早期用户价值。
 
### 高价值形态
 
```text
可监督 AI 桌面员工
+ 任务留痕
+ 工作流沉淀
+ 重复电脑苦活代劳
+ 习惯用语和个人规则
```
 
如果能让用户把真实重复任务交给它，有付费潜力。

## 付费模式：积分制（Token 计量）（VALID.CREDITS）

### 核心判断：收费点只能在服务端

AI 光标是桌面端软件，面向中国市场。两个现实约束决定了付费设计：

```text
约束 1：桌面端软件容易被破解 → 任何本地校验/计费逻辑都不可靠
约束 2：中国破解生态成熟 → 客户端加密、功能分级解锁形同虚设
结论：唯一不可绕过的收费点 = 服务端 API 调用（token 计量）
```

### 积分 = Token

积分的本质就是大模型 API 的 token 消耗量，由服务端计量，不可伪造：

```text
用户语音交互 → 经服务端调用大模型 API → 消耗 token → 扣除积分
```

不采用"按操作复杂度计费"——复杂度判断在客户端做会被篡改，在服务端做会增加延迟、违背本地优先定位。

### 三档付费

| 档位 | 说明 | 目标用户 |
|------|------|---------|
| 免费层 | 每日/每月赠送固定基础积分 | 体验用户、轻度用户 |
| 按需充值 | 单次购买积分包（预付 token 费用） | 不想订阅的用户 |
| 月度订阅 | 月付获得更大积分额度 | 重度用户（求职者、研究者、开发者、运营） |

### BYO Key（保留）

用户自带模型 API Key → 不走平台服务端 → 不消耗积分 → 平台不收推理费用。

BYO Key 用户的定位：

```text
技术用户自行部署模型 → 当作免费推广渠道，扩大用户基数
类似 Ollama 模式：软件免费，推理服务收费
```

### 云端增值服务（长期付费壁垒）

软件本体可被破解，但云端服务不行。长期付费壁垒应建立在：

```text
Session 云端同步（跨设备访问历史 Session）
长期记忆云存储（Memory Store 云端备份）
工作流市场（发布/下载/更新工作流模板，通过服务端）
高级 Memory recall（云端强模型辅助的复杂查询）
```

工作流市场是"内容生态"而非纯软件功能，难以被简单破解——类似 Notion 模板市场。

### 根本性矛盾与应对

```text
矛盾：产品定位"本地优先"，但收费点只能在"服务端"
      如果用户完全本地部署（BYO Key + 本地模型），平台没有收费抓手

应对策略：
  1. 接受"完全本地用户免费"，大部分普通用户不会自己部署模型
  2. 核心能力部分上云（Session 同步、工作流市场、长期记忆）
  3. 破解者可用基础功能，但无法享受云端增值服务
  4. 用免费用户基数换市场占有率，用云端服务换持续收入
```

## 最大风险与规避（VALID.RISK）
 
### 1. 太像玩具
 
解决：
 
- 不宣传“语音控制鼠标”
- 宣传“替你做电脑苦活”
- 首屏展示真实任务 Demo
 
### 2. 可靠性不足
 
解决：
 
- 浏览器优先
- 标号机制
- 动作前高亮
- 可暂停纠错
- 危险确认
 
### 3. 成本过高

缓解：

- 全双工模型本地运行（核心推理不依赖云端）
- 本地规则引擎处理安全控制（不消耗 token）
- 积分制按 token 实际消耗计费（成本与收入线性挂钩）
- BYO Key（用户自带 Key 降低平台成本）
- 云端强模型仅在本地模型不足时调用（预计 < 10% 的任务需要）
 
### 4. 巨头竞争
 
解决：
 
- 不争 AI OS
- 不争浏览器入口
- 做现有桌面上的可监督执行层
- 打磨个人电脑苦活代劳体验
 
---
 
## 第一阶段继续标准（VALID.CRITERIA）
 
这个方向值得继续，如果：
 
```text
创始人自己连续 7 天每天使用 10 次以上
至少 3 个真实任务明显减少心理阻力
用户看到 Demo 后 30 秒内理解“它在替我操作电脑”
用户愿意为“可监督电脑代劳”而不是“语音控制鼠标”付费
用户开始主动创建习惯短语和复用工作流
```
 
---
 
## 一句话总结
 
**AI 光标的商业价值要靠真实代劳任务证明：如果用户只是觉得它酷，它就是玩具；如果用户愿意每天让它替自己点、查、填、整理，并把成功流程保存成工作流，它就有机会成为个人桌面自动化的新入口。**
