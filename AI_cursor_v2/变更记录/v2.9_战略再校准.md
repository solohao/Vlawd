# v2.9 战略再校准说明

---

## 变更日期：2026-06-27

## 变更级别：次版本级战略再校准（不推翻 V2 架构，校准方向与边界）

---

## 一、变更背景

### 为什么需要这次再校准

V2 文档体系已经把"全双工耳机协作式 AI 桌面员工"的产品形态和工程骨架描述清楚，
但在一次从第一性原理出发的复盘中，暴露出三处需要被显式固化的判断偏差与现实约束：

```text
1. 痛点表述偏窄
   早期文档常以"重复性网页/电脑操作"概括痛点，
   容易把产品误导向 RPA / 固定工作流的解法空间，
   而真正的差异化战场是"名义重复、实际多变、需实时监督的日常任务"。

2. 护城河与模态被绑死
   "全双工语音"被默认当作产品第一优先级，
   导致战略风险集中压在"最难、最不成熟、且不一定匹配痛点"的技术上，
   忽视了真正护城河（agent 工程 + 工作流记忆 + 可见执行）是模态无关的。

3. 硬件现实未被正视
   候选模型选型默认 7B-9B 全双工"消费级 GPU 可运行"，
   但创始人验证机为 RTX 4060 Laptop（8GB 显存），
   8GB 是普通用户的真实中位配置，却跑不动 9B 原装全双工，
   这一现实必须驱动分层策略和双角色资源分配。
```

### 这次再校准不改什么

```text
不推翻两角色架构（全双工执行大脑 + 记录引擎 + 固定安全层）
不推翻原子动作协议、三种可见性模式、Session Graph 上下文工程
不推翻本地优先隐私原则和积分制商业模型
DuplexModelProvider 可插拔接口保持不变（它正是支撑本次校准的工程基础）
```

## 二、战略决策

### ADR-009：痛点重定义为"监督型多变任务"

```text
替代：以"重复性电脑操作"概括核心痛点
校准：核心痛点 = 名义上重复、但实际内容因变量不断变化、无法预先脚本化的日常任务
原因：
  - 硬重复任务的天然解法是录制-回放 / RPA / 固定工作流，不需要实时语音
  - 真正没人能填的缝 = "每次都在变、必须有人临场判断"的任务
  - 语音的独特价值不在"省重复"，而在"实时监督式执行"：临场纠正、可打断、信任与控制
影响：PROD.STRATEGY / MVP.USER / MVP.SCENE / MVP.METRIC 的表述需要对齐
```

### ADR-010：护城河与模态解耦，语音降为"升级层"

```text
判断：护城河 = agent 工程 + 工作流/Session 记忆 + 可靠可见执行（模态无关）
      全双工语音 = 体验升级层，而非产品成立的前置条件
原因：
  - 把战略押在"全双工小模型变强"上，这道技术墙恰恰会被模型厂商自己填平
  - 真正的壁垒是工作流数据网络效应、流程沉淀锁定、可监督执行的工程成熟度
  - 这些壁垒不在乎输入是语音还是文本，可以先用最简模态积累
推论（不强制改阶段，仅作战略指导）：
  - 护城河照建，但不必让"全双工语音优先"绑架整个产品节奏
  - 语音作为升级层，在技术成熟 + 需求拉动双双满足时叠加
影响：PROD.STRATEGY / PROD.COMPETE / PROD.BOUNDARY 表述对齐
```

### ADR-011：与文本 Agent 配合而非替代

```text
校准：AI 光标（实时监督多变任务）与文本/异步 Agent（OpenAI Operator、Manus 等）
      是配合关系，而非替代关系
分工：
  - 固定、可无人值守的任务 → 交给文本/异步 Agent
  - 多变、需实时判断和监督的任务 → AI 光标的语音监督执行
原因：诚实承认边界能让产品定位更锐利，避免"什么都想做"
影响：PROD.BOUNDARY / PROD.COMPETE 表述对齐；ARCH.LOCAL 本地/云端分工原则保持一致
```

### ADR-012："监督 < 自己做"经济账作为生死假设

```text
新增核心假设：用户"监督 AI 做多变任务"必须比"自己动手做"更省，否则产品是负价值
关键杠杆：上下文积累必须可测量地降低监督成本（AI 越懂你，你越少开口）
          —— 记录/上下文引擎因此从配角升级为"让经济账由负转正的关键"
验证方式（创始人自我 dogfood，不依赖早期用户访谈）：
  - 用它做真实多变日常任务，对比是否真的比自己做快
  - 度量随上下文积累，监督开销是否肉眼可见地下降
  - 关键信号 = 创始人是否自发地天天使用（对自己的留存）
影响：VALID.HYPO 新增假设、VALID.CRITERIA 补充可测指标
```

### ADR-013：8GB 硬件现实驱动分层与双角色资源分配

```text
现实：创始人验证机 = RTX 4060 Laptop / 8GB 显存（普通用户真实中位配置）
后果：
  - 9B 全双工 FP16(~18GB)/INT8(~9GB) 在 8GB 上放不下，仅 INT4(~5-6GB) 勉强
  - 笔记本 4060 带宽低 + 135W 功耗墙 + 与系统显示共享显存 → 实时 200ms 目标存疑
决策：
  1. v0 验证闭环用"能塞进 8GB 的小模型"先行（如 Qwen2.5-Omni 3B），
     旗舰 9B 全双工降级为"高配/未来"档，而非 v0 前置阻塞
  2. 双角色在 8GB 下的资源分配：
     执行大脑 = 实时 GPU 常驻（独占算力，保延迟）
     记录引擎 = 轻量/异步（确定性工程为主 + 按需小模型，可 CPU），不与大脑抢 GPU
  3. 新增"环境自检 → 推荐配置"功能（MODEL.PROBE），把"用户不知道自己能跑什么"这道坎抹平
影响：MODEL.HW 重写、MODEL.RECORD/MODEL.CONFIG 资源分配、新增 MODEL.PROBE、02 阶段分配
```

## 四、关键判断对照

| 校准前表述 | 校准后表述 | 说明 |
|-----------|-----------|------|
| 重复性电脑操作 | 名义重复、实际多变的监督型任务 | 避免落入 RPA 解法空间 |
| 全双工语音是第一优先级 | 语音是升级层，护城河模态无关 | 不把战略押在最难技术上 |
| 与 Codex/Operator 竞争 | 与文本 Agent 配合（多变 vs 固定） | 定位更锐利 |
| 消费级 GPU 可运行 9B 全双工 | 8GB 仅 INT4 勉强；v0 用 3B 小模型先行 | 正视硬件现实 |
| 记录引擎是配角 | 记录引擎是经济账由负转正的关键杠杆 | 上下文降本决定生死 |
| 找早期用户验证方向 | 创始人自我 dogfood + 自留存信号 | 早期不被少数用户带偏 |

## 五、v0 落地指导（不改阶段定义，作为执行提示）

```text
v0 闭环目标：把"语音监督执行闭环"从 0 到 1 立起来（初版糙可接受）
  语音输入 → 执行大脑（8GB 可跑的小模型）→ 可见动作 → 记录/反馈 → 再回到语音
执行大脑：v0 用 Qwen2.5-Omni 3B 起步，并行实测 BayLing-Duplex INT4 能否在 4060 上用
记录引擎：v0 用确定性记录（规则 + JSONL，骨架已有 JsonlSessionStorage），小摘要模型留接口后置
验证方式：创始人拿真实多变任务自我 dogfood，盯"监督<自己做"与"监督成本随上下文下降"两个信号
运行/验证目标机：RTX 4060 Laptop 8GB（DuplexModelProvider 接口保证未来可平滑换大模型/换硬件）
```

