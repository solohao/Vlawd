## 一句话定义

**AI 光标是一个耳机协作式 AI 桌面员工：用户通过耳机实时全双工对话，指挥、监督和纠正 AI 在屏幕上可见地操作电脑，所有操作沉淀为可回溯的 Session，成功流程沉淀为可复用工作流。**

更短的对外表达：

> 用全双工语音指挥一个可见、可控、可纠正的 AI 光标替你操作电脑。

---

## 产品本体（PROD.CORE）

AI 光标的本体不是聊天框、不是浏览器插件、不是语音输入法，而是覆盖在桌面环境上的执行层和协作通道。

```text
用户意图
↓
耳机全双工对话（同时听和说）
↓
AI 理解屏幕与任务
↓
光标 / 键盘 / 浏览器 / 窗口执行
↓
AI 耳机实时汇报，用户随时纠正
↓
Session 记录完整证据链
↓
成功路径沉淀为工作流
```

核心输出：

```text
电脑上的可见动作
+ 执行结果
+ 可回溯的 Session 记录
+ 可复用流程
```

---

## 解决的核心问题

核心痛点不是"固定不变、可被脚本一次性自动化"的硬重复工作——那类任务的天然解
法是录制-回放 / RPA / 固定工作流，不需要实时语音。AI 光标瞄准的是另一类缝隙：

> **名义上重复、但实际内容因变量不断变化、无法预先脚本化、必须有人临场判断的日常任务。**

例如：每天处理"看起来一样、细节每次都不同"的网页操作、资料整理、表单、跨页协同。
这类任务用固定工作流做不了，用户又不想全程亲手做，于是需要一个能被实时监督、可
临场纠正的执行体。

AI 光标要解决的是：

> **让用户不用亲手做这些多变的琐碎电脑操作，只需用语音实时监督和调整细节，同时保留控制权。**

v2 的价值层：

> **让每次操作都留下可回溯的记录，让语音纠正成为自然的分支操作而非错误，让 AI 通过上下文积累越用越懂你——从而让"监督它做"持续比"自己动手做"更省。**

> 注：语音的独特价值在"实时监督式执行"（临场纠正、可打断、信任与控制），
> 而非"省重复"。这是与 RPA / 固定工作流解法空间的根本分界。

---

## 用户心理价值

沿用 v1 的核心心理价值：老板感、解压感、陪伴感、隔离感。

v2 新增：

- **记忆感**："帮我再做一次昨天那个"——AI 能找到历史记录并复现
- **进化感**：AI 越用越懂用户的习惯和偏好
- **安全感**（增强）：全双工对话让打断和纠正更自然，用户更敢放手

## 竞争格局（v2 新增）（PROD.COMPETE）

### vs OpenAI Codex

| 维度 | Codex | AI 光标 |
|------|-------|---------|
| 交互方式 | 文字输入，异步执行 | 全双工语音，实时协作 |
| 执行环境 | 云端沙箱 | 本地桌面 |
| 隐私 | 代码上传云端 | 本地运行，数据不出机 |
| 目标用户 | 开发者 | 所有电脑用户 |
| 多光标 | Mac 私有 API，仅限 macOS | 跨平台（CursorLease 调度） |
| 上下文管理 | 对话记录 | Session Graph（fork/merge） |
| 纠正方式 | 重新输入文字 | 语音自然纠正，自动 fork |

**核心差异：** Codex 是面向开发者的云端异步代码 Agent；AI 光标是面向所有用户的本地实时桌面 Agent。

### 与文本 / 异步 Agent 的关系：配合而非替代

AI 光标与 OpenAI Operator、Manus 等文本/异步 Agent **不是替代关系，而是配合关系**，
按任务性质分工：

```text
固定、可无人值守、可预先脚本化的任务 → 交给文本/异步 Agent
多变、需实时判断与临场监督的任务     → AI 光标的语音监督执行
```

诚实承认这条边界，能让产品定位更锐利，避免"什么都想做"。

### vs Claude Computer Use

| 维度 | Claude Computer Use | AI 光标 |
|------|-------------------|---------|
| 交互 | 文字指令 | 全双工语音 |
| 执行 | 远程虚拟桌面 | 本地真实桌面 |
| 监督 | 事后查看截图 | 实时可见执行 |
| 隐私 | 截图上传云端 | 本地处理 |

### 差异化锚点

AI 光标不应与 Codex/Claude 正面竞争"谁的 AI 更聪明"，而应锚定以下差异：

```text
1. 本地优先隐私架构 — 数据不出机，适合隐私敏感用户和中国市场
2. 实时监督式执行 — 多变任务可临场纠正、可打断，信任与控制感
3. Session 可回溯 — 每次操作都是证据链，纠正是分叉而非覆盖
4. 上下文个性化沉淀 — AI 越用越懂你，持续降低监督成本
5. 中文/非英语市场 — 以 BayLing-Duplex 等中文 full-duplex 执行大脑为核心，GLM-4-Voice 作为语音组件/基座参考
```

> 注意：上述锚点中真正构成护城河的是"实时监督式执行 + Session 可回溯 + 上下文个
> 性化"这层 **agent 工程 + 工作流记忆**，它是**模态无关**的。全双工语音是这层之上
> 的**体验升级层**，而非护城河本身——详见战略定位。

## 历史机会

沿用 v1 的螺旋上升判断。

v2 新增判断：

```text
全双工语音模型成熟（BayLing-Duplex/PersonaPlex 等执行大脑候选，GLM-4-Voice 作为中文语音生态参考）
→ 语音交互从"说完等回复"升级为"边说边做"
→ 这是第一次语音 Agent 的交互体验可以媲美人与人的对话
→ 在此之前没有产品能做到"用户说话的同时 AI 就开始执行"

Session-first 上下文工程成熟（OpenRath 范式）
→ Agent 的记忆从"聊天记录"升级为"结构化证据链"
→ 语音纠正天然是分叉操作，Session Graph 完美匹配
→ 这为语音 Agent 提供了之前不存在的上下文管理能力
```

## 产品边界（PROD.BOUNDARY）

### 它是

- 可监督的桌面执行层（多变任务的实时监督执行）
- 耳机协作式 AI 桌面员工
- Session 驱动的任务记录系统
- 上下文个性化沉淀工具（越用越懂你）
- 文本/异步 Agent 的配合者（多变任务侧）
- 全双工语音控制的 AI 光标（语音作为体验升级层）

### 它不是

- 不替代操作系统
- 不替代浏览器
- 不做纯聊天机器人
- 不做单纯语音输入法
- 不承诺完全无人值守
- 不做企业级复杂 RPA 编排平台
- 不做固定重复任务的脚本化自动化（那交给文本/异步 Agent）
- 不在第一阶段覆盖所有桌面软件
- 不做云端执行（区别于 Codex）

## 战略定位（PROD.STRATEGY）

> **耳机协作式 AI 桌面员工：用户实时监督、口头纠正 AI 在屏幕上可见地完成"名义重复、实则多变"的日常任务，所有操作沉淀为可回溯的 Session，AI 通过上下文积累越用越懂你，让"监督它做"持续比"自己动手"更省。**

### 护城河与模态解耦（核心战略判断）

```text
护城河 = agent 工程 + 工作流/Session 记忆 + 可靠可见执行（模态无关）
全双工语音 = 体验升级层，不是产品成立的前置条件

为什么这样判断：
  - 把战略押在"全双工小模型变强"上，这道技术墙恰恰会被模型厂商自己填平
  - 真正壁垒是工作流数据网络效应、流程沉淀锁定、可监督执行的工程成熟度
  - 这些壁垒不在乎输入是语音还是文本，可以先用最简模态积累
```

第一阶段的战略选择：

```text
不争 AI OS
不争浏览器入口
不做万能 Agent
不做云端执行
不做固定重复任务的脚本化自动化（与文本/异步 Agent 配合）
先做本地桌面上的可监督执行层（多变任务）
用真实代劳体验建立价值，用 Session 记录与上下文个性化建立粘性
护城河照建，但不让"全双工语音优先"绑架产品节奏；语音作为升级层按需叠加
```

> 经济账生死线：用户"监督 AI 做多变任务"必须比"自己动手"更省，否则产品是负价值。
> 让这道账由负转正的关键杠杆 = 上下文积累可测量地降低监督成本（详见增长验证/01 VALID.HYPO）。

## 与 Agentic Web 项目群的关系（PROD.AGENTIC）

Vlawd 是 Agentic Web 项目群中的一个独立产品，不是整个网络本身：

```text
Vlawd：
  全双工个人终端
  本地 Agent Runtime
  可见执行和安全监督
  Session / Evidence 的产生端

Agentic Web：
  长期愿景
  跨项目分层和依赖
  Capsule / Living Project 共享对象
  Registry、平台和垂直项目阶段门
```

两者分别进行版本管理：

```text
Vlawd 产品：
  AI_cursor_v1 → AI_cursor_v2 → 未来真正的 AI_cursor_v3

项目群蓝图：
  Agentic_Web v1.0 → v1.1 → v2.0
```

当前规则：

- `AI_cursor_v2/` 是 Vlawd 产品定位、交互、Runtime、Session、UI、MVP 和代码的唯一权威来源；
- `Agentic_Web/` 只定义 Vlawd 在项目群中的层级、依赖和后续阶段门；
- V2 可以产生未来 Capsule 所需的 Session / Evidence，但当前不实现完整协议、Registry 或开放平台；
- 项目群蓝图升级不会自动触发 Vlawd 产品主版本升级；
- 只有 Vlawd 自身发生不可兼容的产品或架构替代时，才建立真正的 `AI_cursor_v3/`。

这使 V2 可以继续围绕同一产品目标做深入迭代，同时避免被长期生态规划拖入多项目并行开发。

---

## 一句话总结

**AI 光标 v2 的战略是"全双工语音 + 本地隐私 + Session 可回溯"三位一体：全双工让交互延迟降到 200ms，本地运行保护用户隐私，Session Graph 让语音纠正成为自然的分叉操作。这三个锚点共同构成了与 Codex/Claude 的结构性差异——它们做云端异步文字 Agent，我们做本地实时语音 Agent。**
