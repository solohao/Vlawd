## 二、已完成：生产悬浮窗交互改造（均已合并 main）

对应文件：
- `AI_cursor_v2/app/packages/main/src/electron/main.ts`（BrowserWindow + IPC）
- `AI_cursor_v2/app/packages/main/src/electron/preload.ts`
- `AI_cursor_v2/app/packages/renderer/src/app/desktop-api.ts`
- `AI_cursor_v2/app/packages/renderer/src/ui/runtime/OverlayApp.tsx`
- `AI_cursor_v2/app/packages/renderer/src/app/index.css`

已解决的问题与做法：
1. **可拖动到桌面任意位置**（PR #56 初版 → #57 → #58 定稿）。
   - 最终方案（参考 Open-LLM-VTuber）：左键 `pointerdown` → `setPointerCapture` + `setInteractive(true)` + IPC `overlay:dragStart`；**主进程**用 `screen.getCursorScreenPoint()` 每 16ms `setPosition` 跟随光标；`pointerup` → `overlay:dragEnd`，位移 < 4px 视为点击。
   - 关键坑（#56/#57 失败原因）：拖动若依赖 renderer 的 `mousemove`，光标一滑出吉祥物不透明区就会 hover→重新穿透，丢失后续事件，拖不动也点不了。改由主进程光标跟随后不再依赖转发事件。
2. **命中范围只限吉祥物不规则多边形**：主进程默认 `setIgnoreMouseEvents(true, { forward: true })` 整窗穿透；renderer 把 sprite 的 alpha 预绘到离屏 canvas，`mousemove` 时逐像素 alpha 命中检测（阈值 `ALPHA_HIT=24`），命中才 `overlay:setInteractive(true)`。**拖动期间短路 hover 切换**（`if (dragRef.current) return`）。
3. **去掉 hover 出的两个按钮**：删除原 `OverlayMini` 的齿轮/展开按钮。
4. **点击=真实暂停/继续**：左键点击（未拖动）调 `conversationPreempt("pause")` / `conversationResume()`；`paused` 来自 `conversationSnapshot().paused` 与 `preemption`/`snapshot` 事件。是真实 duplex-runtime 暂停，不只是视觉。
5. **背景曲线阴影**：移除 sprite `img` 及 `paused` 态的 `drop-shadow`。
6. **面板与悬浮窗职责重复**：取消“点击→展开面板”，面板改为**右键**（`onContextMenu`）按需唤出，作次级控制入口（取消/接管/设置）；吉祥物本体是唯一主入口。

相关 IPC：`overlay:setInteractive`、`overlay:dragStart`、`overlay:dragEnd`、`overlay:getBounds`、`overlay:resize`。

相关 PR：#56、#57、#58（均已 merge）。验证命令：`npm run typecheck` / `npm test`（63 passed）/ `npm run build` / `npm run build:renderer`。
注意：客户端拖动/穿透/点击暂停需在 **Windows 打包版**实测（`setIgnoreMouseEvents(forward:true)` 转发在 Win/mac 良好、Linux 可能受限）。用户环境 Node 20.18.1，Vite 8 建议升级到 20.19+/22.12+。

## 三、待决策：精灵动画质量 & 可插拔头像（核心议题）

用户痛点：现在动画太丑、想要 Live2D 级质量但嫌成本高；且精灵只有眼睛+嘴，担心套 Live2D 模板难做表情；并考虑未来开源（本地产品付费点弱、需网络效应）。

### 已给出的分析结论

1. **别上 Live2D 做自家极简图标**：Live2D 强项是“给多部件人形插画绑骨骼网格”，我们只有眼睛+嘴、模板也是人形，拓扑不匹配，性价比最差，且 **Cubism SDK 商用许可有条件**（对商业化 OSS 是法律风险）。
2. **“只有眼睛+嘴做不出表情”是误解**：Memoji 笑脸豆、Google Assistant 点、Siri 球、Duolingo、emoji 都只有眼睛+嘴。灵动靠**运动设计**（眼型/嘴型曲率表情、眨眼 squash、扫视+凝视时序、呼吸/微摆、看向光标），不靠器官多。
3. **自家极简吉祥物的最佳引擎 = Rive**（非 Live2D、非 Lottie）：矢量 + **状态机/输入参数**可实时驱动（看向/口型/打断），运行时小，authoring 成本远低于绑骨骼；Lottie 是预烘焙时间线、难做连续实时驱动。
4. **“人人接入自己喜欢的头像”方向对，但不是二选一**：它把美术负担外包给现成生态（Live2D/VRM/VRoid/VTuber 模型），Open-LLM-VTuber 已有先例（Live2D + 状态/表情映射 + 口型同步）。但它是**换掉**极简限制（换成完整人形绑定），而非解掉——所以是两条路线：极简自有吉祥物(品牌) vs 可插拔头像(个性化+社区)。
5. **推荐“都要”**：默认自带第一方吉祥物（开箱即用、品牌脸）+ 支持可插拔头像（进阶/社区）。落地关键是先抽一层 **Avatar Provider 接口**（统一 `setState / lipSync(level) / lookAt / blink`），背后可切换：内置吉祥物 / Rive / Live2D / VRM。美术决策与管线解耦，先出默认版，以后加插拔头像不返工。
6. **许可**：优先 **VRM**（glTF、完全开放，许可干净），同时兼容 **Live2D**（生态最大），并把许可写清楚。
7. **开源/网络效应**：可插拔头像是社区飞轮杠杆（头像库/画廊 + skills/playbook 共享）。本地 OSS 常见变现：云同步/云大脑可选、团队版(审计/SSO/管理)、托管模型、市场分成、支持/SLA。

### 方案对比（针对“只有眼睛+嘴”）

| 方案 | 质感上限 | 实时可控 | 制作成本 | 运行体积 | 适配我们 |
|---|---|---|---|---|---|
| 手写 SVG 程序动画（现状之上打磨） | 中低 | ★最好 | 低 | 极小 | 能用但难好看 |
| Lottie | 高 | ✗ 预烘焙 | 中 | 中 | 只适合离散状态 |
| **Rive** | 高 | ★ 状态机+输入 | 中低 | 小(WASM) | **最契合默认吉祥物** |
| Live2D | 最高 | ★ | ✗✗ 最高 | 中 | 自家图标不划算；作可插拔头像可选 |
| VRM(3D) | 高 | ★ | 外包生态 | 中 | 可插拔头像**首选**(许可干净) |
