## 一、当前产品目标与阶段纪律

Vlawd 的长期目标是成为一个“AI Cursor / AI Desktop Employee”：

```text
用户意图
→ 耳机全双工对话
→ AI 理解语音、屏幕与任务上下文
→ AI 在受控范围内执行
→ 用户可插话、纠正、中断、接管
→ Session 保存完整 Evidence
→ 成功路径沉淀为可复用工作流
```

当前优先级是 **Cycle 1：真实全双工入口**，不是 Computer Use，也不是完整 Pi 复刻。

### Cycle 1 权威验收标准

> 用户通过真实麦克风与一个固定 Provider 对话，在 AI 说话途中自然插话改变约束；AI 停止原回答，并依据新约束继续。

目标数据链：

```text
真实麦克风
→ VAD / endpointing
→ 流式或 Worker ASR
→ 用户文本进入 Duplex Runtime
→ LLM 流式输出
→ 流式 TTS
→ 音频播放
→ 用户插话时 flush TTS/audio queue + abort generation
→ Runtime 事件同步 Overlay
→ 记录延迟和打断指标
```

### 不能违反的边界

1. Cycle 1 只验证语音链路，不启用真实浏览器、鼠标、键盘或窗口执行器。
2. 不能用 Echo/Mock Provider 宣称真实 Cycle 1 通过。
3. 不能把浏览器云识别静默当作本地语音闭环。
4. 高风险操作未来仍需受控确认、可取消、可审计。
5. 参考仓库只借鉴架构、接口和实现方法，不修改参考仓库，也不整包复刻。

## 三、参考仓库：代码结构与借鉴结论

### 3.1 Alice：Electron 语音助手整体参照

参考目录：`/home/ubuntu/repos/Alice`

核心可借鉴点：

- `src/modules/conversation/turnManager.ts`
  - LLM 流式 delta 直接追加到 assistant placeholder。
  - abort 后保留已经产生的半截文本。
  - 通过 generation / temp id 防止错误更新其他回合。
- `src/modules/conversation/chatOrchestrator.ts`
  - LLM AbortController 独立管理。
  - OpenAI Responses API 保留 `currentResponseId`，用 `previous_response_id` 继续会话。
  - 非 OpenAI Provider 使用本地完整历史。
- 语音链路
  - VAD 的 `onSpeechStart` 先于转写触发打断。
  - 文本按句切分后进入 TTS 队列。
  - LLM 与 TTS 的取消应分开处理。

对 Vlawd 的结论：

- Alice 是 Electron/TypeScript 同栈整体结构的首要参照。
- 只借“partial 保留、Abort、generation、队列 flush、会话续接”的思想。
- Vlawd 当前使用本地历史，不依赖 OpenAI `previous_response_id`。

### 3.2 ricky0123/vad：VAD 与 endpointing 调参参照

参考目录：`/home/ubuntu/repos/vad`

核心借鉴点：

- `onSpeechStart` 是最低延迟 barge-in 入口，不应等待 ASR。
- `redemptionFrames` 控制静音宽限时间。
- `minSpeechFrames` 过滤咳嗽、点击和短噪声。
- `preSpeechPadFrames` 避免吞掉句首。
- VAD 模型资产路径在 Electron `file://` 打包时需要单独验证。

当前 Vlawd 暂时没有直接接入 `@ricky0123/vad-web`，而是使用自定义能量 VAD + PCM 采集，原因是先减少 ONNX/WASM 资产和运行时路径变量。后续可将两者做 A/B 测试，不应仅凭类型检查决定最终方案。

### 3.3 whisper-web / transformers.js：Worker ASR 参照

参考目录：

- `/home/ubuntu/repos/whisper-web`
- `/home/ubuntu/repos/BrowserAI`

核心借鉴点：

- Whisper pipeline 放在 Web Worker，避免阻塞 React UI、精灵动画和音频播放。
- pipeline 使用 singleton，避免每次说话重复加载模型。
- 支持多语言、`language`、`quantized` 和 `transcribe/translate` 参数。
- 模型文件加载应有逐文件进度，首次下载是可感知状态，不应假装即时可用。
- BrowserAI 证明 renderer 可在 WebGPU 下把 STT、TTS、LLM 都放到端上，但其默认模型/语音偏英文，中文方案必须单独验证。

当前 Vlawd 已实现：

- `packages/renderer/src/runtime/whisper-worker.ts`
  - Worker 内使用 `@huggingface/transformers`。
  - 默认模型 `Xenova/whisper-tiny`。
  - 优先 WebGPU，失败回退 WASM。
  - 使用请求 id 返回结果。
- `packages/renderer/src/runtime/audio-io.ts`
  - `WhisperTranscriber` 主线程包装器。
  - Worker lifecycle、pending Promise、progress、error、dispose。

需要继续验证：

- Electron 打包后的 module Worker 与 `file://` 路径。
- 首次模型下载、缓存位置、断网重启。
- 中文识别准确率、首句延迟、长音频内存占用。
- WebGPU/WASM fallback 的用户可见状态。

### 3.4 RealtimeVoiceChat：自然打断与 generation 可靠性

参考目录：`/home/ubuntu/repos/RealtimeVoiceChat`

核心借鉴点：

- 用句子完成度模型动态决定 endpoint 等待时间，不只依赖固定静音时长。
- 用 generation counter / id 丢弃被打断后迟到的 LLM、TTS、音频输出。
- LLM、快速 TTS、最终 TTS 分级 abort。
- quick answer-first 可降低首音延迟。

对 Vlawd 的落地顺序：

1. 先用当前 VAD 静音 endpoint 跑通。
2. 再增加 generation epoch 检查，确保旧回合输出不能写入新回合。
3. 最后评估句子完成度模型，避免过早抢话。

### 3.5 pipecat：事件帧与防自打断

参考目录：`/home/ubuntu/repos/pipecat`

核心借鉴点：

- `UserStartedSpeakingFrame`、`UserStoppedSpeakingFrame`、`BotStartedSpeakingFrame`、`BotStoppedSpeakingFrame` 等 typed frame。
- interrupt 是高优先级系统事件，可越过普通数据队列并 flush 下游 TTS。
- `UserMuteStarted/Stopped` 用于播放期间门控，防止机器人外放被识别成用户。

对 Vlawd 的落地结论：

- shared runtime event 应逐步补齐 user/bot speaking start/stop 与高优先级 interrupt 语义。
- `echoCancellation`、`noiseSuppression`、`autoGainControl` 只是第一层防护；耳机、扬声器、系统混音仍需真机测试。
- 如果能可靠使用 VAD mute/window gating，应与浏览器回声消除组合，而不是二选一。

### 3.6 Open-LLM-VTuber：打断历史的正确语义

参考目录：

- `/home/ubuntu/repos/Open-LLM-VTuber`
- `/home/ubuntu/repos/Open-LLM-VTuber-Web-src`

核心借鉴点：

- WebSocket 将 `CONVERSATION`、`CONTROL`、`DATA` 分层。
- `interrupt-signal` 取消当前 conversation task。
- 打断时保存的是用户实际听到的 assistant 文本，不是模型已经生成但尚未播放的全部文本。
- 追加 interrupted 标记，下一轮模型知道上一轮没有说完。
- ASR、TTS、VAD、agent 都是可替换模块。

当前 Vlawd 已合并的对应实现：

- renderer `TtsPlayer` 只在 `SpeechSynthesisUtterance.onend` 后累计 `spokenText`。
- VAD 开口时先 `tts.cancel()`，再经 IPC 报告 `conversationBargeIn(heardText)`。
- main runtime 在 finalize 时将 interrupted assistant turn 截断为 heard text。
- `DuplexHistoryTurn.interrupted` 进入下一轮上下文。
- Pipeline Provider 将 interrupted assistant 文本追加“说到这里被用户打断”的提示。

### 3.7 Mastra：统一 Voice 抽象

参考仓库：`/home/ubuntu/repos/mastra`

可借鉴的接口分层：

- 回合式：`speak(text)`、`listen(audioStream)`。
- 实时式：`connect()`、`send(audioData)`、`answer()`、`on(event)`。

对 Vlawd 的建议：

- Cycle 1 先维持 `listen → runtime → speak` 的 pipeline。
- shared 层预留统一 Voice Adapter，未来切换原生全双工时不改上层 Session/Overlay。
- 不要为了抽象提前引入完整 Mastra runtime；只抽最小接口。

### 3.8 Supertonic / SummerTTS：TTS 组件候选

参考目录：

- `/home/ubuntu/repos/supertonic`
- `/home/ubuntu/repos/SummerTTS`

结论：

- Supertonic：优先评估的端上 ONNX TTS，MIT，多语言，适合替换当前 `SpeechSynthesis`。
- SummerTTS：MIT、离线中文兜底，可作为 CLI/N-API sidecar，但音质和工程接入成本需评估。
- 任何模型权重、音色、运行时都必须单独核对 license 与体积。

### 3.9 其他项目：当前只作设计/基础设施参考

- `livekit/agents`：实时轮次、端点与打断设计；Python，不能直接搬进 renderer。
- `vocode-core`：模块化语音 agent；与 LiveKit/pipecat 设计重叠。
- `LocalAI`：可提供 OpenAI 兼容的本地 STT/LLM/TTS 服务，适合作为 sidecar 备选，不是当前端上优先路线。
- `huggingface/speech-to-speech`：本地 STT→LLM→TTS 管线与模型选型参考。
- `VoltAgent/voltagent`：Agent 编排、工具和可观测更有价值，暂不作为 Cycle 1 语音核心。
- `injaneity/pi-computer-use`、Pi CLI、CUA 项目：只借“少量原子工具、observe→act→observe、结构化协议、风险闸门”思路，留给 Cycle 2/Phase 2。

## 四、当前代码结构与已经实现的闭环

### Main process

- `packages/main/src/runtime/duplex-runtime.ts`
  - 会话状态、turn history、generation cancel、preemption、latency。
  - 支持 `bargeIn(heardText?)`。
  - interrupted assistant turn 进入下一轮 history。
- `packages/main/src/model/pipeline-duplex-provider.ts`
  - OpenAI-compatible streaming LLM。
  - `AbortSignal` 取消。
  - 将 interrupted history 映射为带中断说明的 assistant message。
- `packages/main/src/electron/main.ts`
  - conversation IPC handlers。
  - `conversation:bargeIn` 接收 renderer 上报的 heard text。
- `packages/main/src/electron/preload.ts`
  - 暴露 typed conversation API。

### Renderer

- `packages/renderer/src/runtime/audio-io.ts`
  - `MicVad`：麦克风、RMS VAD、endpointing、16kHz mono PCM。
  - `TtsPlayer`：句子边界、SpeechSynthesis queue、flush/cancel、spoken text。
  - `BrowserSpeechRecognizer`：旧浏览器识别 fallback。
  - `WhisperTranscriber`：Worker wrapper。
- `packages/renderer/src/runtime/whisper-worker.ts`
  - transformers.js Whisper worker。
  - WebGPU→WASM fallback。
- `packages/renderer/src/runtime/useConversation.ts`
  - VAD、Whisper、TTS、IPC 和 Runtime event 的编排。
  - VAD 开口立即 barge-in。
- `packages/renderer/src/app/desktop-api.ts`
  - renderer 侧 API 类型。

### Shared

- `packages/shared/src/types/model.ts`
  - `DuplexHistoryTurn`
  - `DuplexModelInput.history`
  - runtime/provider 事件类型。

### 当前闭环状态

```text
MicVad onSpeechStart
  → 立即停止当前 SpeechSynthesis
  → conversation:bargeIn(已播完句子文本)
  → main AbortSignal 取消 LLM

MicVad onSpeechEnd(Float32Array)
  → Whisper Worker
  → submit(transcript)
  → duplex-runtime

assistant_delta
  → TtsPlayer.feed()
  → 句子边界进入 SpeechSynthesis

assistant_end
  → 正常回合 flush()
  → interrupted 回合 cancel()
```

### 已验证

在 `AI_cursor_v2/app/`：

- `npm run typecheck`：通过。
- `npm test`：74 tests 通过。
- `npm run build:renderer`：通过，Whisper worker 与 ONNX WASM 正确分包。

已知环境提示：

## 五、当前正在处理的问题

### 5.1 语音链路还没有完成真机验收

必须在真实 Electron 打包环境验证：

- 麦克风权限与耳机输入。
- 16kHz PCM 是否真实成立，浏览器重采样是否符合预期。
- VAD 阈值、trigger、redemption、min speech 的中文环境调参。
- Whisper 首次模型下载、缓存、断网启动。
- WebGPU 失败时 WASM 回退是否能工作。
- TTS 外放是否触发自打断。
- AI 说话中自然插话时是否在目标延迟内停止。

### 5.2 当前 TTS 仍是 SpeechSynthesis

它能验证队列/中断语义，但不是最终的本地低延迟流式 TTS：

- 浏览器系统语音的音色和中文质量不可控。
- `SpeechSynthesisUtterance.onend` 只能按句确认“听到”，不能提供精确播放进度。
- 还没有 Supertonic / Kokoro / SummerTTS 的音频 buffer 播放队列。

### 5.3 浏览器云识别 fallback 的隐私与愿景问题

`webkitSpeechRecognition`：

- 是浏览器提供的识别 API，不是 Vlawd 自己申请的 Google Cloud API。
- 一般不需要用户填写 API Key，但底层服务、网络和数据处理由浏览器/厂商控制。
- Electron 中通常不可用或行为不一致。
- 默认静默启用会与“本地优先、隐私可控”的产品愿景冲突。

产品原则：

1. Whisper 本地模型是默认主链。
2. 浏览器云识别如保留，只能是用户显式开启的实验性兼容选项，并明确标注“音频可能上传云端”。
3. Whisper 不可用时，优先退回文字输入，而不是静默切换到云。
4. UI 必须展示当前 STT 引擎：`本地 Whisper / 云端浏览器识别 / 纯文字`。

### 5.4 模型越来越多，不能把原始模型列表直接暴露给普通用户

现在涉及：

- STT/ASR：Whisper、sherpa-onnx、Paraformer、SenseVoice 等。
- LLM：Execution Brain / Conversation Brain / Record Notebook。
- TTS：SpeechSynthesis、Supertonic、Kokoro、SummerTTS 等。
- 原生全双工模型与传统 `STT → LLM → TTS` pipeline。

如果把这些直接做成一张长列表，用户必须理解模型、语言、硬件、延迟、隐私、质量和架构，认知负荷过高。

## 六、模型中心与配置产品方向

### 6.1 不建议现在用 n8n 式无限画布做模型配置

n8n 画布适合高级用户编排任务步骤，不适合首次配置模型。用画布连接“听/想/说”会把简单选择变成一套新的节点语言。

建议分离：

- **模型中心**：选择和安装模型能力。
- **工作流画布**：未来编排搜索、浏览、执行等任务步骤。

### 6.2 推荐渐进式暴露

#### 第一层：一键就绪

启动时检测：

- CPU / RAM / GPU / VRAM。
- 系统与用户语言。
- 本地磁盘空间。
- WebGPU/WASM 能力。
- 是否允许联网下载。

然后自动选择整套语音配置，用户不用理解具体模型。

#### 第二层：面向结果的预设模式

建议预设：

- 极速模式：低资源、低延迟。
- 均衡模式：默认推荐。
- 高质量模式：更高 RAM/VRAM 和更高延迟。
- 完全本地模式：隐私优先。
- 中文优先模式：中文识别与合成优先。

用户切换的是“体验目标”，不是四个孤立模型。

#### 第三层：高级槽位配置

折叠展示：

```text
听：STT/ASR
想：LLM
说：TTS
```

每个槽位的模型按推荐分排序，并显示：

- 支持语言。
- 最擅长任务。
- 设备要求。
- 下载大小。
- 本地/云端。
- 推荐/可选。
- 当前是否已安装。

不兼容模型默认隐藏，不让用户先看到错误选项。

### 6.3 与现有 Ollama 模型中心对齐

已有结构可复用：

- `EnvironmentProbe`：环境检测和推荐理由。
- `ModelCatalogEntry`：展示名、徽章、能力、大小、内存要求。
- `ModelPullProgress`：下载阶段、进度和错误。
- `ModelStorageConfig` / Ollama models directory：用户指定存储位置。

STT/TTS 模型中心应复用同样的概念，但不要强行复用 Ollama 的下载协议。transformers.js 侧需要评估：

- `env.cacheDir`
- `env.localModelPath`
- `env.allowRemoteModels`
- `env.allowLocalModels`
- `env.useFSCache`

### 6.4 推荐系统

推荐分可按以下维度计算：

```text
推荐分 =
  设备匹配 × 语言匹配 × 任务匹配 × 隐私偏好 × 延迟偏好 × 已安装加分
```

最低可行版本：

- 低内存/无 GPU：Whisper tiny 量化。
- 有 GPU/较大内存：Whisper base 或更高质量模型。
- 中文优先：多语言 Whisper 或经过验证的中文 STT。
- 完全离线：隐藏浏览器云识别，优先本地 STT/TTS/LLM。

不要只依据模型名字推荐；必须做设备 probe、模型元数据和真实健康检查。

## 七、下一阶段执行计划

### P0：真机验证（优先于扩展模型列表）

1. 在真实 Electron 环境打开麦克风。
2. 下载并缓存 Whisper tiny。
3. 验证中文短句、连续多轮、耳机播放。
4. 记录：
   - VAD 开口到 TTS 停止。
   - VAD 结束到 ASR 文本。
   - LLM 首 delta。
   - 首个可听 TTS。
5. 验证打断后上下文只保留已听到文本。
6. 产出真实测试报告；不能用 Echo provider 代替。

### P1：补齐语音引擎状态与 fallback 可见性

- shared/runtime 增加当前 STT/TTS 引擎状态。
- Overlay/工作台展示“本地 Whisper / 云端浏览器识别 / 纯文字”。
- 云端 fallback 改为显式 opt-in。
- Whisper 下载、加载、失败和回退给出用户可理解的提示。

### P2：Generation epoch 与音频队列可靠性

- 给每次 generation 绑定递增 id/epoch。
- 旧 generation 的迟到 `assistant_delta`、TTS 音频和完成事件全部丢弃。
- 将 `SpeechSynthesis` 替换为可控的 Web Audio buffer queue 做技术验证。

### 10.3 UI 复刻实施原则

1. **先复刻结构和信息层级，再做组件抽象**：优先让页面完整表达概念图中的功能，不先追求大规模设计系统重构。
2. **视觉参考不是静态图片替换**：卡片、状态、下载进度、模型选择、架构切换和语音入口都必须连接真实状态或明确的占位状态。
3. **模型复杂性继续渐进式暴露**：普通用户看到一键就绪和预设模式；高级槽位默认折叠；不兼容模型默认隐藏。
4. **工作台与模型中心职责分离**：工作台负责“现在能否开始工作”，模型中心负责“如何配置模型”；不要把 n8n 式无限画布提前塞进模型配置。
5. **先做浅色 V2 版本**：当前两张参考图都是 V2 浅色全彩版本，深色版和响应式细节后续再规划。
6. **复刻顺序**：
   - 第一阶段：静态结构、导航、卡片、信息层级和概念图视觉；
   - 第二阶段：接入当前 Model Center / Duplex Runtime 状态；
   - 第三阶段：接入模型推荐、下载、存储和真实语音引擎状态；
   - 第四阶段：在真实 Electron 环境验证窗口尺寸、滚动、权限提示和全双工入口。

### 10.4 当前明确不做

- 不把概念图当成已完成的生产功能；
- 不在这一步直接引入 n8n 式无限画布；
- 不为了视觉复刻修改 Cycle 1 的验收边界；
- 不在没有真实后端状态时伪造“本地运行中”或“模型已就绪”。
