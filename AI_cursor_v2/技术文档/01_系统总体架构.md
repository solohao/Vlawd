## 架构结论

AI 光标 v2 采用**两角色架构**：

```text
全双工模型（大脑）= 交互 + 理解 + 决策 一体化
记录引擎（笔记本）= Session Graph + 任务面板 + 长期记忆
```

对应的技术系统是：

```text
Local-first Desktop App
+ Full-Duplex Voice Model（全双工语音模型）
+ Structured Action Protocol（结构化动作协议）
+ Executor Layer（执行器层）
+ Session-first Context Engine（Session 驱动的上下文引擎）
+ Observation Layer（屏幕感知层）
+ Safety Policy（安全策略层）
+ Personalization Layer（可监督自配置层）
+ AI-maintainable Modular Architecture（面向 AI 可维护架构）
```

## 一句话技术定义

**AI 光标的技术系统是一套以全双工语音模型为大脑、以 Session Graph 为记忆、以结构化动作协议为核心、以执行器为手脚、以安全策略为边界的桌面 Agent 执行架构。全双工模型同时完成听、说、理解和决策；记录引擎把一切交互和执行沉淀为可回溯、可分叉、可查询的 Session。**

| 维度 | V1 架构 | V2 架构 |
|------|---------|---------|

| 记忆能力 | 短期 context window | 持久 Session + 长期 Memory |

## 两角色架构总览（ARCH.ROLE）

```text
用户音频 → 全双工模型 → 同时输出两路：
                        │
                        ├→ 语音流 → 耳机（"好的，正在打开"）
                        │
                        └→ 动作指令流 → 执行器 → 操作桌面
                                         │
                                         ↓
                              记录引擎 ← 监听一切 → 右下角任务面板
```

### 角色 1：全双工模型（大脑）（DUPLEX.MODEL）

全双工模型是 AI 光标的核心大脑，同时完成：

```text
听用户说话 → 理解意图
决定执行什么 → 下发动作指令
同时生成语音反馈 → 耳机播报
处理打断/纠正 → 实时调整执行
```

它不是三个独立模型，而是**一个模型同时完成理解 + 决策 + 语音输出**。在输出端接执行器，把模型输出的指令翻译成真实的鼠标键盘动作。

**关键特性：**

- 端到端语音处理，无需 ASR/TTS 中间环节
- 原生支持打断、纠正、附和等全双工交互
- 延迟 < 200ms（轮替 170ms，打断 240ms）
- 同时输出语音流和动作指令流

### 角色 2：记录引擎（笔记本）（SESSION.ENGINE）

记录引擎不是一个神经网络模型，而是一个**结构化记录系统**，负责：

```text
记录全部交互和执行过程 → Session chunks
维护任务树/拓扑视图 → Session Graph
提供历史查询（"昨天做过什么"） → Memory recall
右下角面板渲染 → 日志/树/拓扑三视图
```

它可以用轻量模型（3B）或规则引擎实现，核心是结构化数据管理而非智能推理。

### 角色 3：执行器（手脚）（EXEC.DUAL）

执行器接收全双工模型的动作指令，翻译为真实操作。v2.1 采用**双执行器架构**：

```text
系统执行器（System Executor）：
  通过操作系统 API 操作系统光标和键盘
  用于非浏览器的桌面应用
  需要 CursorLease 排他调度

内置浏览器执行器（BrowserView Executor）：
  通过 CDP / DOM API 操作内置 BrowserView
  不需要系统光标 → 多个 BrowserView 可真正并行
  每个 BrowserView 有独立执行器实例
  可选渲染虚拟光标动画
```

内置浏览器执行器是 v2.1 的核心新增能力——大部分日常操作（网页、Web 应用、在线表单）都在内置 BrowserView 中完成，实现真正的多任务并行。系统执行器作为后备，处理桌面原生应用操作。

执行器不需要 AI 能力，是纯工程实现层。

## 总体运行闭环

```text
用户音频/快捷键
↓
全双工模型
├→ 语音反馈 → 耳机
├→ 动作指令（原子动作协议）→ Safety Policy → Execution Scheduler
│                                              ↓
│                          ┌─────────────────────────────────────┐
│                          │ 根据 target_view 选择执行器：        │
│                          │  system → 系统执行器（系统光标）      │
│                          │  browser_view_X → BrowserView 执行器 │
│                          └─────────────────────────────────────┘
│                                              ↓
│                                      Observation Feedback
│                                              ↓
└→ 接收观察结果 → 调整下一步
                    ↓
              记录引擎 ← 监听全部数据流（两路执行器）
              ├→ Session Graph（结构化存储）
              ├→ 任务面板（实时渲染）
              └→ Memory Store（长期记忆）
```

## 核心技术目标

### 1. 全双工实时交互

```text
用户可以随时说话
AI 正在说话时可以被打断
用户纠正后 AI 立刻调整
整个过程延迟 < 200ms
```

### 2. 可见执行与真多光标

AI 的所有动作在屏幕上可见发生，支持三种可见性模式：

```text
visible_system   — 系统光标可见执行（桌面应用）
visible_virtual  — 虚拟光标动画 + CDP/DOM 执行（内置 BrowserView）
silent           — 静默执行，无光标动画（后台任务）
```

内置 BrowserView 实现真多光标：

```text
BrowserView A — 虚拟光标 A 在操作
BrowserView B — 虚拟光标 B 在操作
两个真正并行，互不干扰
```

### 3. Session 可回溯

每次交互和执行都记录为 Session chunk：

```text
用户纠正 → fork 新分支（旧分支保留）
用户说"回到刚才" → merge 回之前的分支
所有历史路径都可回溯
```

### 4. 可监督安全

```text
动作前目标高亮
高风险确认
可暂停/继续/取消
Session 完整留痕
回退和错误恢复
```

### 5. 低成本本地运行

```text
全双工模型：本地 GPU（10-16GB 显存）
记录引擎：本地 CPU/GPU（2-4GB）
执行器：纯本地系统 API
隐私数据不出本机
```

## 分层架构（ARCH.LAYER）

### 1. Interface Layer（用户入口层）

- 全局快捷键
- 耳机音频输入/输出
- 悬浮状态窗
- 屏幕标号 overlay
- 右下角任务面板（日志/树/拓扑三视图）
- 设置与偏好面板

### 2. Full-Duplex Model Layer（全双工模型层）

替代原来的 Voice Interaction Runtime + Intent Router + Planning Layer：

```text
用户音频 → 全双工模型 → 语音流 + 动作指令流
```

全双工模型一体化处理：
- 语音理解（替代 VAD + ASR）
- 意图识别（替代 Command Router）
- 任务规划（替代 Planning Layer）
- 语音生成（替代 TTS Controller）
- 打断/纠正处理（原生支持）

关键要求：

```text
安全控制指令（停/暂停/继续）仍由本地规则引擎抢占处理
全双工模型负责复杂理解和决策
两者通过优先级机制协调
```

### 3. Context & Observation Layer（屏幕感知层）

沿用 v1 设计，负责理解当前电脑状态：

```text
Screenshot + OCR + DOM + Accessibility Tree + Window State
→ 候选目标列表
→ 供全双工模型决策
```

### 4. Action & Safety Layer（动作与安全层）

全双工模型输出原子动作协议（六个命名空间：pointer/keyboard/clipboard/window/overlay/webview），经安全策略评估后执行：

```text
全双工模型输出 → ActionProposal（原子/序列/复合）
→ SafetyPolicy 评估
→ ExecutionScheduler
→ 根据 target_view 选择执行器：
   system → 系统执行器（系统光标，排他）
   browser_view_X → BrowserView 执行器（CDP/DOM，可并行）
```

详见 [技术文档/02_观察动作执行与留痕](./02_观察动作执行与留痕.md)

### 5. Session & Trace Layer（Session 与记录层）

替代原来的 Trace & Workflow Layer，升级为 Session-first 设计：

```text
Session Graph（fork/merge/lineage）
+ Session chunks（时间线记录）
+ Memory Store（长期记忆）
+ 任务面板（三视图渲染）
```

详见 [技术文档/06_Session上下文工程](./06_Session上下文工程.md)

### 6. Personalization Layer（可监督自配置层）

沿用 v1 设计，详见 [技术文档/07_个性化与工作流系统](./07_个性化与工作流系统.md)

### 7. Architecture Governance Layer（架构治理层）

沿用 v1 设计，新增 Session Context 作为 bounded context，详见 [技术文档/08_工程架构与AI可维护性](./08_工程架构与AI可维护性.md)

## 运行时事件流

```text
UserAudioReceived
→ FullDuplexModelProcessing
  ├→ VoiceResponseStreaming（语音反馈）
  └→ ActionCommandGenerated（动作指令）
→ SafetyChecked
→ ActionExecuted
→ ObservationCaptured
→ SessionChunkRecorded
→ SessionGraphUpdated
→ TaskPanelRefreshed
→ NextActionSelected / TaskFinished / UserConfirmationRequired
```

特殊事件流 — 用户打断：

```text
UserInterruptDetected（用户开口打断）
→ LocalRuleEngine 抢占（停/暂停/继续 等安全控制词）
→ FullDuplexModel 停止当前语音输出
→ FullDuplexModel 理解新指令
→ SessionForkCreated（记录引擎 fork 新分支）
→ ActionAdjusted
```

## 本地与云端分工原则（ARCH.LOCAL）

### 本地处理（全双工模型 + 规则引擎 + 执行器）

```text
全双工语音交互（听、说、理解、决策）
安全控制词抢占（停/暂停/继续）
鼠标键盘执行
屏幕感知（截图/OCR/DOM）
Session 记录与查询
偏好和工作流匹配
动作结果验证
```

### 云端或强模型处理（可选增强）

```text
复杂任务拆解（全双工模型能力不足时）
长上下文推理
写作和总结
跨网站任务规划
生成新工作流模板
```

分工原则：

```text
全双工模型本地运行，处理绝大多数交互和决策
云端强模型作为后备，处理全双工模型力不能及的复杂任务
安全控制词始终本地规则引擎抢占，不经过任何模型
```

## 三者互相打断的机制设计（ARCH.INTR）

```text
用户打断全双工模型：
  用户说"停" → 本地规则引擎抢占 → 全双工模型立刻停止
  
全双工模型请求用户确认：
  模型遇到高风险操作 → 发送 CONFIRM_NEEDED → 耳机播报确认请求

记录引擎打断全双工模型：
  记录引擎发现任务冲突/异常 → 发送 ALERT → 全双工模型中断当前任务
  记录引擎发现历史上下文关联 → 发送 CONTEXT → 全双工模型获得额外信息
```

优先级：

```text
用户打断（规则引擎抢占） > 记录引擎安全告警 > 全双工模型确认请求 > 正常执行流
```

## 一句话总结

**AI 光标 v2 的架构核心是"全双工模型做大脑 + 双执行器做手脚 + 记录引擎做笔记本"：全双工模型一体化完成听、说、理解和决策；系统执行器操作桌面应用，内置浏览器执行器通过 CDP/DOM 操作多个 BrowserView 实现真多光标并行；记录引擎把一切沉淀为可回溯的 Session Graph。原子动作协议（pointer/keyboard/clipboard/window/overlay/webview）支持三种可见性模式，复杂操作通过原子动作自由组合。**
