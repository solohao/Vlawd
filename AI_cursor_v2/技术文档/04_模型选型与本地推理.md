## 核心变化

v1 采用 L0-L4 五层推理分层：

```text
L0：规则与状态机
L1：本地小模型（短命令分类）
L2：本地 VLM / 小 LLM（屏幕理解）
L3：云端强模型（复杂规划）
L4：工作流（成功路径复用）
```

v2 简化为**两个主要角色 + 一个固定安全层**：

```text
全双工执行大脑 = 交互 + 理解 + 决策 + 语音生成 + ActionProposal
记录笔记本 = Session chunk + 任务面板 + Memory/Workflow 沉淀
固定安全层 = 本地抢占控制词 + 安全策略
```

原来 L0-L4 的功能被重新分配：

| V1 层级 | V1 功能 | V2 归属 |
|---------|---------|---------|
| L0 | 停/暂停/继续 | 本地规则引擎（保留） |
| L1 | 短命令分类 | 全双工模型（内部处理） |
| L2 | 屏幕理解 | 全双工模型 + Observation Layer |
| L3 | 复杂规划 | 全双工模型（大部分）+ 可选云端后备 |
| L4 | 工作流复用 | 记录引擎 + Personalization Layer |

## 全双工模型选型（MODEL.DUPLEX）

### 选型标准

```text
1. 端到端延迟 < 200ms（轮替 + 打断）
2. 中文能力优先（中国市场）
3. 开源可本地部署
4. 7B-9B 参数量（消费级 GPU 可运行）
5. 原生全双工（非 Pipeline 拼接）
```

### 候选模型对比

| 模型 | 团队 | 参数量 | 定位 | 中文 | 全双工方式 | AI Cursor V2 角色 |
|------|------|--------|------|------|-----------|------------------|
| BayLing-Duplex | 中山大学 | 9B | 中文实时监督首选 | 强 | 原生/接近原生 full-duplex | 执行大脑主候选 |
| PersonaPlex | NVIDIA | 7B | 英文低延迟验证 | 待验证 | 原生 full-duplex | 执行大脑强备选 |
| Moshi | Kyutai | 7B | 研究/英文低延迟实验 | 弱 | 原生 full-duplex | 执行大脑实验项 |
| Qwen 2.5 Omni | 阿里 | 3B/7B | 中文多模态与屏幕理解 | 强 | 准全双工/多模态 | 复杂理解或记录增强备选 |
| GLM-4-Voice | 智谱 | 9B | 中文语音组件/基座参考 | 强 | 端到端语音对话，非主推 native full-duplex | tokenizer/decoder/生态参考 |

### 推荐策略

```text
中文实时监督首选：BayLing-Duplex
理由：
  - 最贴近“执行大脑”角色：听、说、理解、决策、动作意图由同一全双工模型承担
  - 中文潜力最好，且可复用 GLM-4-Voice tokenizer/decoder 生态
  - 需要重点实测工程成熟度、显存占用和结构化 action 输出

英文低延迟强备选：PersonaPlex 7B
理由：
  - NVIDIA 开源 full-duplex speech-to-speech 模型，低延迟和工程工具链更强
  - 适合 FullDuplexBench、英文 demo、barge-in/overlap 验证
  - 需接受 NVIDIA Open Model License，并实测中文能力

实验项：Moshi / Qwen2.5-Omni / GLM-4-Voice
理由：
  - Moshi：经典 full-duplex 研究框架，但中文弱
  - Qwen2.5-Omni：适合复杂多模态理解，不作为主执行大脑
  - GLM-4-Voice：不作为 native full-duplex 主候选，只作为中文语音组件/基座参考
```

### FullDuplexBench 评测

建议使用 FullDuplexBench（v3.0）对候选模型进行标准化评测：

```text
评测维度：
1. Smooth Turn-Taking（平滑轮替成功率）
2. User Interruption（用户打断成功率 + 延迟）
3. Pause Handling（暂停处理成功率）
4. Backchanneling（附和反馈适当率）
5. Correction Handling（纠正理解率）
6. Entity Tracking（实体追踪准确率）
7. Multi-Turn Instruction Following（多轮指令遵循率）
8. Safety（安全性）
```

## 双角色模型配置入口（MODEL.CONFIG）

模型选择不拆成语音、视觉、Planner、TTS 多个普通用户槽位，而是映射到桌面端两个角色：

```text
执行大脑：full-duplex 模型，负责实时对话、理解、决策和动作指令。
记录笔记本：记录引擎，负责 Session chunk、任务面板、Memory/Workflow。
安全抢占：本地规则引擎，固定开启，不参与模型选择。
```

默认配置：

| Preset | 执行大脑 | 记录笔记本 | 安全抢占 |
|--------|----------|------------|----------|
| 中文实时监督 | BayLing-Duplex | 标准记录（规则 + JSONL/SQLite） | 本地规则，锁定 |
| 英文低延迟验证 | PersonaPlex | 本地轻量摘要 | 本地规则，锁定 |
| 开发/低配测试 | Mock/Stub | 标准记录 | 本地规则，锁定 |

桌面端详细界面与数据结构见 [技术文档/10_双角色模型配置与桌面端设计](./10_双角色模型配置与桌面端设计.md)。

## 规则引擎设计（MODEL.RULE）

规则引擎处理必须即时响应的安全控制，不经过全双工模型：

### 抢占控制词

```text
停 → 暂停执行器
暂停 → 暂停当前任务
继续 → 恢复执行
取消 → 取消当前任务
不要 → 阻止下一步执行
我来 → 释放光标租约，用户接管
退回 → 回退上一步
```

### 实现方式

```text
输入：音频流（并行送入规则引擎和全双工模型）
处理：关键词匹配 + 有限状态机
延迟：< 50ms
部署：纯 CPU，无 GPU 依赖
```

### 规则引擎与全双工模型的协调

```text
场景 1：用户说"停"
  规则引擎：< 50ms 内暂停执行器
  全双工模型：收到中断通知，停止输出

场景 2：用户说"帮我搜天气"
  规则引擎：不匹配安全词，放行
  全双工模型：正常处理

场景 3：用户说"停，不对，改成后天"
  规则引擎：< 50ms 暂停
  全双工模型：处理后续纠正指令
```

## 记录引擎的模型需求（MODEL.RECORD）

记录引擎的任务不需要"聪明"的大模型：

| 任务 | 实现方式 | 资源 |
|------|---------|------|
| 格式化 chunk | 规则引擎（模板） | CPU |
| 维护 Session Graph | 状态机 | CPU |
| 简单查询（"昨天搜过什么"） | 关键词 + 时间过滤 | CPU |
| 复杂查询（"和上次类似的任务"） | 轻量模型 | 2-4GB GPU |
| 长期记忆筛选 | 轻量模型 | 2-4GB GPU |
| 任务面板摘要 | 轻量模型 | 2-4GB GPU |

推荐轻量模型：

```text
Qwen2.5-3B / Phi-3-mini / Gemma-2B
部署：CPU 或共享 GPU
量化：INT4（降低显存占用）
```

## 可选云端后备（MODEL.CLOUD）

全双工模型覆盖大部分交互和决策需求，但以下场景可能需要云端强模型：

```text
超长文档理解（超出全双工模型 context window）
复杂跨网站任务规划
高质量写作和总结
全双工模型不确定的困难决策
```

云端调用策略：

```text
全双工模型自评信心不足 → 触发云端请求
本地先响应（"我想一下"） → 云端后台思考 → 结果返回后继续
用户感知到的是"员工在工作"，不是"系统卡住了"
```

云端模型候选：

```text
GPT-4o / Claude 3.5 / DeepSeek-V2 / Qwen-Max
调用频率：预计 < 10% 的任务需要
```

## 硬件配置建议（MODEL.HW）

### 基础配置（8GB GPU）

```text
全双工模型：7B INT4 量化（~5GB 显存）
记录引擎：纯 CPU（规则引擎为主）
适合：轻度使用，简单任务
```

### 推荐配置（16GB GPU）

```text
全双工模型：9B INT8 量化（~10GB 显存）
记录引擎：3B 轻量模型（~3GB 显存）
适合：日常使用，中等复杂任务
```

### 高性能配置（24GB GPU）

```text
全双工模型：9B FP16（~18GB 显存）
记录引擎：共享 GPU
云端后备：极少使用
适合：高频使用，复杂任务
```

## 与 V1 分层的兼容

V2 不是否定 V1 的分层思想，而是把分层内化到全双工模型中：

```text
V1 的 L0（规则） → V2 的规则引擎（保留独立）
V1 的 L1（短命令） → V2 全双工模型内部处理
V1 的 L2（屏幕理解） → V2 全双工模型 + Observation Layer
V1 的 L3（复杂推理） → V2 全双工模型 + 可选云端
V1 的 L4（工作流） → V2 记录引擎 + Personalization Layer
```

分层的核心原则不变：

```text
高频、低风险、实时要求 → 本地处理
低频、高价值、高推理需求 → 可选云端
成功路径 → 沉淀为工作流
```

---

## 一句话总结

**V2 把 V1 的 L0-L4 五层推理简化为“全双工执行大脑 + 记录笔记本 + 固定安全抢占 + 可选云端后备”。中文实时监督优先验证 BayLing-Duplex，英文低延迟验证优先 PersonaPlex；GLM-4-Voice 降级为中文语音组件/基座参考。安全控制词由规则引擎 < 50ms 抢占处理；记录引擎用规则系统、3B 轻量模型或云端增强即可。**
