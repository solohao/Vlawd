## 核心变化

v1 采用 L0-L4 五层推理分层：

```text
L0：规则与状态机
L1：本地小模型（短命令分类）
L2：本地 VLM / 小 LLM（屏幕理解）
L3：云端强模型（复杂规划）
L4：工作流（成功路径复用）
```

v2 简化为**两个主要角色 + 一个固定安全层**：

```text
全双工执行大脑 = 交互 + 理解 + 决策 + 语音生成 + ActionProposal
记录笔记本 = Session chunk + 任务面板 + Memory/Workflow 沉淀
固定安全层 = 本地抢占控制词 + 安全策略
```

原来 L0-L4 的功能被重新分配：

| V1 层级 | V1 功能 | V2 归属 |
|---------|---------|---------|
| L0 | 停/暂停/继续 | 本地规则引擎（保留） |
| L1 | 短命令分类 | 全双工模型（内部处理） |
| L2 | 屏幕理解 | 全双工模型 + Observation Layer |
| L3 | 复杂规划 | 全双工模型（大部分）+ 可选云端后备 |
| L4 | 工作流复用 | 记录引擎 + Personalization Layer |

## 全双工模型选型（MODEL.DUPLEX）

### 选型标准

```text
1. 端到端延迟 < 200ms（轮替 + 打断）
2. 中文能力优先（中国市场）
3. 开源可本地部署
4. 7B-9B 参数量（消费级 GPU 可运行）
5. 原生全双工（非 Pipeline 拼接）
```

### 候选模型对比

| 模型 | 团队 | 参数量 | 定位 | 中文 | 全双工方式 | AI Cursor V2 角色 |
|------|------|--------|------|------|-----------|------------------|
| BayLing-Duplex | 中山大学 | 9B | 中文实时监督首选 | 强 | 原生/接近原生 full-duplex | 执行大脑主候选 |
| PersonaPlex | NVIDIA | 7B | 英文低延迟验证 | 待验证 | 原生 full-duplex | 执行大脑强备选 |
| Moshi | Kyutai | 7B | 研究/英文低延迟实验 | 弱 | 原生 full-duplex | 执行大脑实验项 |
| Qwen 2.5 Omni | 阿里 | 3B/7B | 中文多模态与屏幕理解 | 强 | 准全双工/多模态 | 复杂理解或记录增强备选 |
| GLM-4-Voice | 智谱 | 9B | 中文语音组件/基座参考 | 强 | 端到端语音对话，非主推 native full-duplex | tokenizer/decoder/生态参考 |

### 推荐策略

```text
中文实时监督首选：BayLing-Duplex
理由：
  - 最贴近“执行大脑”角色：听、说、理解、决策、动作意图由同一全双工模型承担
  - 中文潜力最好，且可复用 GLM-4-Voice tokenizer/decoder 生态
  - 需要重点实测工程成熟度、显存占用和结构化 action 输出

英文低延迟强备选：PersonaPlex 7B
理由：
  - NVIDIA 开源 full-duplex speech-to-speech 模型，低延迟和工程工具链更强
  - 适合 FullDuplexBench、英文 demo、barge-in/overlap 验证
  - 需接受 NVIDIA Open Model License，并实测中文能力

实验项：Moshi / Qwen2.5-Omni / GLM-4-Voice
理由：
  - Moshi：经典 full-duplex 研究框架，但中文弱
  - Qwen2.5-Omni：适合复杂多模态理解，不作为主执行大脑
  - GLM-4-Voice：不作为 native full-duplex 主候选，只作为中文语音组件/基座参考
```

### FullDuplexBench 评测

建议使用 FullDuplexBench（v3.0）对候选模型进行标准化评测：

```text
评测维度：
1. Smooth Turn-Taking（平滑轮替成功率）
2. User Interruption（用户打断成功率 + 延迟）
3. Pause Handling（暂停处理成功率）
4. Backchanneling（附和反馈适当率）
5. Correction Handling（纠正理解率）
6. Entity Tracking（实体追踪准确率）
7. Multi-Turn Instruction Following（多轮指令遵循率）
8. Safety（安全性）
```

## 双角色模型配置入口（MODEL.CONFIG）

模型选择不拆成语音、视觉、Planner、TTS 多个普通用户槽位，而是映射到桌面端两个角色：

```text
执行大脑：full-duplex 模型，负责实时对话、理解、决策和动作指令。
记录笔记本：记录引擎，负责 Session chunk、任务面板、Memory/Workflow。
安全抢占：本地规则引擎，固定开启，不参与模型选择。
```

默认配置：

| Preset | 执行大脑 | 记录笔记本 | 安全抢占 |
|--------|----------|------------|----------|
| 中文实时监督 | BayLing-Duplex | 标准记录（规则 + JSONL/SQLite） | 本地规则，锁定 |
| 英文低延迟验证 | PersonaPlex | 本地轻量摘要 | 本地规则，锁定 |
| 8GB 低显存 v0 | Qwen2.5-Omni 3B（INT4） | 确定性记录（规则 + JSONL），摘要模型后置 | 本地规则，锁定 |
| 开发/低配测试 | Mock/Stub | 标准记录 | 本地规则，锁定 |

### 8GB 显存下的双角色资源分配

在 8GB 显存（如 RTX 4060 Laptop）上，两个角色不能都是常驻 GPU 的大模型。两角色的
资源/延迟画像不同，应区别对待：

```text
执行大脑 = 硬实时（200ms 内响应语音/打断/出动作） → GPU 常驻、独占算力
记录引擎 = 软实时/异步（动作间隙整理上下文、写 Session） → 不与大脑抢 GPU

8GB 分配原则：把 GPU 几乎全留给实时执行大脑，记录引擎走 CPU / 轻量 / 按需加载
```

桌面端详细界面与数据结构见 [技术文档/10_双角色模型配置与桌面端设计](./10_双角色模型配置与桌面端设计.md)。

## 规则引擎设计（MODEL.RULE）

规则引擎处理必须即时响应的安全控制，不经过全双工模型：

### 抢占控制词

```text
停 → 暂停执行器
暂停 → 暂停当前任务
继续 → 恢复执行
取消 → 取消当前任务
不要 → 阻止下一步执行
我来 → 释放光标租约，用户接管
退回 → 回退上一步
```

### 实现方式

```text
输入：音频流（并行送入规则引擎和全双工模型）
处理：关键词匹配 + 有限状态机
延迟：< 50ms
部署：纯 CPU，无 GPU 依赖
```

### 规则引擎与全双工模型的协调

```text
场景 1：用户说"停"
  规则引擎：< 50ms 内暂停执行器
  全双工模型：收到中断通知，停止输出

场景 2：用户说"帮我搜天气"
  规则引擎：不匹配安全词，放行
  全双工模型：正常处理

场景 3：用户说"停，不对，改成后天"
  规则引擎：< 50ms 暂停
  全双工模型：处理后续纠正指令
```

## 记录引擎的模型需求（MODEL.RECORD）

> 记录引擎不是配角：它是让"监督 < 自己做"这道经济账由负转正的关键杠杆——上下文
> 积累越好，用户监督开销越低。但这不意味着它要是个常驻 GPU 的大模型。

记录引擎的任务不需要"聪明"的大模型，也不应与实时执行大脑争 GPU：

| 任务 | 实现方式 | 资源 |
|------|---------|------|
| 格式化 chunk | 规则引擎（模板） | CPU |
| 维护 Session Graph | 状态机 | CPU |
| 简单查询（"昨天搜过什么"） | 关键词 + 时间过滤 | CPU |
| 复杂查询（"和上次类似的任务"） | 轻量模型 | 2-4GB GPU |
| 长期记忆筛选 | 轻量模型 | 2-4GB GPU |
| 任务面板摘要 | 轻量模型 | 2-4GB GPU |

推荐轻量模型：

```text
Qwen2.5-3B / Phi-3-mini / Gemma-2B
部署：CPU 优先（不抢实时大脑的 GPU）；高配机可共享 GPU
量化：INT4（降低显存占用）
调用时机：软实时/异步（动作间隙或任务结束后整理），不进实时语音闭环热路径
```

8GB 机器上的 v0 建议：记录引擎先用**纯确定性实现**（规则模板 + JSONL/SQLite，
工程骨架中已有 `JsonlSessionStorage`），语义摘要/检索的小模型留接口后置，
避免 v0 阶段与执行大脑争抢本就紧张的 8GB 显存。

## 可选云端后备（MODEL.CLOUD）

全双工模型覆盖大部分交互和决策需求，但以下场景可能需要云端强模型：

```text
超长文档理解（超出全双工模型 context window）
复杂跨网站任务规划
高质量写作和总结
全双工模型不确定的困难决策
```

云端调用策略：

```text
全双工模型自评信心不足 → 触发云端请求
本地先响应（"我想一下"） → 云端后台思考 → 结果返回后继续
用户感知到的是"员工在工作"，不是"系统卡住了"
```

云端模型候选：

```text
GPT-4o / Claude 3.5 / DeepSeek-V2 / Qwen-Max
调用频率：预计 < 10% 的任务需要
```

## 硬件配置建议（MODEL.HW）

> 重要现实：8GB 显存（如 RTX 4060 Laptop）是普通用户的真实中位配置，却**跑不动 9B
> 原装全双工**。分层 + 小模型 + 量化不是可选项，是主路径。显存账：9B FP16≈18GB、
> INT8≈9GB、INT4≈5-6GB。

### 入门配置（8GB GPU / 笔记本 4060 等）

```text
执行大脑：3B 类小模型 INT4（~2-3GB，如 Qwen2.5-Omni 3B）作为 v0 首选
          或 9B 全双工 INT4（~5-6GB）勉强加载，需实测延迟/质量
记录引擎：纯 CPU（确定性记录为主），不与大脑争 GPU
注意：笔记本 GPU 带宽低、功耗墙 + 与系统显示共享显存，实时 200ms 全双工存疑
适合：v0 闭环验证、轻度使用；旗舰 9B 原装全双工不在此档
```

### 推荐配置（16GB GPU）

```text
全双工模型：9B INT8 量化（~9-10GB 显存）
记录引擎：3B 轻量模型（~3GB 显存，按需/异步）
适合：日常使用，中等复杂任务
```

### 高性能配置（24GB GPU / 桌面 4090 等）

```text
全双工模型：9B FP16（~18GB 显存）或 INT8 留更多余量
记录引擎：共享 GPU
云端后备：极少使用
适合：高频使用，复杂任务，隐私+护城河理想形态
```

### 低配 / 办公本（<8GB / 核显）

```text
本地只跑安全抢占 + 轻量记录（护城河层永远本地、永远在线）
复杂推理/执行大脑 → 走云端后备（此时云端从"体验升级"变为低配刚需后备）
```

## 环境自检与配置推荐（MODEL.PROBE）

用户装上 App 后往往不知道自己能跑什么模型、该下原装还是量化版——让用户自己
选 = 勝退。MODEL.PROBE 把这道坎抹平：自动探测硬件 → 推荐配置预设，是降低进入
门槛、扩大可触达市场的关键一环。

### 探测项

```text
GPU 型号 + 显存（最关键）、系统内存、CPU、OS、磁盘可用空间（模型权重几个 GB）
```

### 推荐逻辑（映射到分层策略与双角色资源分配）

```text
显存 ≥16GB  → 原装全双处理大模型（隐私+护城河理想形态）
显存 8-12GB → 量化版小模型 + 部分能力降级（记录引擎转 CPU）
显存 <8GB / 核显 → 仅本地抢占+轻量记录，复杂推理走云端后备
```

### 工程要点与坎

```text
1. NVIDIA 优先：用 nvidia-smi 查型号/显存最准；AMD/Intel 核显/Apple Silicon 各不相同，
   初版只精确支持 NVIDIA，其余给保守降级建议
2. Windows 上 Win32_VideoController.AdapterRAM 为 32 位字段，>4GB 会读错，
   须用 nvidia-smi 或 DXGI 才准
3. 量化能加载 ≠ 质量不掉：推荐时须标注"降级版效果会下降"，不过度承诺
4. 探测+推荐是默认值，必须允许用户手动覆盖
```

### 位置

建议作为"模型中心"里的"环境自检 → 推荐预设"一步，放在模型下载之前，与
MODEL.CONFIG 预设选择衔接。不阻塞主闭环开发，可并行实现。

## 与 V1 分层的兼容

V2 不是否定 V1 的分层思想，而是把分层内化到全双工模型中：

```text
V1 的 L0（规则） → V2 的规则引擎（保留独立）
V1 的 L1（短命令） → V2 全双工模型内部处理
V1 的 L2（屏幕理解） → V2 全双工模型 + Observation Layer
V1 的 L3（复杂推理） → V2 全双工模型 + 可选云端
V1 的 L4（工作流） → V2 记录引擎 + Personalization Layer
```

分层的核心原则不变：

```text
高频、低风险、实时要求 → 本地处理
低频、高价值、高推理需求 → 可选云端
成功路径 → 沉淀为工作流
```

## 一句话总结

**V2 把 V1 的 L0-L4 五层推理简化为“全双工执行大脑 + 记录笔记本 + 固定安全抢占 + 可选云端后备”。中文实时监督优先验证 BayLing-Duplex，英文低延迟验证优先 PersonaPlex；GLM-4-Voice 降级为中文语音组件/基座参考。安全控制词由规则引擎 < 50ms 抢占处理；记录引擎用规则系统、3B 轻量模型或云端增强即可。现实约束：8GB 显存是用户中位配置却跑不动 9B 原装全双工，v0 以 Qwen2.5-Omni 3B 等小模型先行；双角色在 8GB 上采取“实时 GPU 执行大脑 + 轻量异步记录引擎”分配；新增 MODEL.PROBE 环境自检→推荐配置。**
