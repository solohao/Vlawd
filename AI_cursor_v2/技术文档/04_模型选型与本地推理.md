## 核心变化

v1 采用 L0-L4 五层推理分层：

```text
L0：规则与状态机
L1：本地小模型（短命令分类）
L2：本地 VLM / 小 LLM（屏幕理解）
L3：云端强模型（复杂规划）
L4：工作流（成功路径复用）
```

v2 简化为**两个主要组件**：

```text
全双工模型 = 大脑（交互 + 理解 + 决策 + 语音生成）
规则引擎 = 安全护栏（抢占控制词 + 安全策略）
```

原来 L0-L4 的功能被重新分配：

| V1 层级 | V1 功能 | V2 归属 |
|---------|---------|---------|
| L0 | 停/暂停/继续 | 本地规则引擎（保留） |
| L1 | 短命令分类 | 全双工模型（内部处理） |
| L2 | 屏幕理解 | 全双工模型 + Observation Layer |
| L3 | 复杂规划 | 全双工模型（大部分）+ 可选云端后备 |
| L4 | 工作流复用 | 记录引擎 + Personalization Layer |

## 全双工模型选型（MODEL.DUPLEX）

### 选型标准

```text
1. 端到端延迟 < 200ms（轮替 + 打断）
2. 中文能力优先（中国市场）
3. 开源可本地部署
4. 7B-9B 参数量（消费级 GPU 可运行）
5. 原生全双工（非 Pipeline 拼接）
```

### 候选模型对比

| 模型 | 团队 | 参数量 | 轮替延迟 | 打断延迟 | GPT-4o 评分 | 中文 | 全双工方式 |
|------|------|--------|---------|---------|------------|------|-----------|
| PersonaPlex | NVIDIA | 7B | 170ms | 240ms | 4.29 | 待验证 | 原生 |
| GLM-4-Voice | 智谱 | 9B | ~300ms | ~400ms | — | 原生 | 原生 |
| BayLing-Duplex | 中山大学 | 9B | ~300ms | ~400ms | — | 原生 | GLM 微调 |
| Moshi | Kyutai | 7B | ~200ms | ~300ms | 0.77 | 弱 | 原生 |
| Qwen 2.5 Omni | 阿里 | 7B | — | — | 4.59 | 原生 | 准全双工 |

### 推荐策略

```text
中文市场首选：GLM-4-Voice 9B
理由：
  - 原生中文训练，中文理解准确
  - 9B 参数，16GB 显存可运行
  - 智谱开源，社区活跃
  - BayLing-Duplex 是基于 GLM-4-Voice 的全双工微调版，可直接使用

延迟优先备选：PersonaPlex 7B
理由：
  - 延迟最低（170ms 轮替，240ms 打断）
  - NVIDIA 出品，部署工具链成熟
  - 7B 参数，消费级 GPU 友好
  - 需验证中文能力

英文/国际市场：Moshi 7B
理由：
  - Apache 2.0 协议
  - 社区成熟
  - 延迟表现优秀
```

### FullDuplexBench 评测

建议使用 FullDuplexBench（v3.0）对候选模型进行标准化评测：

```text
评测维度：
1. Smooth Turn-Taking（平滑轮替成功率）
2. User Interruption（用户打断成功率 + 延迟）
3. Pause Handling（暂停处理成功率）

5. Correction Handling（纠正理解率）
6. Entity Tracking（实体追踪准确率）
7. Multi-Turn Instruction Following（多轮指令遵循率）
8. Safety（安全性）
```

## 规则引擎设计（MODEL.RULE）

规则引擎处理必须即时响应的安全控制，不经过全双工模型：

### 抢占控制词

```text
停 → 暂停执行器
暂停 → 暂停当前任务
继续 → 恢复执行
取消 → 取消当前任务
不要 → 阻止下一步执行
我来 → 释放光标租约，用户接管
退回 → 回退上一步
```

### 实现方式

```text
输入：音频流（并行送入规则引擎和全双工模型）
处理：关键词匹配 + 有限状态机
延迟：< 50ms
部署：纯 CPU，无 GPU 依赖
```

### 规则引擎与全双工模型的协调

```text
场景 1：用户说"停"
  规则引擎：< 50ms 内暂停执行器
  全双工模型：收到中断通知，停止输出

场景 2：用户说"帮我搜天气"
  规则引擎：不匹配安全词，放行
  全双工模型：正常处理

场景 3：用户说"停，不对，改成后天"
  规则引擎：< 50ms 暂停
  全双工模型：处理后续纠正指令
```

## 记录引擎的模型需求（MODEL.RECORD）

记录引擎的任务不需要"聪明"的大模型：

| 任务 | 实现方式 | 资源 |
|------|---------|------|
| 格式化 chunk | 规则引擎（模板） | CPU |
| 维护 Session Graph | 状态机 | CPU |
| 简单查询（"昨天搜过什么"） | 关键词 + 时间过滤 | CPU |
| 复杂查询（"和上次类似的任务"） | 轻量模型 | 2-4GB GPU |
| 长期记忆筛选 | 轻量模型 | 2-4GB GPU |
| 任务面板摘要 | 轻量模型 | 2-4GB GPU |

推荐轻量模型：

```text
Qwen2.5-3B / Phi-3-mini / Gemma-2B
部署：CPU 或共享 GPU
量化：INT4（降低显存占用）
```

## 可选云端后备（MODEL.CLOUD）

全双工模型覆盖大部分交互和决策需求，但以下场景可能需要云端强模型：

```text
超长文档理解（超出全双工模型 context window）
复杂跨网站任务规划
高质量写作和总结
全双工模型不确定的困难决策
```

云端调用策略：

```text
全双工模型自评信心不足 → 触发云端请求
本地先响应（"我想一下"） → 云端后台思考 → 结果返回后继续
用户感知到的是"员工在工作"，不是"系统卡住了"
```

云端模型候选：

```text
GPT-4o / Claude 3.5 / DeepSeek-V2 / Qwen-Max
调用频率：预计 < 10% 的任务需要
```

## 硬件配置建议（MODEL.HW）

### 基础配置（8GB GPU）

```text
全双工模型：7B INT4 量化（~5GB 显存）
记录引擎：纯 CPU（规则引擎为主）
适合：轻度使用，简单任务
```

### 推荐配置（16GB GPU）

```text
全双工模型：9B INT8 量化（~10GB 显存）
记录引擎：3B 轻量模型（~3GB 显存）
适合：日常使用，中等复杂任务
```

### 高性能配置（24GB GPU）

```text
全双工模型：9B FP16（~18GB 显存）
记录引擎：共享 GPU
云端后备：极少使用
适合：高频使用，复杂任务
```

## 与 V1 分层的兼容

V2 不是否定 V1 的分层思想，而是把分层内化到全双工模型中：

```text
V1 的 L0（规则） → V2 的规则引擎（保留独立）
V1 的 L1（短命令） → V2 全双工模型内部处理
V1 的 L2（屏幕理解） → V2 全双工模型 + Observation Layer
V1 的 L3（复杂推理） → V2 全双工模型 + 可选云端
V1 的 L4（工作流） → V2 记录引擎 + Personalization Layer
```

分层的核心原则不变：

```text
高频、低风险、实时要求 → 本地处理
低频、高价值、高推理需求 → 可选云端
成功路径 → 沉淀为工作流
```

---

## 一句话总结

**V2 把 V1 的 L0-L4 五层推理简化为"全双工模型（大脑）+ 规则引擎（安全护栏）+ 可选云端后备"。中文市场推荐 GLM-4-Voice 9B 作为全双工模型，本地 16GB 显存即可运行；安全控制词由规则引擎 < 50ms 抢占处理；记录引擎用 3B 轻量模型或规则引擎即可。云端强模型从"每步控制器"降级为"偶尔咨询的顾问"。**
