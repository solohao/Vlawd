## 0. 本文档的定位与使用方式

本文档回答一个问题：**为了实现 Vlawd 规划的 Cycle 1（真实全双工入口）及后续 Cycle，我们可以借鉴哪些开源仓库、各自怎么集成、具体该怎么做。**

三条硬边界（与 `02_MVP开发规划与阶段路线` 一致，任何借鉴都不得违反）：

1. **借鉴不是复刻**：只吸收架构思路、接口设计与关键实现手法，不整包搬运、不改变 Vlawd 自身产品定位与安全模型。
2. **按 Cycle 分层**：Cycle 1 只做语音对话，不启用任何真实执行器；Computer Use / 编程智能体属于 Cycle 2 / Phase 2，本文档中相关内容**当前只作架构思路存档，不在 Cycle 1 落地**。
3. **本地优先、许可干净**：优先可离线、TypeScript 同栈、商用许可清晰的方案；引入任何依赖前逐个核对 License。

术语对齐：
- **Conversation Brain**：负责听说对话的“脑”（Cycle 1 的固定 Provider）。
- **Execution Brain**：负责动作规划的“脑”（Cycle 2+）。
- **端上 / 本地**：模型/推理运行在用户机器，不依赖云。

## 1. Vlawd 当前代码事实（集成的起点）

在借鉴之前先明确“现在有什么、缺什么”，避免重复造轮子或误接。

| 位置 | 现状 | 类型 |
|------|------|------|
| `packages/main/src/runtime/duplex-runtime.ts` | 单链路运行时已存在：输入 → 本地抢占检测 → Provider 流式生成 → 可取消输出 → 状态事件投影 → 最小延迟记录 | 真实骨架（可用） |
| `packages/main/src/model/pipeline-duplex-provider.ts` | 已能对接 **OpenAI 兼容端点**做流式文本，取消经 `AbortSignal` 透传；端点不可用回退离线 Echo | 真实骨架（可用） |
| `packages/main/src/model/provider-registry.ts` | `createProvider` 支持 `mock` / `pipeline` / stub；`createLlmAdapter` 走 OpenAI 兼容 | 真实骨架 |
| `packages/main/src/audio/conversation-endpoint.ts` | 只做**设备选路**（蓝牙 Hands-Free 优先、电脑麦克风 fallback） | 骨架 |
| `packages/main/src/audio/mock-audio-session.ts` | 纯 Mock：吐 listening/speaking/stopped 事件，**无真实音频** | Mock |
| `packages/main/src/safety/preemption.ts` | 本地关键词抢占检测（停/暂停/取消） | 真实骨架 |
| Overlay（renderer） | 已订阅 Runtime 事件、Rive 式精灵已接入 | 真实骨架 |

**结论：Cycle 1 真正缺的是“音频前后端”这一段**：

```text
真实麦克风采集 → VAD/说话结束检测 → ASR(STT) → 文本进 duplex-runtime
duplex-runtime 的 LLM 流式文本 → TTS → 播报
用户开口(VAD 命中) → barge-in：flush 当前 TTS 播放 + AbortSignal 取消生成
```

其中“脑”（LLM 流式文本）与“取消”链路**已经具备**；要补的是 **STT、TTS、VAD、barge-in 的音频闭环**，且按文档设计这些放在 **renderer 采集/播报**，主进程只编排。

## 3. Cycle 1 集成方案：真实全双工语音闭环

目标（对齐 `02` §5.1）：用户用真实麦克风与一个固定 Provider 对话，可在 AI 说话中自然插话改约束，AI 停止原回答并按新约束继续。

### 3.1 目标数据链

```text
[renderer] 麦克风(getUserMedia)
   → VAD(ricky0123/vad, Silero)  ── 说话开始 → 触发 barge-in
   → ASR(whisper-web / BrowserAI, 端上离线)  ── 说话结束 → 出文本
   → IPC: conversationUtterance(text)
[main] duplex-runtime
   → 本地抢占检测(safety/preemption) 先行
   → pipeline-duplex-provider.generate() 流式文本(可 AbortSignal 取消)
   → conversation:event 投影(state/speech 增量)
[renderer] 收到 speech 增量
   → TTS(BrowserAI Kokoro / supertonic / SummerTTS) 边收边播
   → 收到用户 VAD 命中(barge-in) → flush TTS 播放队列 + 调 conversationBargeIn()
```

### 3.2 与 Build Slices 的对应（`02` §5.4）

| Build Slice | 现状 | 借哪个 / 怎么做 |
|-------------|------|-----------------|
| C1-1 Runtime 单链路 | 已具备 | 保持 `duplex-runtime` 为唯一编排；音频事件统一经它投影 |
| C1-2 真实音频与 Provider | **缺音频** | renderer 加麦克风采集(Alice 样板) + ASR(whisper-web/BrowserAI) + TTS(BrowserAI/supertonic)；Provider 用 pipeline 接固定本地 LLM（或先用 LocalAI 一个端点兜底） |
| C1-3 可取消输出 | 取消链路已具备 | VAD 命中即 flush renderer TTS 播放队列 + 主进程 `AbortSignal`；参考 RealtimeVoiceChat/pipecat 的打断时序 |
| C1-4 本地抢占 | 已具备 | 保持“停/暂停”不经模型；关键词抢占已在 `safety/preemption` |
| C1-5 Runtime 事件投影 | 已具备 | Overlay 继续订阅真实事件（已完成） |
| C1-6 最小记录与指标 | 部分 | 记录 barge-in 延迟(<200ms)、约束变更、恢复；沿用 `DuplexLatencySample` |

### 3.3 具体集成做法

**A. 麦克风 + VAD（renderer）**
- 参考 **pmbstyle/Alice** 的语音输入组件与 **ricky0123/vad** 的用法：`@ricky0123/vad-web` 加载 Silero，`onSpeechStart`→标记用户开口（触发 barge-in）、`onSpeechEnd`→拿到本段音频交给 ASR。
- VAD 命中即刻通过 IPC 通知主进程（不要等 ASR 出文本再打断，那样超 200ms）。

**B. ASR / STT（renderer，离线优先）**
- 首选 **BrowserAI** 或 **xenova/whisper-web**（transformers.js）在 renderer 内 WebGPU 跑 Whisper，完全离线。
- 中文场景重点验证 whisper 中文小模型的准确率与延迟；不达标再考虑 sherpa-onnx（Node 绑定）或 LocalAI 的 whisper 端点。

**C. Conversation Brain（main，已具备）**
- 复用 `PipelineDuplexModelProvider` + OpenAI 兼容端点。固定 Provider 建议：本地 Qwen2.5（llama.cpp/LM Studio/Ollama）或 **LocalAI**。
- 取消：已有 `AbortSignal`，barge-in 时主进程 abort 当前 generate。

**D. TTS（renderer）**
- 首选 **BrowserAI(Kokoro)** 或 **supertonic**（端上、低延迟、可流式边收边播）。
- 中文兜底：**SummerTTS**（离线无依赖）或 Qwen3-TTS（服务化）。
- 关键：TTS 必须**分句流式播报**且**可被 flush**，否则打断会有尾音——参考 RealtimeTTS 的分句与队列 flush。

**E. Barge-in 时序（借 pipecat / RealtimeVoiceChat）**
```text
VAD.onSpeechStart
  → renderer: 立即停止 TTS 播放 + 清空播放队列
  → IPC → main: conversationBargeIn() → abort 当前 generate
  → 目标：从检测到停止 < 200ms
```

### 3.4 集成顺序建议（先跑通再优化）

1. 先用 **LocalAI 或本地 Qwen**把“文本进→流式文本出”在真实端点上跑通（脱离 Echo 回退）。
2. renderer 接 VAD + Whisper，把真实语音转文本喂进去。
3. renderer 接 TTS 把流式文本播出来。
4. 接 barge-in：VAD 命中 flush TTS + abort。
5. 记录延迟指标，连续成功三次，录 Cycle 1 视频。

> 技术选择原则（`02` §12.2）：Cycle 1 先用最可靠的固定 Provider，不为未来能力提前改造 Runtime；只有实测证明 JS 路径延迟不达标，才把音频关键路径迁到 Rust native。

## 4. 后续 Cycle 的思路借鉴（存档，不在 Cycle 1 落地）

> 以下内容仅指导 `executors/`、`agent-loop/` 的**接口设计方向**，Cycle 1 视频通过前不动手集成。

### 4.1 命令行式编程智能体的可借思路

从 Pi / pi-computer-use / aider / plandex 一类 CLI 编程智能体提炼，值得吸收的五点：

1. **少量可组合的原子工具**：读文件/跑命令/搜索/浏览器/桌面操作等收敛为统一工具集，而不是每个功能一套专用 UI。Vlawd 后续所有能力都应工具化。
2. **ReAct 循环**：观察 → 计划 → 执行一个工具 → 再观察 → 迭代到完成。对应现有 `agent-loop/` + `executors/` + `action-proposal-parser`。
3. **结构化工具协议 + 严格解析校验**：对应 `02` §6.5“Provider 原始输出必须解析和运行时校验、拒绝未知动作”。
4. **先观察后动作 + ref 绑定**：pi-computer-use 的“先 observe 窗口再操作”对应 `observation_state_id` / epoch / 目标资源绑定，防止拿旧引用误操作。
5. **风险闸门 + 可逆 + 审计**：高风险动作需确认、可 undo、留证据——映射安全引擎 `blocked > confirmation_required > safe` 与 Session 证据链。

### 4.2 映射到 Cycle / 功能编号

| 目标 | 借鉴对象 | 对应编号 |
|------|----------|----------|
| Cycle 2 只读研究：BrowserView navigate/read/search/scroll/extract 工具集 | ReAct + 工具化 + 先观察后动作 | `MVP.BROWSER / OBS.* / ACTION.WV / ACTION.PROPOSAL` |
| 共享浏览器内每个任务独立 Space + 用户可接管 | **ego lite Space + 所有权/handoff 模型** | `LEASE.* / WORK.* / VIS.MODE`（target_view 隔离、任务控制状态机） |
| 强页面 Snapshot + `@N` ref 生命周期（快照重建/失效重照） | **ego-browser observe + ref-map** | `OBS.STATE / OBS.LABEL / OBS.QUERY`（state_id、状态绑定 ref、resource epoch） |
| 能力工具化（“Code base 而非 CLI base”，一次组合多步） | **ego-browser JS 工具面 + help() 契约** | `TOOL.* / ACTION.PROPOSAL / SEMANTIC.LAYER` |
| 结果验证：postcondition 已满足视为完成、失败分 transient/permanent | **ego-browser SKILL 纪律 + ElementResolutionError** | `ACTION.VERIFY / SAFE.LEVEL` |
| 站点经验沉淀包（learnings） | **ego-browser learnings/<site>** | `TOOL.SKILL 结晶化 / Capsule（Agentic_Web L3）` |

| 安全引擎强化：风险模式库 + 挑战式确认 | **shellfirm** | `SAFE.LEVEL / SAFE.MULTI` |
| computer-use 动作规划/基准 | trycua/cua、Agent-S、fara | Phase 2 证据驱动扩展 |

### 4.3 纪律边界（重申）

- Computer Use / 编程智能体属于 Cycle 2 / Phase 2，**不得倒置为 Cycle 1 前置**（`02` §6.5、§12.3）。
- 外部 Computer Use 框架只作 backend 研究与基准，Vlawd 自身走 native Environment Interaction Adapter（`09` v1.2 边界）。
- 引入任何执行能力前，对应风险门槛（只读白名单、目标隔离、最高风险聚合、postcondition 校验）必须先成立。

### 4.4 citrolabs/ego-lite —— 共享浏览器 + Spaces + 强 Snapshot（Cycle 2+ 关键参考，源码级精读）

> 结论先行：ego lite 基本是 Vlawd 战略文档里“模态无关护城河（agent 工程 + 工作流记忆 + 可靠可见执行）**去掉语音**”的成品，且已出货、有 benchmark、约 1.2k stars。它验证了 Vlawd 大量非语音赌注，也是 Cycle 2 浏览器执行层最直接的工程参照。仓库：`https://github.com/citrolabs/ego-lite`（官网 lite.ego.app，MIT）。

#### 4.4.1 它是什么

一个“人和 AI agent 并行工作的浏览器”。每个 agent 在自己完全隔离的 **Space** 里跑任务，用户自己的标签页不受影响，且有一个**面板展示所有运行中的 Space/任务**，可随时接管/停止。任何 agent CLI（Claude Code / Codex / Cursor）都能通过 `ego-browser` 驱动它。号称复杂浏览器任务比 Vercel agent-browser 快约 2.5×、token 更少。

#### 4.4.2 实现分层（读过源码后的事实）

```text
开源 ego-browser harness（package/ego-browser，纯 Node.js + CDP，跨平台）
  └─ 通过 globalThis.ego 注入的运行时调用浏览器能力：
     ego.sendCDPMessage / ego.snapshot / ego.createTaskSpace / ego.useTaskSpace
     / ego.claimTaskSpace / ego.createTab / ego.getBrowserVersion ...
闭源 ego lite.app（定制 Chromium 浏览器）
  └─ 注入上面的 ego 运行时；实现 Space 隔离上下文 + 任务面板 UI；
     提供“kernel-level”强 Snapshot（改过 Chromium 源码，稳定吃深层嵌套 iframe）；
     把 ego-browser 命令注册到 PATH。
```

- `ego-browser` 本体只是 **CDP 客户端 + task-space 封装**（`browser-runtime.ts` 走 `ego.sendCDPMessage`，2s session TTL、自动重连、事件缓冲上限 1w；`ref-map.ts`/`ref-state.ts` 用数值 `backendNodeId` 作 `@N` ref，每次快照重建、ref 失效自动重照）。
- “最强 Snapshot”在代码里是 `browserEgo().snapshot(...)`，即由**闭源 App 原生提供**，不是开源 harness 用普通 CDP 算出来的。

#### 4.4.3 平台结论（回答“是否依赖 macOS / 是否普通 Chrome 即可”）

- **不依赖 macOS**：只有安装脚本是 macOS-only（下载 DMG），README/roadmap 明确 Windows/Linux 在计划中；核心机制（Space=Chromium 隔离上下文、Snapshot=Chromium 源码定制、CDP=跨平台协议）**没有一处依赖 macOS 特性**（区别于当年 Codex 多光标用 Mac 私有 API）。
- **不是“普通 Chrome + 扩展”**：它需要一个**定制 Chromium 构建**（ego lite.app），因为 `ego.snapshot`、Space 隔离、任务面板、tab 级控制都在浏览器内核/App 层注入。它只是**继承** Chrome 的数据（登录/cookie/扩展），运行的是自己那套改过的 Chromium。
- **对 Vlawd 无平台障碍**：Vlawd 走 **Electron（=Chromium）+ 内置 BrowserView 多光标**，与 ego lite 的 Spaces **同基座、同思路**（都在 Chromium 内部开隔离上下文，而非驱动外部浏览器）。ego lite 证明“共享浏览器 + 每 agent 独立 Space + 用户接管 + 强 snapshot”可在 Chromium 基座上跨平台实现。

#### 4.4.4 逐条借鉴点（对照 Vlawd 编号）

1. **Space 隔离 + 所有权/接管 API**（`useOrCreate` / `claim` / `switch` / `newTaskSpace` / `complete(keep)` / `handOff` / `takeOver` / `waitForAgentControl`；所有权 `agent`/`user`，不自动抢用户 Space）→ 直接映射 Vlawd 的 `LEASE.*`（光标/资源 Lane 调度）、`WORK.*`（工作区协同）、`VIS.MODE` 与任务控制状态机（queued/active/waiting_confirmation/paused/…）。**Cycle 2 只读研究任务可先只用“agent 独占一个 Space + 用户可接管”这一最小子集。**
2. **强 Snapshot + `@N` ref 生命周期**：快照即 agent “看”页面的文本视图；ref 是短生命周期 `backendNodeId`，导航/DOM 变化后重照，用旧 ref 时若 map 为空自动重照 → 正是 `OBS.STATE`（不可变 state_id、状态绑定 ref、resource epoch）、`OBS.LABEL`（屏幕标号）、`OBS.QUERY`（渐进式观察）的一份现成实现。
3. **“Code base 而非 CLI base”执行模型**：浏览器能力包装成页内 JS 函数（snapshot/fill/click/wait/navigate/capture），agent 写一段 JS 把“观察→动作→等待→提取→验证→兜底”**组合成一次执行**，而非“调两条看结果再调两条”的循环。这与 `TOOL.*`（AI 面工具契约）、`ACTION.PROPOSAL`（sequence/compound 提案）、`SEMANTIC.LAYER`（语义级编排）方向一致，且是省 token/提速的关键手法。
4. **结果验证纪律**：SKILL 明确“**已满足的 postcondition 视为已完成工作**，不重放交互”“区分 worked/didnt/unknown”，`ElementResolutionError` 诚实标 `transient`（可重试）/`permanent` → 直接对应 `ACTION.VERIFY`（worked/didnt/unknown、postcondition verified/preexisting/failed）与安全分级 `SAFE.LEVEL`。
5. **结构化接口优先、无像素坐标**：site skill 要求“稳定 URL/持久选择器、no pixel coordinates、no secrets” → 对应 Vlawd 能力阶梯与 `Agentic_Web/共享架构与协议/04_可验证环境交互与ComputerUse` 的“结构化接口优先于屏幕坐标”。
6. **site learnings（经验沉淀包）**：`skills/ego-browser/learnings/<site>/`（manifest + notes + tools + browser-tools），把成功动作蒸馏成可复用工具，宣称相似任务后续快约 5× → 对应 Vlawd 的 `TOOL.SKILL/结晶化` 护城河与 `Agentic_Web` 的 Capsule（L3，可移植、可验证、可授权）。
7. **时间窗冻结**：SKILL 要求“current/latest”只从任务环境或已验证页面状态确定一次，滚动/虚拟列表/缓存暴露的旧记录不得静默改写锚点 → 可纳入观察/证据纪律，防“任务被最新内容日期悄悄改写”。

#### 4.4.5 取舍与边界（重申纪律）

- **强 snapshot 的代价**：ego lite 的稳定强 snapshot 靠 **fork Chromium 源码**换来。Vlawd 若走标准 Electron BrowserView（不 fork），观察层只能靠 CDP + DOM + AX，处理超深嵌套 iframe 时会略弱。**这是一个“接受较弱 / 未来 fork Chromium”的取舍点，留待 Cycle 2 之后按真实证据评估，不在 Cycle 1/2 前置。**
- ego lite 只做**浏览器**；Vlawd 是**整个桌面**（系统执行器 + 浏览器 + 窗口/应用），二者作用域不同。
- ego lite 靠**文字 CLI**驱动、偏“后台并行 + 事后接管”；Vlawd 的差异化仍是**全双工语音实时监督 + 临场口头纠正 + Session Graph fork/merge + 中文市场**。
- 遵守 §0/§4.3：外部 Computer Use/浏览器执行框架只作 backend 研究与基准，Vlawd 自身走 native Environment Interaction Adapter（`09` v1.2 边界）；相关内容属 Cycle 2/Phase 2，不得倒置为 Cycle 1 前置。
