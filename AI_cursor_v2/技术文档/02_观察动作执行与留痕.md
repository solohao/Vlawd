## 核心判断

AI 光标要能替用户操作电脑，必须形成稳定闭环：

```text
看见屏幕
→ 生成候选目标
→ 输出结构化动作
→ 安全评估
→ 可见执行
→ 观察结果
→ 写入 Session chunk（v2 更新）
→ 继续 / 询问 / 恢复 / 完成
```

这个闭环比"让模型看截图点坐标"更重要。

## 多源屏幕感知

屏幕感知不能只依赖截图，也不能只依赖 DOM。更稳妥的方式是多源融合：

```text
Screenshot
+ OCR
+ DOM
+ Accessibility Tree
+ Window State
+ Cursor State
+ Clipboard
```

### Screenshot

适合：

- 跨软件统一观察
- 视觉布局判断
- 鼠标目标高亮
- 用户监督和回放

缺点：

- 结构化程度低
- 成本高
- 需要视觉模型或 OCR

### OCR

适合：

- 识别按钮文字
- 识别网页标题
- 识别表单标签
- 支持屏幕标号

### DOM

适合网页：

- 获取链接
- 获取按钮
- 获取输入框
- 获取表单结构
- 获取页面文本

缺点：

- 只适用于网页
- 有些网页结构混乱
- Shadow DOM / Canvas / 跨域 iframe 可能有问题

### Accessibility Tree

适合桌面和浏览器可访问性元素：

- 按钮
- 输入框
- 菜单
- 标签页
- 窗口控件

它比 OCR 更结构化，比 DOM 更跨应用。

### Window State

需要知道：

- 当前前台窗口
- 所有打开窗口
- 窗口位置和尺寸
- 浏览器标签页
- 分屏布局

## 目标候选生成

Observation 模块不应只把原始截图丢给模型，而应生成目标候选：

```json
[
  {
    "id": "element_1",
    "type": "input",
    "label": "搜索",
    "bounds": [120, 80, 620, 120],
    "source": "accessibility",
    "confidence": 0.92
  },
  {
    "id": "element_2",
    "type": "button",
    "label": "筛选",
    "bounds": [760, 210, 830, 248],
    "source": "ocr",
    "confidence": 0.81
  }
]
```

全双工模型（v2）或规则/本地模型再根据用户指令选择目标。

## 屏幕标号

屏幕标号是 MVP 的关键交互。

流程：

```text
观察屏幕
↓
生成可点击 / 可操作元素
↓
给元素编号
↓
在屏幕上 overlay 编号
↓
用户说"点 3"
↓
执行器点击 element_3
```

## 动作协议

AI 输出结构化动作（沿用 v1 设计）：

```json
{
  "action": "mouse.click",
  "target": {
    "type": "screen_element",
    "ref": "element_3",
    "description": "第一个搜索结果"
  },
  "button": "left",
  "safety": "safe",
  "expected_result": "打开搜索结果详情页"
}
```

动作分类、安全等级、执行器职责沿用 v1 设计。

---

## 安全等级

### safe

可以直接执行：滚动、切标签、打开网页、点击普通链接、输入草稿、复制内容。

### confirmation_required

必须确认：提交、发送、删除、覆盖、授权、付款、发布、申请。

### blocked

默认禁止：绕过验证码、自动输入敏感密码、未确认付款、未确认删除大量文件。

---

## 执行器职责

执行器沿用 v1 设计：

1. 目标定位
2. 动作前高亮
3. 执行动画 / 光标移动
4. 执行后观察
5. 判断是否达到预期
6. 失败重试或询问用户
7. 写入 Session chunk（v2 更新）

每个动作完成后返回结果，**同时写入当前 Session**：

```json
{
  "step_id": "step_001",
  "status": "success | failed | uncertain | interrupted",
  "observation": "页面已打开岗位详情",
  "screenshot_ref": "screen_001",
  "session_chunk_id": "chunk_015"
}
```

## 留痕与 Session 的对接（v1.1 变更）

v1 的留痕是独立的事件日志（TaskRun + StepEvent + UserCorrection）。

v2 中，所有留痕统一写入 Session：

```text
V1：事件 → 独立 Trace Store
V2：事件 → Session chunk → Session Graph
```

### 事件到 chunk 的映射

| V1 事件类型 | V2 Session chunk type |
|------------|----------------------|
| TaskRun created | session_started |
| StepEvent (action_executed) | action |
| StepEvent (result) | result |
| UserCorrection | correction + session.fork() |
| Error | error |

### UserCorrection 触发 fork

```text
V1：UserCorrection 记录为一条事件
V2：UserCorrection 触发 Session fork
    旧分支保留完整历史
    新分支继续执行
    两个分支都可回看
```

### 工作流沉淀

```text
V1：多次类似 TaskRun → 生成 WorkflowTemplate
V2：多次类似 Session → 记录引擎检测模式 → 提议 WorkflowTemplate
    Session 提供更丰富的模式识别数据（包括纠正历史、分支信息）
```

## 隐私原则

沿用 v1：

```text
默认本地保存
用户可删除
敏感字段脱敏
不默认上传截图
云端调用前显示将发送的上下文摘要
```

v2 新增：Session 数据同样遵循以上原则。

## 一句话总结

**AI 光标必须把"看见屏幕、协议化动作、可见执行、结果验证、Session 留痕"作为底层能力：看见才能操作，协议才能安全，留痕才能复盘，Session 留痕才能支持 fork/merge 和工作流沉淀。**
