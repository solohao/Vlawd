## 核心判断

AI 光标要能替用户操作电脑，必须形成稳定闭环：

```text
看见屏幕
→ 生成不可变 Observation State 和状态作用域目标
→ 输出结构化原子动作（或动作序列）
→ 安全评估
→ 选择执行路径、交付策略和可见性模式
→ 校验目标状态和资源 epoch
→ 执行动作事务
→ 观察后继状态并验证 postcondition
→ 写入 Session chunk
→ 继续 / 询问 / 恢复 / 完成
```

v2.0 的核心升级：

```text
1. 原子动作协议：模块化、可组合、可扩展的动作体系
2. 三种可见性模式：同一动作可以用不同方式执行
3. 内置 BrowserView 真多光标：多个任务真正并行
4. 文字操作能力：输入、复制粘贴、AI 生成文本、表单填写
5. Overlay 自有 UI：AI 可以在屏幕上创建浮层
```

v2.1 补充底层一致性契约：

```text
1. 不可变 Observation State 和状态作用域 ref
2. API / CDP / Accessibility 优先的能力路由
3. 可见性与 delivery policy 分离
4. resource epoch 和旧状态拒绝
5. successor state、postcondition 和三态结果
```

## 多源屏幕感知（OBS.SCREEN）

沿用 v1 设计，多源融合：

```text
Screenshot + OCR + DOM + Accessibility Tree + Window State + Cursor State + Clipboard
```

v2.0 新增：

```text
BrowserView DOM（内置浏览器的 DOM，通过 CDP 直接获取，比外部截图更准确）
BrowserView State（每个 BrowserView 的 URL、标题、加载状态）
```

内置浏览器场景下，DOM 是主要感知源（快速、结构化、准确），Screenshot 作为辅助和用户监督回放。外部桌面应用优先使用 Accessibility / UI Automation；视觉和 OCR 只在语义树不足时升级使用。

### 执行入口优先级

Vlawd 不把 Computer Use 等同于屏幕坐标操作。执行入口按可靠性、权限和可验证性排序：

```text
应用 API / MCP / SDK
→ CLI / 文件受控接口
→ BrowserView DOM / CDP
→ Accessibility / UI Automation
→ Semantic + Screenshot / OCR
→ 当前图像约束下的原始坐标与全局输入
```

每次降级必须记录原因、所需权限、是否改变前台焦点，以及结果能否独立验证。已有结构化入口时，不得仅因模型更熟悉点击而选择高权限 UI 操作。

## 状态作用域观察（OBS.STATE）

每次实时观察生成不可变状态：

```yaml
ObservationState:
  state_id: "state_001"
  root_ref: "root_app_42"
  resource_ref: "desktop_process:9421"
  resource_epoch: 17
  captured_at: "2026-07-16T00:00:00Z"
  sources: [accessibility, screenshot]
  outline_ref: "outline_001"
  image_ref: "screen_001"
  privacy_classification: "sensitive"
```

目标 ref 只在产生它的 `state_id` 中有效：

```text
observe(root)
→ state_id + element refs

act(state_id, ref)
→ 校验 root、resource 和 epoch
→ 有效才执行
```

以下情况使旧状态失效或要求重新观察：

- 页面导航、窗口切换、弹窗或布局显著变化；
- 焦点、登录、权限或前台应用变化；
- 用户移动鼠标、输入键盘或接管任务；
- 另一个动作修改同一物理资源；
- 状态缓存到期或被回收。

Runtime 不得把旧 ref 静默映射为新界面的相似元素，也不得在目标身份不足时猜测坐标。

## 渐进式观察（OBS.QUERY）

完整 DOM / Accessibility Tree 保存在 Runtime 的短期本地缓存，模型先获得折叠语义 outline：

```text
observe(root)
→ folded outline

search(state_id, query)
→ 搜索当前状态中的局部元素

expand(state_id, ref, depth)
→ 展开指定子树

inspect(state_id, ref)
→ 读取角色、名称、状态、属性、边界和来源
```

缓存 query 不占执行 Lease，但仍受 Session、权限和隐私过滤约束。查询不能把密码、完整通知内容或其他高敏感字段默认暴露给模型。

## 原子动作协议（ACTION.PROTO）

### 设计原则

```text
原则 1：每个原子动作只做一件事（单一职责）
原则 2：原子动作之间可自由组合（组合性）
原则 3：新增原子动作不影响已有动作（开放扩展）
原则 4：复杂操作 = 原子动作序列（编排层）
原则 5：新增命名空间 = 注册执行器 + 注册安全等级（即可扩展）
```

### 命名空间

五个基础命名空间 + 两个扩展命名空间：

---

### `pointer.*` — 指针操作（ACTION.PTR）

```text
pointer.move(target)               — 移动光标到目标位置
pointer.click(target, button)      — 单击（left / right / middle）
pointer.double_click(target)       — 双击
pointer.right_click(target)        — 右键菜单
pointer.press(target)              — 按下不松开（拖拽/框选起点）
pointer.release(target)            — 松开（拖拽/框选终点）
pointer.scroll(direction, amount)  — 滚动（up / down / left / right）
```

组合示例：

```text
拖拽 = pointer.press(A) → pointer.move(B) → pointer.release(B)
框选 = pointer.press(area_start) → pointer.move(area_end) → pointer.release(area_end)
```

不需要单独的 `drag` 原子动作——拖拽是 press + move + release 的编排。未来"框选移动"也是同样的组合，只是 target 不同。

---

### `keyboard.*` — 键盘操作（ACTION.KBD）

```text
keyboard.type(text)            — 输入文本字符串
keyboard.press(key)            — 按下单个键（Enter / Tab / Escape / Delete 等）
keyboard.shortcut(keys)        — 组合键（Ctrl+C / Ctrl+V / Ctrl+Z 等）
keyboard.hold(key)             — 按住不松（配合 pointer 使用，如 Shift+点击多选）
keyboard.release(key)          — 松开按住的键
```

组合示例：

```text
Shift 多选 = keyboard.hold("Shift") → pointer.click(file_1) → pointer.click(file_2) → keyboard.release("Shift")
Ctrl+A 全选 = keyboard.shortcut("Ctrl+A")
```

---

### `clipboard.*` — 剪贴板操作（ACTION.CLIP）

```text
clipboard.copy()               — 复制选中内容（= Ctrl+C）
clipboard.paste()              — 粘贴到当前焦点（= Ctrl+V）
clipboard.cut()                — 剪切选中内容（= Ctrl+X）
clipboard.read()               — 读取剪贴板内容（供模型理解）
clipboard.write(content)       — 写入内容到剪贴板（不通过选中）
```

---

### `window.*` — 窗口/标签操作（ACTION.WIN）

```text
window.switch(window_id)       — 切换到指定窗口
window.minimize(window_id)     — 最小化
window.maximize(window_id)     — 最大化
window.close(window_id)        — 关闭
window.resize(window_id, size) — 调整大小
tab.switch(tab_id)             — 切换浏览器标签
tab.open(url)                  — 打开新标签
tab.close(tab_id)              — 关闭标签
```

---

### `overlay.*` — AI 自有浮层（ACTION.OVL）

AI 光标自己往屏幕上创建的 UI 元素，不是操作已有界面：

```text
overlay.pin(content, position, style)    — 在屏幕上钉一个浮层
overlay.update(overlay_id, content)      — 更新已钉内容
overlay.move(overlay_id, position)       — 移动浮层位置
overlay.dismiss(overlay_id)              — 关闭浮层
overlay.minimize(overlay_id)             — 最小化为图标
overlay.list()                           — 列出所有浮层
```

浮层类型：

```text
note       — 便签（用户口述的备忘）
monitor    — 监控卡片（绑定页面元素，定时刷新）
checklist  — 步骤清单（从 Session 提取）
comparison — 对比卡片（并排展示信息）
snippet    — 文本片段（从页面提取的内容）
timer      — 计时器/提醒
```

实现方式：

```text
AI 光标应用（Electron / Tauri）的透明窗口
和屏幕标号 overlay、状态悬浮窗是同一个渲染通道
渲染层级：桌面 → AI overlay 层 → pin 层 → 状态窗 → 任务面板
```

浮层与 Session：

```text
overlay.pin → Session chunk: {type: "overlay_created", overlay_id, content}
overlay.dismiss → Session chunk: {type: "overlay_dismissed", overlay_id}
临时浮层：跟随 Session，任务结束自动消失
持久浮层：用户说"一直留着" → 存入 Memory，跨 Session 保留
```

---

### `webview.*` — AI 自有浏览器视图操作（内置 BrowserView）（ACTION.WV）

AI 光标内置浏览器引擎，可以创建独立的 BrowserView，通过 CDP/DOM 直接操作：

```text
webview.create(url)                    — 创建新 BrowserView 并加载页面
webview.navigate(view_id, url)         — 导航到新页面
webview.dom_click(view_id, selector)   — 通过 DOM 点击元素
webview.dom_input(view_id, selector, text) — 通过 DOM 输入文本
webview.dom_select(view_id, selector, range) — 通过 DOM 选中文本
webview.dom_read(view_id, selector)    — 读取页面元素内容
webview.extract(view_id, query)        — 提取结构化信息
webview.screenshot(view_id)            — 截图（可展示给用户）
webview.close(view_id)                 — 关闭 BrowserView
webview.layout(arrangement)            — 调整多个 BrowserView 的排列
```

关键特性：

```text
每个 BrowserView 有独立 DOM、独立 CDP 通道
多个 BrowserView 互不干扰，可真正并行操作
不需要系统光标 → 不存在光标抢占问题
可选渲染虚拟光标动画（给用户看）
Cookie/登录状态可共享（同一个 Chromium profile）
```

## 语义级编排层（SEMANTIC.LAYER）

语义级操作不是原子动作，是原子动作的有序组合：

### `text.*` — 文字操作（SEMANTIC.TEXT）

```text
text.input(target, content)
  = pointer.click(target) + keyboard.type(content)

text.select(target, range)
  = pointer.click(range_start) + pointer.press(range_start) + pointer.move(range_end) + pointer.release(range_end)

text.replace(target, old, new)
  = text.select(target, old) + keyboard.type(new)

text.clear(target)
  = pointer.click(target) + keyboard.shortcut("Ctrl+A") + keyboard.press("Delete")

text.generate(target, prompt)
  = 全双工模型生成文本 + text.input(target, generated_text)
```

### `form.*` — 表单操作（SEMANTIC.FORM）

```text
form.fill(form_target, fields)
  = text.input(field_1, value_1) + text.input(field_2, value_2) + ...

form.submit(form_target)
  = pointer.click(submit_button)
```

### 口语 → 书面语转换（SEMANTIC.VOICE2TEXT）

全双工模型在生成 `keyboard.type` / `text.input` 动作时，自动进行口语到书面语的转换：

```text
用户说          → AI 理解为        → keyboard.type 参数
"一三八"         "138"             数字格式化
"今天"           "2026-06-17"      日期（视上下文）
"大概三千块"      "3000"            金额格式化
"帮我写个理由"    [生成段落文本]     文本创作（→ text.generate）
"就写张三吧"      "张三"            人名识别
"at gmail"       "@gmail.com"      邮箱补全
```

转换在全双工模型内部处理，需要上下文感知（当前在填什么表单）。

## 三种可见性模式（VIS.MODE）

同一个动作可以用不同方式执行，由全双工模型根据场景决策：

### `visible_system` — 系统光标可见执行

```text
使用系统光标
光标移动 + 高亮 + 点击动画 → 用户看得到
用于：操作用户桌面上的应用、操作系统级 UI
排他资源：需要 CursorLease
```

### `visible_virtual` — 虚拟光标可见执行

```text
在内置 BrowserView 中操作
底层通过 CDP/DOM 执行（快速稳定）
表面渲染虚拟光标动画（给用户看，保持可见执行一致性）
用于：AI 自有窗口中的操作，用户想看过程
不占系统光标 → 可与其他任务并行
```

### `silent` — 静默执行

```text
在内置 BrowserView 中操作
纯 CDP/DOM 操作，无光标动画
用于：后台任务，用户不关心过程只要结果
最高效
```

用户可通过语音切换：

```text
"让我看看你在干什么" → visible_virtual
"后台处理就行" → silent
"两边都给我看着" → visible_system + visible_virtual
```

可见性只描述用户如何看见过程，不等于安全能力。执行器另行声明交付策略（ACTION.DELIVERY）：

```text
semantic_only：
  只允许 DOM / CDP / Accessibility / UIA 语义动作

foreground_allowed：
  可以激活经过验证的窗口并使用受控前台输入

raw_input_allowed：
  可以使用当前图像约束下的原始坐标和全局输入
  必须经过权限、风险、状态和焦点校验
```

`silent` 不能自动获得 `raw_input_allowed`；后台任务也不能绕过用户确认或结果验证。

## ActionProposal 协议（ACTION.PROPOSAL）

```json
{
  "proposal_id": "prop_001",
  "type": "atomic | sequence | compound",
  "visibility": "visible_system | visible_virtual | silent",
  "delivery_policy": "semantic_only | foreground_allowed | raw_input_allowed",
  "target_view": "system | browser_view_A | browser_view_B",
  "observation_state_id": "state_001",
  "resource_ref": "browser_page:target_A",
  "resource_epoch": 7,
  "actions": [
    {
      "action": "pointer.click",
      "target": {"ref": "element_5", "description": "姓名输入框"},
      "params": {"button": "left"}
    },
    {
      "action": "keyboard.type",
      "params": {"text": "张三", "source": "voice_transcription", "original_voice": "就写张三"}
    }
  ],
  "safety": "safe | confirmation_required | blocked",
  "postconditions": [
    {"type": "action_target_value", "action_index": 0, "equals": "张三"}
  ],
  "rollback": [
    {"action": "keyboard.shortcut", "params": {"keys": "Ctrl+Z"}}
  ]
}
```

关键字段：

```text
type：
  atomic    — 单个原子动作
  sequence  — 有序原子动作序列
  compound  — 语义级复合动作（text.input 等，执行时展开为 sequence）

visibility：
  执行时的可见性模式

delivery_policy：
  执行器可以使用的交付上限，与 visibility 分离

target_view：
  system          — 系统桌面（需要系统光标）
  browser_view_X  — 指定的内置 BrowserView（不需要系统光标）

rollback：
  回退动作序列，用户说"退回"时执行

observation_state_id / resource_ref / resource_epoch：
  将动作绑定到产生目标 ref 的状态和物理资源

postconditions：
  执行后如何在 successor state 中验证业务结果，不能把旧 ref 当新状态动作目标，也不能只检查输入事件是否已发送
```

## 安全等级（SAFE.LEVEL）

### safe

可以直接执行：

```text
pointer: 滚动、移动、无外部副作用且目标身份已验证的普通点击
keyboard: 输入草稿、Tab 切换
clipboard: 复制、读取剪贴板
window: 切换标签、打开网页
overlay: 创建/更新/移动浮层
webview: 创建视图、导航、读取内容
text: 在搜索框/草稿区输入
```

### confirmation_required

必须确认：

```text
提交、发送、删除、覆盖、授权、付款、发布、申请
批量表单填写（填完后确认再提交）
AI 生成的文本（text.generate，需确认内容）
跨应用粘贴（可能覆盖目标内容）
替换已有文本
overlay.pin 持久浮层
```

### blocked

默认禁止：

```text
绕过验证码
自动输入敏感密码
未确认付款
未确认删除大量文件
输入密码/敏感凭证
发送包含个人信息的内容（未确认）
```

## 执行器架构（EXEC.ARCH）

### 双执行器设计

```text
系统执行器（System Executor）：
  ・通过 Environment Interaction Adapter 调用 Accessibility / UIA / Visual / Raw Input backend
  ・用于非浏览器的桌面应用
  ・语义观察与语义动作优先
  ・全局光标和键盘需要 CursorLease 排他调度

内置浏览器执行器（BrowserView Executor）：
  ・通过 CDP / DOM API 操作内置 BrowserView
  ・不需要系统光标
  ・多个 BrowserView 可同时操作
  ・每个 BrowserView 有独立执行器实例
```

### 执行器统一接口

```text
observe(root) → ObservationState
query(state_id, query) → CachedObservationResult
execute(proposal) → ActionTransactionResult

ActionTransactionResult = {
  proposal_id: "prop_001",
  outcome: "worked | didnt | unknown",
  verification: "verified | preexisting | failed",
  successor_state_id: "state_002",
  stopped_at: null | 0..N,
  diff_ref: "diff_001",
  evidence_refs: ["evidence_001"],
  failure_type: null | "stale_state | target_missing | delivery_failed | postcondition_failed | interrupted"
}
```

结果语义（ACTION.VERIFY）：

```text
worked：
  有证据证明期望结果由本次动作达成

didnt：
  有证据证明未达成

unknown：
  动作可能已送达，但不能可靠判断业务结果

verified：
  postcondition 在动作后成立，并可归因于当前事务

preexisting：
  postcondition 在动作前已经成立，不能冒充本次成功

failed：
  postcondition 未成立或验证失败
```

### 动作事务

多动作只有在中间不需要重新观察或决策时才允许批量执行：

```text
捕获 pre-transaction baseline
→ 校验 state / resource / epoch / permission / safety
→ 预检所有可预检动作
→ 按顺序执行
→ 失败、用户接管或环境失效时立即停止
→ 返回 stopped_at 和已执行步骤
→ 观察 successor state
→ 计算 diff
→ 验证 postconditions
```

指针动作如果交付结果不明确，不得盲目重试；重复点击可能造成两次提交、两次购买或两次切换。

### 执行流程

```text
1. 接收 ActionProposal
2. 严格验证 target_view 和已注册 Executor，不允许 fallback
3. 校验 observation_state_id、resource_ref 和 resource_epoch
4. 安全评估、权限检查和 delivery_policy 限制
5. 根据 visibility 决定是否渲染光标动画
6. 执行动作事务
7. 捕获后继状态、差异和停止位置
8. 验证 postconditions
9. 写入 Session / Evidence chunk
10. 返回结构化结果
```

## 留痕与 Session 对接（TRACE.SESSION）

所有动作执行都写入 Session：

```text
每个 Observation → Session chunk 或短期引用: {type: "observation", state_id, root_ref, resource_ref, epoch, privacy}
每个 ActionProposal → Session chunk: {type: "action", proposal_id, state_id, resource_ref, actions, postconditions}
每个 ActionResult → Session chunk: {type: "result", outcome, verification, successor_state_id, stopped_at, diff_ref, evidence_refs}
每个 UserCorrection → Session chunk: {type: "correction"} + session.fork()
每个 overlay 操作 → Session chunk: {type: "overlay_created/updated/dismissed"}
每个 BrowserView 操作 → Session chunk: {type: "webview_action", view_id, action}
```

工作流沉淀：

```text
多次类似的 ActionProposal 序列
→ 记录引擎检测模式
→ 提议 WorkflowTemplate
→ 用户确认后绑定为 WorkflowShortcut
```

## 隐私原则

```text
默认本地保存
用户可删除
敏感字段脱敏
不默认上传截图
不默认永久保存完整 Accessibility / UIA Tree
状态缓存按 TTL 回收，高敏感字段先过滤再提供给模型
BrowserView 中的页面数据不出本机
overlay 内容本地存储
```

## 一句话总结

**Vlawd 的环境交互以结构化接口和语义观察优先：目标 ref 绑定不可变状态，动作绑定资源与 epoch，执行事务必须返回后继状态、差异和 postcondition 结果。可见性、安全等级与交付策略彼此独立；事件已发送不等于业务成功，任何外部 Computer Use 实现都只能作为 Adapter backend 或比较基准，不能绕过 Vlawd Runtime、Session 和用户接管。**
