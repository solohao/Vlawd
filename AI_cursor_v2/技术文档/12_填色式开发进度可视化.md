## 这是什么 / 为什么这么做

单人开发容易缺乏即时正反馈、容易遗漏、也容易"这也想做一点那也想做一点"。
本机制把开发进度做成一个**填色游戏**：

- **默认整个界面是"未处理态"——去色成浅灰**（好像一张没上色的线稿）。
- 每当一个功能单元**闭环并验证通过**，就把它恢复成正常主题色（那版克制的
  鼠尾草绿点缀）。
- 越做越多，界面逐步"浮现"出最终配色；全部完成时满屏恢复正常色 = 完工。

价值不只是好玩，而是**逼出纪律**：必须"做完一个 + 验证过"才允许上色，从而
防止半成品自我欺骗、天然形成"做完一个再做下一个"的节奏，并让遗漏和优先级
一眼可见。

> 重要：这是**开发期可视化层**，不是最终产品主题。发布给真实用户的构建应
> 显示"完成态配色"，不能让用户看到半灰界面。

## 颜色语义（规划）

只用"一种彩色 + 灰度深浅"，避免多色打架：

| 状态 | 颜色 | 含义 |
|------|------|------|
| todo（未处理） | 浅灰（低对比、发淡） | 还没做，自然退到背景 |
| wip（开发中） | 深灰 / 灰色描边 | 当前正在做的战场 |
| done（已验证） | 主题绿（克制的鼠尾草绿点缀） | 闭环完成且验证通过 |

- 颜色只表达**进度**，单一含义；**重要性（P0/P1/P2）不画在界面上**，放到进度
  HUD 的清单排序里，避免"深灰"同时代表"重要"和"在做"造成歧义。
- 上色以**功能单元（feature）为单位、全有或全无**：一个 feature 只要还有任意
  必要环节没完成/没验证，整块停在 wip（深灰），不允许上色；全部闭环验证通过
  才整块变绿。

## 单元怎么划分

按**用户可感知的最小可验证闭环**划分，不是单个按钮，也不是整页。

- 界面型功能：锚定到屏幕上某块 UI，直接在界面填色。
- 抽象/跨层闭环（如"本地意图 → 云端 Capsule → 回写本地 Session"）：没有单一
  UI 元素，用**链路图/Pipeline 视图**表达，每个"可独立验证的跳"是一个节点，
  连线显示断点；端到端真跑通一次才整条点亮。这类闭环的"允许上色"判据建议
  绑定到**存在一条真实跑通的 Session 记录（带 Evidence）**，与项目"证据驱动"
  方法论一致。

单元编号复用 `01_功能模块注册表.md` 的现有编号体系。

## 当前实现（v0.1）

只实现了"全局未处理态"（整屏浅灰），用于先验证去色效果：

- `packages/renderer/src/app/paint-mode.ts`：`initPaintMode()` 读取上次选择
  （默认 `on`），在 `<html>` 上设 `data-paint="on|off"`，并暴露开发期切换入口
  `window.__paint`（`on()/off()/toggle()/get()`）。
- `packages/renderer/src/app/index.css`：
  `html[data-paint="on"] #root { filter: grayscale(1) contrast(.62) brightness(1.14); }`
  ——主窗口与透明的运行时悬浮窗统一去色为浅灰。
- `packages/renderer/src/app/main.tsx`：启动时调用 `initPaintMode()`。

### 如何验证 / 切换

- 打开主窗口：整个界面应变成浅灰（未处理态）。
- DevTools 控制台：
  - `__paint.off()` 恢复正常配色（用于对比）
  - `__paint.on()` 回到浅灰未处理态
  - `__paint.toggle()` 切换
  - 选择会记在 localStorage，下次启动保持。
