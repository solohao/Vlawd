## 0. 本文档的定位

`13_开源参考项目与集成路线` 回答“选谁、怎么分层、按 Cycle 怎么落地”。本文档是它的**源码级补充**：把这批仓库真正 clone 下来读过之后，记录**每个仓库对我们目标功能（麦克风 / VAD / ASR / LLM 流式 / TTS 流式 / barge-in 打断 / 状态与历史）具体是怎么实现的**，以及**哪一段可以直接借、借什么、注意什么**。

目标功能对照（Vlawd Cycle 1 需要补齐的链路）：

```text
麦克风采集 → VAD(说话开始/结束) → ASR → 文本进 duplex-runtime
→ LLM 流式文本 → 分句 → TTS 流式 → 播放队列
→ 用户开口(barge-in) → flush 播放 + abort LLM/TTS + 记录“已听到部分”
```

精读的本地副本（只读，不修改）：`/home/ubuntu/repos/` 下 `Alice`、`BrowserAI`、`vad`、`whisper-web`、`RealtimeVoiceChat`、`pipecat`、`mastra`、`supertonic`、`SummerTTS`、`Open-LLM-VTuber(+ -Web-src)`。

> 纪律不变：本文档只做借鉴分析。Cycle 1 只做语音，不落地任何执行器 / Computer Use。

## 1. pmbstyle/Alice —— 最贴合的整体样板（TS/Electron，可直接借代码）

**它是什么**：Electron + Vue + Pinia 的语音优先桌面助手；STT/LLM/TTS 支持云与本地（本地走内置 Go 后端，whisper.cpp + piper）。是与 Vlawd **同栈、同形态**的最佳参照。

**它怎么处理目标功能（关键代码）**：

- **麦克风 + VAD（renderer）** `src/composables/useAudioProcessing.ts`
  - 直接用 `@ricky0123/vad-web` 的 `MicVAD.new({ onSpeechStart, onSpeechEnd })`。
  - **barge-in 就在 `onSpeechStart` 里**：若当前状态是 `SPEAKING`/`WAITING_FOR_RESPONSE`，立即 `eventBus.emit('cancel-llm-stream')` + `stopPlaybackAndClearQueue()`——**先打断，不等 ASR 出文本**。
  - `onSpeechEnd(audio: Float32Array)`：仅当状态为 `LISTENING` 才 `float32ArrayToWav(audio, 16000)` → 转写。
  - Electron 生产环境要点：VAD 的 onnx/wasm 资产路径在 `file://` 下要经 IPC 拿 `rendererDistPath` 拼 `baseAssetPath`，否则加载失败（对我们打包很关键）。
- **状态机**：`IDLE / LISTENING / PROCESSING_AUDIO / WAITING_FOR_RESPONSE / SPEAKING / GENERATING_IMAGE / CONFIG`。
- **LLM 流式 → 分句 → TTS** `src/modules/conversation/streamHandler.ts`
  - 边读 LLM 增量边累加 `currentSentence`；命中 `SENTENCE_END_REGEX = /[.!?]\s*$/` 或换行就 `flushSentence()` → `enqueueSpeech(sentence)`。**句级切分是低延迟 TTS 的关键**。
- **TTS 队列 + flush** `src/composables/useAudioPlayback.ts`
  - `audioQueue` 顺序播放（`HTMLAudioElement` + `URL.createObjectURL(blob)`）；`stopPlaybackAndClearQueue()` = pause + 清空队列 + `revokeObjectURL`。
  - `watch(audioState)`：一旦离开 `SPEAKING` 立即停播——把“打断”和“状态切换”统一。
- **双 AbortController** `src/modules/conversation/turnManager.ts` + `speechQueue.ts`
  - **LLM 与 TTS 各一个 `AbortController`**；`cancel-llm-stream`/`cancel-tts` 事件分别 abort。
  - 打断后**保留 `currentResponseId`** 以维持对话连续性（下一轮接得上）。
- **STT provider 抽象** `backendService.ts`：`openai / groq / google / local` 分支，`local` 走内置 whisper.cpp。

**对 Vlawd 的借鉴点**：
1. **barge-in 触发点放在 VAD `onSpeechStart`**，且“停播放 + abort LLM”并行，不等转写——直接映射我们的 <200ms 目标。
2. **句级 flush → 逐句 TTS 入队**：直接抄到 renderer 的 speech 增量消费端。
3. **LLM/TTS 双 AbortController 分离**：对应我们 `duplex-runtime` 现有单 `AbortController`，应拆成“生成” + “播报”两级，barge-in 时都取消。
4. **VAD 资产路径在 Electron 打包下的 IPC 处理**——提前规避打包坑。
5. `float32ArrayToWav` 16kHz 转换工具可几乎照搬。

**注意**：Alice 本地 STT/TTS 是 **Go sidecar（whisper.cpp/piper）**，不是纯 renderer；若我们要纯端上，STT/TTS 换成 BrowserAI/Supertonic，但**上层编排（VAD/分句/队列/双 abort）与 provider 无关，可直接复用**。

## 2. ricky0123/vad —— renderer 端 VAD（可直接依赖）

**它是什么**：浏览器端 Silero VAD 封装，`@ricky0123/vad-web`（+ `@ricky0123/vad-react`）。

**关键 API / 参数**：`MicVAD.new({ onSpeechStart, onSpeechEnd(audio: Float32Array), onVADMisfire, positiveSpeechThreshold, negativeSpeechThreshold, redemptionFrames, minSpeechFrames, preSpeechPadFrames, baseAssetPath, onnxWASMBasePath })`；实例方法 `start()/pause()`。

**借鉴点**：
- 直接作为 renderer VAD 依赖（Alice 已验证可在 Electron 用）。
- **调参就是延迟与误触发的平衡**：`redemptionFrames`（判定说话结束前的宽限帧）直接影响 endpoint 快慢；`preSpeechPadFrames` 保证句首不被吃掉；`minSpeechFrames` 抑制咳嗽/杂音误触发。这些是 Cycle 1 指标调优旋钮。
- barge-in 只需 `onSpeechStart` 事件本身，不需要等 `onSpeechEnd` 的音频，最快。

## 3. BrowserAI —— 纯 renderer 的 STT+TTS+LLM（端上一体，可直接借代码）

**它是什么**：基于 transformers.js + WebGPU，在浏览器内跑 Whisper(STT)、Kokoro(TTS)、小 LLM，全离线。

**它怎么处理目标功能**：
- **STT**：`loadModel('whisper-tiny-en')` + 内置录音 `startRecording()/stopRecording()` + `transcribeAudio(blob, { language, return_timestamps })`。
- **TTS**：`src/engines/tts-engine.ts`，Kokoro（`StyleTextToSpeech2Model`）WebGPU，`SAMPLE_RATE = 24000`；**`async *generateSpeechStream(text)` 逐块 yield `Float32Array`（流式）**，内部按 `MAX_CHUNK_LENGTH = 250` 切分长文本。

**借鉴点**：
1. 若走**纯端上路线**，BrowserAI 是最短路径：STT+TTS+LLM 一个包搞定，无需 Go/Python sidecar。
2. `generateSpeechStream` 的**流式 Float32 输出**可直接喂 Web Audio 播放，边合成边播，进一步降低首音延迟（比 Alice 的“整段 blob 入队”更细）。
3. 文本切块（250 字符）与 Alice 的句级切分可结合：**先句级、句内再按长度切**。

**注意**：默认 Whisper 权重偏英文（`whisper-tiny-en`）、Kokoro 语音偏英文；**中文需换多语言 whisper 权重并验证 Kokoro/替代 TTS 的中文质量**（见第 8 节）。WebGPU 依赖较新 Electron/Chromium。

## 4. xenova/whisper-web —— renderer ASR 的 worker 化范式

**它是什么**：transformers.js 的 Whisper Web Demo（React）。

**关键实现** `src/worker.js`：在 **Web Worker** 里 `pipeline('automatic-speech-recognition', model)`；支持 `multilingual`、`language`、`quantized`、`subtask('transcribe'|'translate')`；单例工厂避免重复加载；medium 用 `no_attentions` revision 省内存。

**借鉴点**：
- **ASR 放 Web Worker**，不阻塞 renderer UI/动画（我们精灵动画在主渲染线程，尤其要这样做）。
- **多语言 + `language` 参数**证明 transformers.js Whisper 可做中文（BrowserAI 默认英文只是选型问题）。
- 单例 pipeline 工厂 + 量化模型的加载/内存策略可直接借。

## 5. KoljaB/RealtimeVoiceChat —— “可自然打断”的最完整参考（借设计/可作 sidecar）

**它是什么**：Python 客户端-服务器低延迟语音对话（RealtimeSTT + RealtimeTTS + Ollama/OpenAI）。文档与打断实现最完整。

**它怎么处理目标功能（关键代码）**：
- **智能轮次检测** `code/turndetect.py`
  - 不只靠固定静音时长：用 **句子完成度分类模型**（`KoljaB/SentenceFinishedClassification`）预测“这句话说完了没”，再**按完成概率动态插值出应等待的停顿时长**（越像完整句，等待越短→更快接话；越不完整，等待越久→不打断用户思考）。
  - `sentence_end_marks = ['.', '!', '?', '。']`（**含中文句号**），不同标点给不同停顿（问号/感叹/省略号各异）。
- **多级 abort** `code/speech_pipeline_manager.py`
  - `RunningGeneration` 带 `id` 与 `llm_aborted / audio_quick_aborted / audio_final_aborted / abortion_started` 等标志；一组 `threading.Event`（`stop_llm_request/finished`、`stop_tts_quick_*`、`stop_tts_final_*`、`abort_completed`）实现 **LLM / 快速 TTS / 最终 TTS 分级停止**。
  - **`generation_counter` / `is_valid_gen()`**：用递增 id 判定当前生成是否仍有效——**打断后丢弃旧 id 的迟到输出**（防止“已打断还蹦出上一句的音频”）。
  - **“quick answer”**：先合成很短的首句降低首音延迟，再接完整答复。

**对 Vlawd 的借鉴点**：
1. **动态 endpoint（语义级轮次检测）** 是比纯 VAD 静音更自然的做法——可作为 Cycle 1 之后的增强（先用 VAD 静音跑通，再引入完成度判定降低“抢话”）。
2. **generation id / epoch 丢弃迟到输出**：与 Vlawd 文档里的 `epoch` / `observation_state_id` 思想一致，应在 `duplex-runtime` 里给每次生成打 id，barge-in 后旧 id 的 speech/audio 一律丢弃。
3. **多级 abort + finished 事件**：比单 abort 更稳，能确保“真的停干净了”再进入下一轮。
4. **quick-answer-first** 降低感知延迟的技巧可选用。

**注意**：Python + 偏 GPU（Coqui/DeepSpeed），**不适合直接搬进 Electron 主进程**；借设计，或作为独立 sidecar。

## 6. pipecat —— 帧/管线与打断的设计权威（借设计）

**它是什么**：Python 实时语音/多模态管线框架。

**关键设计** `src/pipecat/frames/frames.py`：一切皆 **Frame**，在 processor 管线里流动；`UserStartedSpeakingFrame / UserStoppedSpeakingFrame / BotStartedSpeakingFrame / BotStoppedSpeakingFrame` 等 **SystemFrame** 优先级高，可越过普通队列；打断即一个系统帧向下游传播并 flush TTS。还有 **`UserMuteStartedFrame/StoppedFrame`（mute 策略）**。

**对 Vlawd 的借鉴点**：
1. **typed event/frame 管线 + 系统级“打断帧”**：Vlawd 已有 `DuplexModelEvent`/`DuplexRuntimeEvent`，应显式补 `user_started_speaking`/`user_stopped_speaking`/`bot_started_speaking`/`bot_stopped_speaking` 与一个高优先 `interrupt` 事件，语义与 pipecat 对齐。
2. **mute 策略防自打断**：机器人自己的 TTS 播放期间，要防止外放声音被麦克风 VAD 当成用户说话而误触发 barge-in。做法：播放时对本地 VAD 做 mute/门控，或用回声消除。**这是纯软件全双工必踩的坑，务必在 Cycle 1 处理**（Alice 因常是耳机/半双工影响小，但我们主打耳机全双工要显式处理）。

## 7. Open-LLM-VTuber —— 打断后的“历史真相”处理（借设计，含中文栈）

**它是什么**：Python 后端（ASR/LLM/TTS/VAD 模块化，`asr/ tts/ vad/ agent/`）+ Electron/Web 前端，WebSocket 通信。

**它怎么处理目标功能（关键代码）**：
- **通信协议** `src/open_llm_vtuber/websocket_handler.py`
  - 消息分类：`CONVERSATION=['mic-audio-end','text-input','ai-speak-signal']`、`CONTROL=['interrupt-signal','audio-play-start']`、`DATA=['mic-audio-data']`。
  - 每个客户端一个 `asyncio.Task` 跑对话；**打断 = 对该 Task `task.cancel()`**。
- **打断时保存“已听到部分”** `src/open_llm_vtuber/conversations/conversation_handler.py` → `handle_individual_interrupt(..., heard_response)`
  - `task.cancel()` 后：`agent_engine.handle_interrupt(heard_response)`，并 `store_message(role='ai', content=heard_response)` + `store_message(role='system', content='[Interrupted by user]')`。
  - 即：**写进历史的不是 AI 想说的完整答复，而是用户实际听到的那一截**，并标注被打断。

**对 Vlawd 的借鉴点（这条最独特、最重要）**：
1. **打断后必须把上下文截断到“用户实际听到的部分”**，并标记 `[被用户打断]`。否则下一轮 LLM 会以为自己已完整表达，导致“用户改约束后 AI 答非所问”。这正是 Cycle 1 验收核心“AI 停止原回答并按新约束继续”的正确实现。
2. 前端需**统计实际已播放的音频量**来算 `heard_response`（我们句级队列天然可算：已播完的句子集合）。
3. WebSocket 的 `CONTROL / CONVERSATION / DATA` 消息分层，与我们主/渲染进程 IPC 事件划分可对齐。
4. 前端/精灵交互此前已参考（Pet 全屏透明窗、右键原生菜单、面板与头像解耦），此处不重复。

**注意**：后端 Python，借设计为主；其 ASR/TTS 模块化（sherpa/funasr/faster-whisper、多 TTS）可作为**中文栈选型线索**。

## 8. TTS/STT 引擎候选（组件级，可直接集成）

### 8.1 mastra —— voice 抽象接口（TS，借接口设计）
`packages/_internals/voice/src/voice/voice.ts` 的 `abstract class MastraVoice` 同时覆盖两种模式：
- **回合式**：`speak(text) → NodeJS.ReadableStream`（TTS）、`listen(audioStream) → text`（STT）。
- **实时式**：`connect()` / `send(audioData)` / `answer()` / `on(event)`（事件：`'speaking' | 'writing' | 'error'`）。

**借鉴点**：这正是 Vlawd“**现在用 pipeline(ASR→LLM→TTS)、将来换原生全双工**”所需的**统一接口**——两条路径共用一个 Voice 抽象，上层 Runtime 不必改。建议我们的语音适配器接口照此拆：`speak/listen`（Cycle 1）预留 `connect/send/answer/on`（后续原生全双工）。

### 8.2 Supertonic —— 端上 ONNX TTS（MIT，Node 可用，首选）
- **MIT 许可**；31 语言（含中文，或 `lang="na"` 语言无关）；提供 **Node.js SDK（`onnxruntime-node`）** 和 Browser(WebGPU) SDK，`nodejs/example_onnx.js` 可直接参考；也有 OpenAI 兼容 `POST /v1/audio/speech` 服务端点。
- **借鉴点**：Vlawd 的**首选端上 TTS**。既能在 Electron 内用 `onnxruntime-node` 直接跑（main 进程或 worker），也能起 OpenAI 兼容端点当 sidecar。支持 Voice Builder 自定义音色（品牌化声音）。需实测中文自然度与首包延迟。

### 8.3 SummerTTS —— 纯离线中文 TTS 兜底（MIT，C++）
- **MIT**；纯 C++/Eigen，**无第三方 NN 运行时依赖**，一键编译，离线中英文，CLI `tts_test in.txt model.bin out.wav`。
- **借鉴点**：**完全离线/隐私最强的中文 TTS 兜底**。集成方式：N-API 绑定或作为本地 CLI/进程调用。音质与灵活性不如 Supertonic，作断网兜底。

### 8.4 STT 小结
- 纯端上：transformers.js Whisper（whisper-web 的 worker 范式，多语言权重做中文）或 BrowserAI（WebGPU）。
- sidecar：whisper.cpp（Alice 路线）、sherpa-onnx（Node 绑定，STT+TTS+VAD 一体）、LocalAI（OpenAI 兼容，最快闭环）。

## 9. 合并后的 Cycle 1 借鉴实现清单（谁的哪一段、用在哪）

| Vlawd 环节 | 直接借鉴来源 | 具体做法 | 落点 |
|-----------|-------------|---------|------|
| 麦克风 + VAD | Alice + ricky0123/vad | `MicVAD.new`，Electron `file://` 资产路径经 IPC 处理 | renderer 新增 audio 采集 |
| barge-in 触发 | Alice | 在 `onSpeechStart` 立即“停播放 + abort”，不等转写 | renderer → IPC → main |
| ASR | whisper-web / BrowserAI | transformers.js Whisper 放 **Web Worker**，多语言权重 | renderer worker |
| LLM 流式（已具备） | 现有 `pipeline-duplex-provider` | 保留 `AbortSignal` | main |
| 分句 → TTS | Alice + BrowserAI | 句级 `flushSentence`，句内按长度切；流式 TTS | renderer 消费 speech 增量 |
| TTS 引擎 | Supertonic（首选）/ SummerTTS（兜底）/ BrowserAI(Kokoro) | `onnxruntime-node` 或 WebGPU；中文实测 | renderer 或 main worker |
| 播放队列 + flush | Alice | `audioQueue` + `stopPlaybackAndClearQueue` | renderer |
| 双/多级 abort + 生成 id | Alice + RealtimeVoiceChat | LLM/TTS 分离 abort；生成 `id` 丢弃迟到输出 | main `duplex-runtime` |
| 打断后上下文截断 | Open-LLM-VTuber | 历史只存“已听到部分” + `[被打断]` 标记 | main `duplex-runtime` |
| 事件语义 | pipecat | 补 `user/bot started/stopped speaking` + 高优先 `interrupt` 事件 | shared 事件类型 |
| 防自打断 | pipecat（mute 策略） | 播放期间门控本地 VAD / 回声消除 | renderer |
| 轮次检测增强（Cycle 1 后） | RealtimeVoiceChat `turndetect` | 句子完成度模型动态停顿 | 增强项，非必须 |
| 语音适配器接口 | mastra `MastraVoice` | `speak/listen` 先行、预留 `connect/send/answer/on` | shared 接口 |

**Cycle 1 落地顺序（不变）**：固定 Provider 真实文本闭环 → 接 VAD+Whisper → 接流式 TTS → 接 barge-in（停播 + abort + 生成 id 丢弃 + 上下文截断）→ 处理防自打断 → 记录延迟指标 → 连续成功三次 → 录视频。

## 10. 许可与风险小结

| 项目 | 许可 | 集成风险 |
|------|------|---------|
| ricky0123/vad、Silero | 宽松（MIT 系） | 低；注意 onnx/wasm 资产打包路径 |
| transformers.js / whisper-web | Apache-2.0 | 低；WebGPU/WASM 内存与首次加载耗时 |
| BrowserAI | 需核对（含第三方权重） | WebGPU 依赖新 Chromium；中文权重需替换 |
| Supertonic | **MIT** | 低；需实测中文自然度/延迟；模型资产体积 |
| SummerTTS | **MIT** | 需 N-API/CLI 集成；音质一般 |
| mastra | 需核对（多为 Apache/MIT） | 只借接口设计，风险低 |
| Alice | 需核对 | 借代码/设计；本地栈是 Go sidecar |
| RealtimeVoiceChat / pipecat / Open-LLM-VTuber | 各含第三方引擎许可 | Python，借设计为主；引擎(Coqui/XTTS 等)许可需单独核 |

**统一原则**：任何要打进产品的模型权重与代码，逐个核对 License 与商用条款；端上模型注意权重许可与体积/内存；优先 MIT/Apache。
