## 设计边界（MODEL.CONFIG）

本文只规划桌面端如何选择、下载、绑定模型角色，不实现真实模型接入。

不采用多模型插槽市场，不把语音、视觉、Planner、TTS 拆成普通用户可见的复杂配置。V2 保持两角色架构：

```text
角色 1：执行大脑
  - 全双工模型
  - 负责实时听说、理解用户、决定下一步、输出 ActionProposal

角色 2：记录笔记本
  - 记录引擎
  - 负责监听事件、写 Session chunk、更新任务面板、沉淀 Memory/Workflow

固定安全层：安全抢占
  - 本地规则引擎
  - 负责 停/暂停/取消/退回 等硬中断
  - 不允许关闭，不作为模型选项出售
```

## 角色配置数据结构

代码层只增加配置骨架，不改变 AgentLoop：

```ts
type ModelRole =
  | "duplex_execution_brain"
  | "session_record_engine"
  | "safety_preemption";

interface WorkflowModelBinding {
  workflow_id: string;
  executionBrain: ProviderConfig;
  recordEngine: RecordEngineConfig;
  safetyPreemption: SafetyPreemptionConfig;
}
```

执行规则：

1. `executionBrain` 只能通过 `DuplexModelProvider` 输出语音状态和 `ActionProposal`。
2. `recordEngine` 监听输入、模型输出、动作结果和用户纠正，不直接操作电脑。
3. `safetyPreemption` 永远是 `locked: true` 的本地规则引擎。
4. 工作流保存 `WorkflowModelBinding`，Session chunk 记录当时模型 kind、路径/版本和记录模式。

## 默认候选与下载入口

| 候选 | 默认角色 | 推荐原因 | 下载入口 |
|------|----------|----------|----------|
| BayLing-Duplex | 执行大脑 | 最贴近中文 native full-duplex 主线 | Hugging Face：`BayLing-Models/BayLing-Duplex` |
| NVIDIA PersonaPlex | 执行大脑 | 英文低延迟、工程成熟度较强 | Hugging Face：`nvidia/personaplex-7b-v1`，需接受 NVIDIA 许可证 |
| Moshi | 执行大脑实验项 | 经典开源 full-duplex 对话框架 | Kyutai Moshi collection |
| 标准记录 | 记录笔记本 | 规则 + JSONL/SQLite，MVP 最稳 | 内置，无需下载 |
| 本地轻量记录 | 记录笔记本 | 用 3B/7B 小模型做摘要和 workflow mining | Qwen2.5-3B / Phi-3-mini / Gemma-2B |
| 云端增强记录 | 记录笔记本 | 只在复杂 Session 摘要时调用 | 用户自行配置云端模型 |

GLM-4-Voice 在本文中不作为 native full-duplex 主执行大脑推荐项；它可作为中文语音组件/生态参考，或被 BayLing-Duplex 等方案复用 tokenizer/decoder。

## 工作流绑定体验

每个工作流显示并保存两项配置：

```text
工作流：浏览器求职筛选

执行大脑：BayLing-Duplex
记录笔记本：标准记录 + 关键节点摘要
安全抢占：本地规则，强制开启

日志粒度：完整记录
工作流沉淀：发现重复步骤后询问我
```

用户理解模型配置的方式应是：

```text
谁陪我实时做？
谁把过程记下来？
哪些安全词永远能让它停？
```

## 一句话总结

**桌面端模型配置只暴露两个角色：执行大脑负责实时对话与动作决策，记录笔记本负责 Session 与工作流沉淀；安全抢占固定为本地规则引擎，不进入模型市场。**
