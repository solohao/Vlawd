## 核心决策总览（STACK.OVERVIEW）

```text
桌面框架：    Electron 33+（BrowserView 原生支持 + CDP + 跨平台）
前端 UI：     React 19 + TailwindCSS（快速迭代 + 组件生态）
性能关键层：  Rust → napi-rs 编译为 Node addon（规则引擎、音频处理、屏幕采集）
模型推理：    独立进程（llama.cpp server / vLLM），通过 HTTP/gRPC 与主进程通信
Session 运行时：借鉴 OpenRath Session-first 思想，TypeScript 实现
Agent 循环：  借鉴 Claude Code ReAct 模式，单循环 + 工具分发 + 上下文压缩
数据存储：    SQLite（Session/Memory 持久化）+ Append-only JSONL（操作日志）
构建工具：    electron-forge + electron-builder
包管理：      pnpm workspace（monorepo）
语言比例：    TypeScript ~70% / Rust ~20% / Python ~10%（辅助脚本 + 模型微调）
```

## 为什么是 Electron（STACK.ELECTRON）

### 技术约束 → 框架映射

| 架构需求 | Electron 如何满足 | 替代框架的缺陷 |
|---------|-------------------|---------------|
| 多 BrowserView 实例 + CDP | Electron 原生 API：`new BrowserView()` + `webContents.debugger` | Tauri 无等价物；Qt WebEngine 多实例 CDP 控制弱 |
| Overlay 透明窗口 | `transparent: true` + `alwaysOnTop: true` + 鼠标穿透 | Tauri/Flutter 支持有限 |
| 系统光标控制 | Node native addon（nut.js / robotjs） | Qt 可做但 UI 迭代慢 |
| 实时音频 I/O | Node 调用 PortAudio / 平台音频 API | Web 技术栈统一处理 |
| 跨平台 | Win / Mac / Linux 同一代码库 | 原生开发需要多套代码 |
| 快速 UI 迭代 | Web 技术 + HMR | Qt/Flutter 迭代周期长 |

### Electron 的已知缺点与缓解策略

```text
问题 1：内存占用高（基础 ~150MB）
缓解：模型推理在独立进程，不占 Electron 内存
      BrowserView 共享 Chromium 渲染进程
      Electron 33+ 对 V8 内存有显著优化

问题 2：二进制包体积大（~150MB+）
缓解：electron-builder 增量更新（delta update）
      按平台裁剪不需要的 Chromium 组件

问题 3：原生 API 需要桥接
缓解：napi-rs（Rust）编译为 .node addon，性能无损
      关键路径（规则引擎、音频）全部 Rust 实现

问题 4：安全性（node integration）
缓解：Renderer 开启 contextIsolation + sandbox
      BrowserView 不开启 nodeIntegration
      preload 脚本最小化暴露面
```

### 先例验证

```text
Cursor IDE     — Electron fork（AI + 代码编辑 + 多面板）
Claude Desktop — Electron（AI 本地客户端 + MCP）
Notion Desktop — Electron（复杂 UI + 本地存储）
Figma Desktop  — Electron（多画布渲染 + 插件系统）
```

### 进程职责边界

| 进程 | 语言 | 职责 | 崩溃影响 |
|------|------|------|---------|
| Main | TypeScript | 窗口管理、IPC 路由、Agent Loop、工具分发 | 整个应用退出 |
| Renderer(s) | TypeScript/React | UI 渲染、用户交互 | 单个窗口白屏，可恢复 |
| BrowserView(s) | Web | 网页任务执行（CDP 控制） | 单个标签页失效，可重建 |
| Native Addon | Rust | 规则引擎、音频、截图、系统操控 | 触发 Main 错误处理 |
| AI Inference | C++/Python | 全双工模型 + 可选云端后备 | Agent Loop 进入降级模式 |
| Recording Engine | TypeScript | Session 管理、Memory、Graph | 丢失本轮记录，下轮恢复 |

### 进程间通信协议

```text
Main ↔ Renderer：   Electron IPC (contextBridge + preload)
Main ↔ BrowserView：CDP (Chrome DevTools Protocol)
Main ↔ Native：     napi-rs 同步/异步调用（N-API）
Main ↔ AI：         HTTP（llama.cpp OpenAI-compatible API）或 gRPC
Main ↔ Recording：  MessagePort（结构化消息）或 Unix Socket
```

## Environment Interaction Adapter（STACK.INTERACT）

Main Process 中的 Adapter 统一 Vlawd 契约，平台 backend 只提供能力：

```text
EnvironmentInteractionAdapter
├─ BrowserView CDP backend
├─ macOS Accessibility backend
├─ Windows UI Automation backend
├─ Linux AT-SPI backend（证据成熟后启用）
├─ Visual / OCR fallback
└─ Raw Input backend（排他、受控）
```

稳定接口：

```typescript
interface EnvironmentInteractionAdapter {
  observe(root: RootRef): Promise<ObservationState>;
  query(stateId: string, query: ObservationQuery): Promise<QueryResult>;
  execute(proposal: ActionProposal): Promise<ActionTransactionResult>;
}
```

Main Process 仍负责：

- Safety 与 Permission；
- `target_view`、resource 和 epoch 校验；
- CursorLease / resource lane；
- pause、cancel、takeover；
- Session / Evidence；
- postcondition 编排和结果汇报。

平台 backend 不得拥有 Agent Loop 或自行重试有副作用动作。外部 `pi-computer-use` 等项目只用于架构比较、独立基准和可选 helper 可行性研究；Vlawd 不依赖 Pi 的扩展生命周期、工具命名或配置格式。

本节是目标工程契约，不表示当前代码已实现 Computer Use。现有 `ActionResult`、Executor 路由和 Mock Desktop Runtime 仍需按 MVP 路线逐步迁移；在严格目标拒绝、状态校验和 postcondition 可用前，不开放通用 System Executor。

## Agent 循环设计（STACK.LOOP）

参考 Claude Code 的 queryLoop() 设计：单一代码路径处理所有场景，ReAct 模式（观察→思考→行动→观察），异步生成器实现。

### 核心循环伪代码

```typescript
async function* agentLoop(session: Session): AsyncGenerator<AgentEvent> {
  while (true) {
    // 1. 组装上下文（Session chunks + Memory + 系统提示）
    const context = await buildContext(session);
    
    // 2. 调用全双工模型（流式）
    const response = await callDuplexModel(context);
    
    // 3. 分发工具执行
    if (response.hasActions()) {
      // 安全评估（规则引擎 < 50ms）
      const safetyResult = await safetyEvaluate(response.actions);
      
      if (safetyResult === 'blocked') {
        yield { type: 'blocked', reason: safetyResult.reason };
        continue;
      }
      if (safetyResult === 'confirmation_required') {
        const approved = yield { type: 'confirm', actions: response.actions };
        if (!approved) continue;
      }
      
      // 执行动作（区分并发安全 vs 互斥）
      const results = await executeTools(response.actions, session);
      
      // 写入 Session chunk
      session = session.appendChunks(results.toChunks());
      yield { type: 'action_completed', results };
    }
    
    // 4. 语音输出（并行）
    if (response.hasVoice()) {
      yield { type: 'voice', stream: response.voiceStream };
    }
    
    // 5. 停止条件检查
    if (response.isComplete() || session.status === 'completed') {
      break;
    }
    
    // 6. 用户打断检测（规则引擎）
    if (ruleEngine.hasInterrupt()) {
      session = session.fork(ruleEngine.interruptReason());
      yield { type: 'interrupted', newSession: session };
    }
  }
}
```

### 与 Claude Code 的设计对比与借鉴

| Claude Code 设计 | AI 光标的适配 |
|-----------------|-------------|
| `queryLoop()` 单一入口 | 所有交互（语音/快捷键/任务面板）都进入同一个 agentLoop |
| StreamingToolExecutor（边流式边执行） | 全双工模型的动作指令流边生成边执行 |
| 5 层上下文压缩 | Session chunk 压缩（超长 Session 自动摘要） |
| Stop Hook（独立评估者判断完成） | 语音场景不需要——用户随时可以说"好了"/"停" |
| 工具并发分类（只读并行 / 写入串行） | pointer/keyboard 串行（系统光标互斥），BrowserView 并行 |

### 与 OpenRath 的设计对比与借鉴

| OpenRath 设计 | AI 光标的适配 |
|--------------|-------------|
| `forward(session) -> session`（Agent 是变换层） | agentLoop 每轮也是 `session → session` 变换 |
| Session 是流动的运行时值（类比 Tensor） | Session 是所有状态的唯一载体 |
| Sandbox 可插拔（`session.to("local")`） | 执行环境由 CursorLease 调度（系统/BrowserView） |
| Selector 动态路由 | 全双工模型自行决定下一步（不需要外部路由） |
| Memory 独立 backend（recall/commit） | Memory Store 在每轮开始 recall，任务完成后 commit |
| Session Graph 动态生长（fork/merge） | 用户纠正/分叉时 Session fork，完成时 merge |

## Session 运行时设计（STACK.SESSION）

直接借鉴 OpenRath 的 Session-first 架构，用 TypeScript 实现。

### Session 生命周期

```text
Create → Place → Transform → Branch → Persist → Release

对应 AI 光标场景：
  Create：用户开口/按下快捷键，创建新 Session
  Place：确定执行环境（哪个窗口/BrowserView）
  Transform：Agent Loop 不断变换 Session（追加 chunks）
  Branch：用户纠正→fork，新任务→fork
  Persist：SQLite 持久化（append-only）
  Release：任务完成，释放 CursorLease
```

### 存储方案

```text
实时数据：内存中的 Session 对象（TypeScript）
持久化：SQLite（主存储）
  - sessions 表：session_id, parent_id, status, created_at, ...
  - chunks 表：chunk_id, session_id, type, content(JSON), timestamp
  - lineage 表：fork/merge 关系
  - memory 表：长期记忆条目

操作日志：Append-only JSONL（参考 Claude Code）
  - 每个操作 append 一行，不可修改
  - 用于审计、调试、回放
  - 路径：~/.ai-cursor/sessions/{session_id}.jsonl
```

### 为什么选 SQLite + JSONL 双轨

```text
SQLite：结构化查询（"昨天搜了什么"、"这个任务 fork 了几次"）
JSONL：完整证据链（可回放、可审计、人类可读）

两者不矛盾：
  JSONL 是写入端（append-only，永不丢失）
  SQLite 是查询端（结构化索引，快速检索）
  启动时可从 JSONL 重建 SQLite（灾难恢复）
```

## 工具分发与扩展机制（STACK.TOOL）

参考 Claude Code 的四机制扩展模型，适配到 AI 光标场景：

### 工具执行分层

```text
FlowAction（模型可见层）
  ↓ 全双工模型输出的原子动作协议
  ↓ 如：{ namespace: "pointer", action: "click", params: { x, y } }
  ↓
Safety Policy（安全评估层）
  ↓ safe / confirmation_required / blocked
  ↓
Execution Scheduler（调度层）
  ↓ 根据 target_view 选择执行器
  ↓ 桌面语义资源 → 按 backend 能力串行或分 Lane
  ↓ 系统光标 / 键盘 → 排他调度（CursorLease）
  ↓ BrowserView → 并行分发
  ↓
Backend Executor（执行层）
  ├─ SystemExecutor：Interaction Adapter → AX/UIA/Visual/Raw Input
  ├─ BrowserViewExecutor：CDP → DOM/JS/网络
  └─ OverlayExecutor：IPC → Overlay Renderer
```

| Workflow Template | 技能（Skills） | 用户沉淀的工作流 | 低（描述） |

## 模型通信架构（STACK.MODEL）

### 全双工模型通信

```text
                    ┌───────────────────────────┐
                    │  llama.cpp server 进程     │
                    │  (独占 GPU)                │
                    │                           │
                    │  端口: localhost:8080      │
                    │  协议: OpenAI-compatible   │
                    │  模型: BayLing/PersonaPlex │
                    └─────────────┬─────────────┘
                                  │ HTTP Streaming
                                  │ (Server-Sent Events)
                    ┌─────────────▼─────────────┐
                    │  Main Process              │
                    │  DuplexModelClient         │
                    │                           │
                    │  - 音频流上行（WebSocket） │
                    │  - 动作指令流下行（SSE）    │
                    │  - 语音流下行（SSE）        │
                    └───────────────────────────┘
```

### 通信协议设计

```text
上行（用户 → 模型）：
  协议：WebSocket（实时音频流）
  格式：PCM 16-bit 16kHz（原始音频帧）
  频率：每 20ms 一帧（640 bytes）

下行（模型 → 应用）双路输出：
  路径 1 - 动作指令流：
    协议：HTTP SSE（Server-Sent Events）
    格式：JSON（原子动作协议）
    示例：{"namespace":"pointer","action":"click","params":{"x":500,"y":300}}
    
  路径 2 - 语音流：
    协议：HTTP SSE
    格式：PCM 16-bit 24kHz（合成语音帧）
    延迟目标：< 200ms（首 token）
```

### 模型进程管理

```typescript
interface InferenceProcessConfig {
  // llama.cpp server 配置
  modelPath: string;          // GGUF 模型文件路径
  contextSize: number;        // 上下文长度（默认 8192）

  // 生命周期
  autoStart: boolean;         // 应用启动时自动拉起
  healthCheck: string;        // 健康检查端点 /health
  restartOnCrash: boolean;    // 崩溃自动重启
  maxRestarts: number;        // 最大重启次数
  
  // 降级策略
  fallbackMode: 'pipeline' | 'cloud' | 'disabled';
  cloudEndpoint?: string;     // 云端后备 API
}
```

### 降级策略

```text
正常模式：全双工模型本地运行（延迟 < 200ms）
  ↓ 如果 GPU 不足 / 模型加载失败
降级模式 1：Pipeline 本地运行（VAD + ASR + LLM + TTS，延迟 1-3s）
  ↓ 如果本地资源完全不足
降级模式 2：云端全双工（需要网络，延迟受网络影响）
  ↓ 如果无网络
降级模式 3：仅规则引擎（基本安全控制词可用，无 AI 能力）
```

## 原生模块策略（STACK.NATIVE）

### 为什么用 Rust + napi-rs

```text
规则引擎延迟要求 < 50ms → 不能用 JS
音频实时处理 → 需要直接访问平台 API
屏幕截图 + OCR → 性能敏感
系统光标操控 → 需要 OS API 调用

Rust 的优势：
  - 零成本抽象，性能接近 C/C++
  - 内存安全（无 GC 暂停）
  - napi-rs 提供完善的 Node.js 绑定
  - 跨平台编译（cargo + cross）
  - 生态丰富（winapi / core-foundation / x11）
```

### Native Addon 模块划分

| 模块 | Crate 名 | 功能 | 平台依赖 |
|------|---------|------|---------|
| rule-engine | `ai-cursor-rules` | 安全控制词匹配 + FSM | 无（纯逻辑） |

## 安全架构（STACK.SAFE）

参考 Claude Code 的分层安全设计（拒绝优先 + 纵深防御）：

### 安全层级

```text
Layer 0：规则引擎抢占（< 50ms，硬性安全词拦截）
```

```text

  - nodeIntegration: false
  - contextIsolation: true
  - webSecurity: true

  - 不能调用 Node API
```

## 与现有设计文档的关系（STACK.REF）

本文档是**实施层**——回答"用什么技术做"。其他文档是**设计层**——回答"做什么"和"为什么这样做"。

```text
技文/01_系统总体架构     → 定义了两角色 + 双执行器（本文的进程架构实现它）
技文/02_观察动作执行     → 定义了原子动作协议（本文的工具分发层实现它）
技文/03_全双工交互       → 定义了模型交互方式（本文的模型通信层实现它）
技文/04_模型选型         → 定义了用什么模型（本文的推理进程管理实现它）
技文/05_光标调度         → 定义了 CursorLease（本文的调度模块实现它）
技文/06_Session 工程     → 定义了 Session 数据结构（本文的 Session 运行时实现它）
技文/07_个性化工作流     → 定义了 Personalization（本文的 Memory Store 实现它）
技文/08_工程架构         → 定义了 bounded contexts（本文的模块分包对应它）
```

### 来自 OpenRath 的核心借鉴

```text
1. Session 作为一等公民：所有状态通过 Session 流动，Agent 不持有状态
2. forward(session) → session：每个处理步骤输入输出都是 Session
3. Sandbox 可插拔：执行环境（系统/BrowserView）绑定在 Session 上
4. Memory 独立后端：recall + commit 模式
5. Session Graph 动态生长：fork/merge 由运行时事件驱动
6. Selector 路由思想：全双工模型自行做路由决策（不需要外部编排器）
```

### 来自 Claude Code 的核心借鉴

```text
1. 最小脚手架 + 最大操作平台：AI 决策层极薄，基础设施层厚重
2. 单一循环入口：所有场景走同一个 agentLoop()，不分裂代码路径
3. ReAct 模式：观察→思考→行动→观察，无显式状态图
4. 工具并发分类：只读操作并行，状态修改操作串行
5. Append-only 持久化：JSONL 操作日志，可审计可回放
6. 四层扩展机制：不同上下文成本的扩展方式，面向插件生态
7. 上下文压缩：5 层渐进管道，在有限窗口中保留关键信息
8. 拒绝优先安全：未识别操作默认拒绝，多层安全并行工作
```

**AI 光标的开发技术栈以 Electron 为应用壳（BrowserView + Overlay + 跨平台），Rust native addon 做性能关键层（规则引擎 < 50ms + 音频 + 系统操控），独立进程运行全双工模型推理（llama.cpp + GPU 隔离），TypeScript 实现 Session 运行时和 Agent Loop（借鉴 OpenRath 的 Session-first + Claude Code 的 ReAct 单循环），SQLite + JSONL 双轨持久化（结构化查询 + 完整审计），pnpm monorepo 组织代码，electron-builder 打包分发。**
