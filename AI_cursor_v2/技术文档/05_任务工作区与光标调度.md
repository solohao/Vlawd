## 核心判断

v1 的核心约束是：

> 真实光标永远只有一个，硬件中断级别的排他资源。

v1.2 的新认识：

> **内置浏览器的 BrowserView 打破了这个约束**——每个 BrowserView 有独立 DOM 和 CDP 通道，不需要系统光标，多个 BrowserView 可以真正并行操作。CursorLease 从"排他光标调度器"扩展为"资源分配管理器"。

v1.4 补充：

> **任务目标和物理资源必须同时显式标识。** `target_view` 决定动作属于哪个 Workspace，`resource_ref + resource_epoch` 决定观察是否仍可执行。缓存查询可以并行，实时观察和动作必须按物理资源排序。

## CursorLease 调度器的角色扩展（LEASE.CURSOR）

### v1.0-v1.1：排他光标调度

```text
系统光标只有一个 → 必须排他调度
CursorLease 保证同一时刻只有一个持有者操作光标
```

### v1.2：双模式资源管理

```text
系统光标调度（排他模式）：
  用于桌面原生应用操作（Word、Excel、文件管理器等）
  同一时刻只有一个 Lease 持有者
  用户随时可以抢回（最高优先级）
  和 v1 行为完全一致

BrowserView 资源分配（并行模式）：
  用于内置浏览器操作
  多个 Lease 可以同时存在
  每个 Lease 绑定一个 BrowserView
  通过 CDP/DOM 操作，不需要系统光标
  真正并行，互不干扰
```

### 统一的 Lease 数据结构

```text
CursorLease = {
  holder: "ai_task_001",
  session_id: "session_abc",
  target: "system" | "browser_view_A" | "browser_view_B" | ...,
  resource_ref: "global_pointer" | "desktop_process:9421" | "browser_page:target_A",
  resource_epoch: 17,
  acquired_at: timestamp,
  expires_at: timestamp,
  priority: 1-10,
  context: ExecutionContext,
  visibility: "visible_system" | "visible_virtual" | "silent"
}
```

v1.2 / v1.4 新增字段：

```text
target:     调度目标
            "system" = 系统光标（排他）
            "browser_view_X" = 内置 BrowserView（可并行）

visibility: 可见性模式
            "visible_system" = 系统光标可见执行
            "visible_virtual" = 虚拟光标动画 + CDP/DOM
            "silent" = 静默执行，无光标动画

resource_ref:
            当前 Lease 实际占用的物理资源

resource_epoch:
            资源状态版本；窗口、页面、焦点、用户输入或其他动作改变资源时递增
```

## 调度规则（LEASE.RULES）

### 系统光标调度（排他，沿用 v1）

```text
Rule 1：同一时刻只有一个执行器控制真实光标
Rule 2：用户操作立刻抢占所有 AI Lease
Rule 3：Lease 超时自动释放
Rule 4：高风险任务不能被低优先级任务打断
Rule 5：所有 Lease 操作写入当前 Session
Rule 6：Lease 切换时，记录引擎写入 session.fork() 或 session.pause()
Rule 7：用户抢回 Lease 时，Session 记录 "user_takeover" chunk
Rule 8：用户输入使相关 Observation State 失效，后续动作必须重新观察
Rule 9：全局指针、键盘和剪贴板按风险分别申请排他资源
```

### BrowserView 资源分配（并行，v1.2 新增）

```text
Rule B1：多个 BrowserView Lease 可以同时存在
Rule B2：每个 BrowserView 同一时刻只有一个 Lease 持有者
Rule B3：BrowserView 创建/销毁由调度器统一管理
Rule B4：资源上限：同时活跃的 BrowserView 数量受内存约束
Rule B5：每个 BrowserView Lease 独立绑定 Session
Rule B6：BrowserView 之间可以共享 Cookie/登录状态（同一 Chromium profile）
Rule B7：同一 browser_page 的实时 observe / act 串行，缓存 query 可以并行
```

### 物理资源 Lane（LEASE.RESOURCE）

```text
desktop_process:<pid>
desktop_window:<window_id>
browser_page:<target_id>
global_pointer
global_keyboard
clipboard
```

调度器维护：

```text
resource_ref
resource_epoch
current_holder
pending_operations
last_user_intervention
```

规则：

- `search / expand / inspect` 只读缓存，不申请实时执行 Lane；
- `observe / act / read_live` 进入对应资源 Lane；
- 独立 BrowserView 页面可以并行；
- 同一桌面进程是否可并行必须由平台 backend 显式声明；
- 全局指针和键盘始终排他；
- 原始坐标动作还必须持有目标窗口和全局输入 Lease；
- 平台无法证明资源独立时，选择更保守的串行调度。

### 调度决策流程

```text
全双工模型输出 ActionProposal
→ 判断 target_view：
   system → 根据 backend 申请桌面进程 / 窗口资源；原始输入再申请系统光标 Lease
   browser_view_X → 申请 BrowserView page Lease（独立页面可并行）
   new_browser_view → 创建新 BrowserView + 申请 Lease

示例：
  用户："帮我同时查岗位和改简历"
  → 全双工模型拆分为两个 ActionProposal 序列
  → Proposal A: target=browser_view_A（查岗位）
  → Proposal B: target=browser_view_B（改简历）
  → 调度器分配两个 BrowserView Lease
  → 两个执行器并行工作
  → 两个 Session 独立记录
```

## Task Card（任务卡）（LEASE.TASK）

沿用 v1 设计，v1.2 新增 BrowserView 绑定：

```text
TaskCard = {
  id: "task_001",
  name: "搜索前端岗位",
  status: queued | active | waiting_confirmation | paused | failed | done | cancelled,
  workspace: browser_view_A,       # v1.2：绑定 BrowserView
  session_id: "session_abc",
  visibility: "visible_virtual",    # v1.2：可见性模式
  current_step: "读取第二个来源",
  next_action: "打开详情页",
  pending_confirmation: null,
  recovery_anchor: "chunk_024",
  steps: [...]
}
```

### 任务卡的生命周期

```text
用户说"帮我搜岗位" → 创建 TaskCard + 创建 Session + 分配 BrowserView
TaskCard 排队 → Session 状态: pending
TaskCard 获得 Lease → Session 状态: active
TaskCard 暂停 → Session 状态: paused（BrowserView 可回收或保留）
TaskCard 完成 → Session 状态: completed（BrowserView 释放）
TaskCard 取消 → Session 状态: cancelled（BrowserView 释放）
```

### Task Workspace 状态契约

```text
queued
  → active
  → waiting_confirmation
  → active
  → done

active
  → paused
  → active

active / waiting_confirmation / paused
  → cancelled

active
  → failed
  → paused（保留恢复锚点）
  → active（用户确认恢复）
```

控制语义：

| 控制 | 行为 |
|------|------|
| 暂停 | 停止继续调度新动作，保留 Workspace、Session 和恢复锚点 |
| 取消 | 终止任务并释放 Lease，不删除 Session 历史 |
| 接管 | 立即释放系统光标或冻结 BrowserView，提高相关 resource epoch，使旧状态失效，并记录 `user_takeover` |
| 确认 | 只批准当前精确动作和参数，不授予后续通用权限 |
| 拒绝 | 记录拒绝原因，任务回到 paused 或重新规划 |
| 恢复 | 从最近安全锚点重新观察环境，再生成下一动作 |
| 回滚 | 仅执行 ActionProposal 明确声明且已验证的补偿动作 |

### `target_view` 隔离

每个 ActionProposal 必须显式声明目标：

```text
target_view =
  system
  | browser_view:<workspace_id>
```

调度器在执行前验证：

- 目标存在且属于当前 TaskCard；
- Lease 持有者与 `session_id` 一致；
- `resource_ref` 属于当前目标，`resource_epoch` 与 Observation State 一致；
- proposal 中的元素 ref 属于声明的 `observation_state_id`；
- BrowserView 动作不能回退到系统执行器；
- 系统动作不能借用其他任务的 BrowserView；
- 未声明、过期或跨任务目标直接 blocked；
- 目标状态失效时重新观察，不把旧 ref 映射到相似元素；
- 所有拒绝和重路由写入当前 Session。

这比通过当前前台窗口猜测目标更重要，因为多任务并行时任何隐式 fallback 都可能把正确动作执行到错误页面。

### 多任务卡的并行管理

```text
BrowserView 并行场景：
  TaskCard A (active) → BrowserView A → Session A (active) → 虚拟光标 A
  TaskCard B (active) → BrowserView B → Session B (active) → 虚拟光标 B
  TaskCard C (active) → BrowserView C → Session C (active) → 虚拟光标 C
  三个任务真正同时执行

系统光标场景：
  TaskCard D (active) → 系统光标 → Session D (active)
  TaskCard E (queued) → 等待系统光标 → Session E (pending)
  串行排队（系统光标排他）

混合场景：
  TaskCard A (active) → BrowserView A（查网页）
  TaskCard D (active) → 系统光标（操作 Word）
  BrowserView 任务和系统光标任务可以同时进行
```

## Workspace（工作区）（LEASE.WORK）

v1.2 的工作区概念扩展：

```text
工作区 = BrowserView 实例（内置浏览器视图）
       + Session（持久记录）
       + 虚拟光标 overlay（可选）
       + 绑定的文件/资料
       + 任务面板视图
```

窗口布局由 AI 光标应用自行管理：

```text
分屏模式：多个 BrowserView 并排显示
标签模式：BrowserView 以标签形式切换
堆叠模式：BrowserView 层叠，前景一个，后台多个
浮窗模式：BrowserView 作为浮动窗口
```

用户可以通过语音控制布局：

```text
"左右分屏" → 两个 BrowserView 并排
"全屏显示这个" → 当前 BrowserView 最大化
"后台处理" → BrowserView 隐藏，静默执行
```

## 多光标安全约束（SAFE.MULTI）

v1.2 扩展安全规则以覆盖 BrowserView 并行场景：

```text
系统光标安全（沿用 v1）：
  Rule 1-9 生效

BrowserView 安全：
  Rule B8：BrowserView 内的高风险操作仍需用户确认
  Rule B9：跨 BrowserView 的数据传递需要记录（Session chunk）
  Rule B10：BrowserView 内的密码/敏感输入仍然 blocked
  Rule B11：用户说"全部暂停"时，所有 BrowserView 同时暂停

状态一致性安全：
  Rule S1：旧 state_id / ref / epoch 不得执行
  Rule S2：目标 Executor 不存在时拒绝，不得 fallback
  Rule S3：用户输入使相关状态失效
  Rule S4：结果 unknown 时不得自动重复有副作用动作

可见性安全：
  Rule V1：visible_system 模式下，用户可以随时看到光标操作
  Rule V2：visible_virtual 模式下，虚拟光标动画忠实反映真实操作
  Rule V3：silent 模式下，操作完成后必须向用户汇报结果
  Rule V4：用户可以随时切换任何任务的可见性模式
```

## 一句话总结

**CursorLease 是状态感知的资源调度器：Workspace 通过 `target_view` 隔离，实时操作通过 `resource_ref + resource_epoch` 校验，缓存查询与执行 Lane 分离；独立 BrowserView 可以并行，全局指针和键盘保持排他，用户接管会立即释放资源并使旧状态失效。**
