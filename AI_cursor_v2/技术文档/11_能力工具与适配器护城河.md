## 1. 核心判断

通用模型的"看屏幕→点击"是**万能下限**，不是**效率上限**。纯 GUI 操作要走一条脆弱长链：

```text
感知截图 → OCR/语义定位 → 找到目标（如某个单元格）
→ 理解操作语义 → 移动/点击/输入 → 回读验证
```

每一步都可能错，累积后又慢又不稳。通用能力人人可复制，**护城河来自：给这个"AI 员工"配一批合适且高效的工具/适配器，并让它们可信、可复用、越用越快。**

原则：**不是二选一，而是"能力阶梯 + 自动降级"**——同一任务准备多条通路，优先走最高保真的那条，没有才降级到 GUI 兜底。

## 2. 能力阶梯与自动降级（TOOL.LADDER）

沿用 `技术文档/02 §执行入口优先级`，把它上升为产品策略：Vlawd 是一个**能力路由器**，识别当前 App 与可用适配器，自动选最高保真通路，失败再降级。

以"修改一个已有 Excel 的某个值/公式"为例，保真度从高到低：

```text
1. 文件级结构化      openpyxl / pandas —— 不开界面，按单元格/表/命名区域精确定位，最快最准
2. 应用自动化 API    COM(pywin32) / Office-JS / AppleScript —— 操作打开的实例，支持公式、透视表等高级操作
3. MCP / 插件工具    标准化 Excel/Office MCP server —— 模型用工具调用而非像素点击
4. 语义层定位        "Q3 销售额那列" → 命名区域/表头/列，而非"第几行第几格"
5. GUI 兜底          以上都没有时才回到通用看屏点击（保证万能，但接受慢）
```

每次降级必须记录：原因、所需权限、是否改变前台焦点、结果能否独立验证（与 ACTION.VERIFY 一致）。已有结构化入口时，不得仅因模型更熟悉点击而选择高权限 UI 操作。

## 3. 两种受众，两种形态（不要混为一谈）

关键区分：**"给 AI 用"与"给人用"是两种形态**，不要用同一个东西同时满足。

```text
AI 消费  → 机器可读的工具接口（结构化参数进、结构化结果出）
人 观察  → 轻量透明面板（看 AI 在改什么、diff、确认/回滚）
```

一个"装完弹出窗口、AI 往里输入查找修改"的独立 App，是**给人的交互形态**；让 AI 去操作它等于把 AI 推回"看界面、找输入框、点击"的低效老路，与 §1 判断相悖。因此窗口应存在，但角色是**人面透明面板**，不是"AI 打字的地方"。

### 3.1 AI 面：工具契约（TOOL.CONTRACT）

AI 需要的不是窗口，而是**有类型契约、可发现的工具接口**：

```text
可发现   工具声明自己有哪些操作、参数、返回（schema）——模型一看就知道能干什么，不必瞎试
语义化   按命名区域/表头/对象定位，而非像素坐标
结构化   返回结构化结果 + 回读证据，便于校验
```

首选载体：

- **MCP 工具 / 类型化函数接口（headless）** 作为对外主契约——标准、语言无关、进程隔离（利于权限/沙箱）、可被其他 Agent 复用；
- 少数旗舰 App 可再配**一方内建适配器（进程内 SDK）**追求极致性能；
- 对模型而言两者统一表现为"工具"。

"本质是对终端的包装"这个直觉成立，但裸 CLI 对 AI 不友好（全字符串、要记 flag、无可发现性），必须升级为上面的类型化契约。这个"带清单的工具契约"正好就是 **Capsule**（Skill + Workflow + 权限清单 + Tests + Provenance + Runtime Metrics）。

### 3.2 人面：透明确认面板（TOOL.PANEL）

Vlawd 里的轻量面板（不是独立 App），复用 `overlay.*` 渲染通道，显示：

```text
定位到的目标（哪张表 / 哪个单元格 / 哪段文字）
将执行的操作（结构化 op 描述）
diff 预览（改动前 → 改动后）
确认 / 拒绝 / 回滚
Evidence（回读结果）
```

高风险写入走 `SAFE.LEVEL = confirmation_required`；面板本身不是执行者，AI 调用的是 §3.1 的工具契约。

## 4. App 适配器 / 工具 Capsule（TOOL.ADAPTER）

现在执行器只有 System / BrowserView（EXEC.DUAL）。长期新增一类 **App 适配器 / Skill Capsule**（ExcelAdapter、WordAdapter、BrowserAdapter、MailAdapter、SAPAdapter…），每个都是标准 Capsule：

```text
Skill            —— 能做哪些操作
Workflow         —— 常见任务的编排
Permission 清单  —— 需要的权限（读文件 / 写文件 / 网络 / 前台焦点）
Tests            —— 自检与回归
Provenance       —— 来源与信任
Runtime Metrics  —— 真实成功率、失败模式、适用环境
```

它们不属于 GUI 执行器，是 headless 结构化操作；通过 Environment Interaction Adapter（EXEC.INTERACT）挂接平台 backend，通过统一提案边界（ACTION.PROPOSAL）与 Safety（SAFE.LEVEL）门禁执行，结果回读产生 Evidence（ACTION.VERIFY / SESSION.EVID）。

## 5. 文档编辑能力：读-定位-补丁-校验（TOOL.DOCEDIT）

"从零生成"和"改已有"是两个不同问题，当前通用 Agent 只解决了前者。

```text
生成新文档   模型写 python 自上而下全新构造，全权掌控，不用理解任何已存在的东西 —— 现有能力可做
修改已有     必须 读懂 → 定位 → 最小化打补丁 → 校验，且无需可视化交互（纯 headless 结构化操作）
```

因此"修改已有文档"应作为一个独立的针对性 Capsule，四阶段：

```text
1. 解析建模   载入文档 → 可寻址结构模型（Excel: sheet/表/命名区域/单元格；
              Word: 节/段落/表格/run/样式），建索引，让模型"查"而不是"猜坐标"
2. 语义定位   把口语目标（"Q3 销售额那格"/"合同乙方名称"）映射到具体地址，
              靠表头/标签/命名区域/邻近文字接地；歧义时返回候选+置信度让用户确认
3. 最小化补丁 表达为结构化 op（diff，而非重写），只动目标：
              Excel: set_cell / set_formula / insert_row / format_range / locate_by_header
              Word:  replace_text(run-aware) / set_table_cell / insert_paragraph_after
4. 回读校验   重新打开读回目标、必要时重算公式、对比改动前后"只有预期变化"
              → 产生 Evidence；保留原件/生成新版本，可回滚
```

### 5.1 引擎选型（按文档特征选，避免踩坑）

```text
Excel:
  openpyxl        文件级读写最快，但可能丢图表/图片、且不重算公式
  COM / Office-JS 完整保真 + 重算，操作打开的实例
  LibreOffice headless  批量、无界面重算
  OOXML XML       库做不到的极端改动，直接改底层 XML

Word:
  python-docx     段落/表格/run，但文字会被拆进多个 run，
                  简单 find/replace 会失败 —— 需 run-aware 替换
  XML / 应用层     修订、页眉页脚等复杂元素
```

### 5.2 与 Cycle 的关系

这块是 **Phase 2A（受控执行）之后**的旗舰样板：拿"改一份已有 Excel 的某个值/公式"演示"读-定位-补丁-校验"闭环，对比 GUI 慢通路，讲清"工具护城河"的价值。它不是 Cycle 1–3 的内容。

## 6. 能力协商与按需装配（TOOL.INSTALL）

AI 在需要一个未装能力时，**征询用户是否安装插件 / MCP 工具**：

```text
1. 检测   任务需要一个未安装的能力（如 Excel 适配器）
2. 征询   弹出"要安装 Excel 适配器吗？"，附：它能做什么、需要哪些权限、
          来源与信任（Registry 已验证 + 运行信号）
3. 批准   用户同意 → 从 Registry 安装 → 经 Safety / Permission 门禁，先沙箱后放行
4. 记忆   记住选择，下次同类任务直接复用；优先推荐已验证、运行信号好的 Capsule
```

信任与安全要点：安装前展示 Permission Manifest、Provenance 与运行信号；第三方代码隔离执行（对齐 `PLUGIN.SANDBOX / PLUGIN.PERM`，deferred）；用户可撤销（对齐 `PERSON.PROTOCOL` propose→confirm→apply→revoke）。

这既是好体验，也把 Registry 变成工具市场——分发与网络效应都在这里（对齐 MOAT.NETWORK）。

## 7. Skill / 知识 / 结晶化（TOOL.SKILL）

工具不只是 API，而是 **API + Skill 库 + 记忆** 三层，目的是**避免 AI 重复摸索、越用越快**：

```text
内建 Skill/Workflow   常见任务怎么做（建透视表、批量替换）
项目专属知识          这份文档/这个项目的表结构、命名区域、约定
结晶化（最强）        把过去成功的操作固化成可复用宏，下次直接走高保真通路
```

结晶化闭环：一次靠 GUI 摸索成功的流程 → 记录引擎检测模式（对齐 `PERSON.WFGEN`）→ 提议固化为 Capsule/宏 → 用户确认 → 下次同类任务直接调用。这正呼应 Moltery"蜕壳/演化（molt）"的产品灵魂：用得越多，能力越沉淀、越快、越准。

## 8. 工具护城河定位（TOOL.MOAT）

通用模型是大宗商品；护城河是四层叠加（与 `Agentic_Web/战略与边界/02` 对齐）：

```text
1. 精选的高质量 App 适配器库    官方主推刚需：Excel / 浏览器 / 邮件 / IDE
2. 路由器 + Safety + 回读校验    把不稳的模型行为变成可信、可复现的操作（MOAT.TRUST）
3. 网络效应                     Moltery/Registry 上社区贡献并被验证的 Capsule，运行信号排序（MOAT.NETWORK）
4. 结晶化闭环                   用得越多越快越准（MOAT.STATE）
```

### 风险与取舍

```text
手工逐个造工具不可规模化   → 靠 MCP 标准 + 社区 + 自动结晶化
工具质量参差              → 用 Tests / 回读 / 运行信号治理（Capsule 内建）
强工具 = 高权限           → Permission Manifest + Safety Engine 收敛，先沙箱
跨平台（COM 仅 Windows）  → 适配器抽象统一接口（EXEC.INTERACT）
```

---

## 9. 一句话总结

**AI 用的是"带清单的工具契约（MCP / Capsule）"，人看的是"透明确认面板"；底层是被包装并叠加了 Skill / 知识 / 结晶宏的终端；能力经 Registry 按需征询安装。护城河不在通用模型，而在这套高效、可信、可复用、越用越快的工具/适配器体系——它在 MVP Gate 后按证据起步，成熟于 Agentic Web 阶段门，不提前建设。**
