## 核心结论

AI 光标的代码要能被 AI（LLM）理解和修改。

如果代码库庞大、耦合、隐式逻辑多，AI 改代码的成功率会急剧下降。所以架构必须面向 AI 可维护性设计。

v1.1 更新：v2 的两角色架构（全双工模型 + 记录引擎）进一步简化了系统结构，有利于 AI 可维护性。

## AI 可维护性的核心原则（GOVERN.PRINCIPLE）

### 1. 模块边界清晰

每个 bounded context 有明确的职责、输入和输出：

```text
V2 的 bounded contexts：

FullDuplexModel Context
  ・输入：用户音频、观察结果、Session 上下文
  ・输出：语音流、动作指令流

RuleEngine Context
  ・输入：音频流
  ・输出：安全控制信号

Observation Context
  ・输入：屏幕截图、DOM、Accessibility Tree
  ・输出：候选目标列表

Action Context
  ・输入：结构化动作协议
  ・输出：执行结果

Session Context（v1.1 新增）
  ・输入：全部数据流（用户音频、AI 输出、动作、结果）
  ・输出：Session chunks、Session Graph、Memory

Safety Context
  ・输入：动作协议
  ・输出：允许/确认/阻止

Personalization Context
  ・输入：Session 历史
  ・输出：偏好规则、工作流模板
```

### 2. 依赖方向单一

```text
FullDuplexModel → Observation（获取屏幕状态）
FullDuplexModel → Session（查询历史上下文）
FullDuplexModel → Action（下发动作指令）
Action → Safety（安全评估）
Action → Session（写入执行结果）
Session → Personalization（偏好学习）
RuleEngine → Action（安全控制抢占）
```

禁止反向依赖。

### 3. 接口比实现重要

每个模块对外暴露的接口要稳定、文档化：

```text
Observation.get_candidates() → List[ScreenElement]
Session.record_chunk(chunk) → ChunkId
Session.fork(reason) → SessionId
Session.recall(query) → List[Memory]
Action.execute(proposal) → ActionResult
Safety.evaluate(proposal) → SafetyDecision
```

内部实现可以随时替换（不同模型、不同算法），但接口不变。

### 4. 配置外置

```text
安全规则 → safety_rules.yaml
偏好默认值 → defaults.yaml
模型配置 → model_config.yaml
Session 存储配置 → session_config.yaml（v1.1 新增）
```

### 5. 命名自解释

```text
好：execute_mouse_click(target, button)
坏：do_action(x, y, t)

好：session.fork(reason="用户纠正")
坏：session.split(flag=1)
```

## Session Context 设计（v1.1 新增）（GOVERN.SESSION）

Session Context 是 v2 新增的 bounded context，管理所有 Session 相关逻辑：

### 职责边界

```text
Session Context 负责：
  ・创建/fork/merge/关闭 Session
  ・记录 chunk
  ・维护 Session Graph
  ・存储/查询 Memory
  ・渲染任务面板数据

Session Context 不负责：
  ・理解用户意图（全双工模型的事）
  ・执行动作（执行器的事）
  ・评估安全性（Safety Context 的事）
```

### 接口定义

```text
SessionService
  .create(task_name) → SessionId
  .record_chunk(session_id, chunk) → ChunkId
  .fork(session_id, reason) → SessionId
  .merge(source_id, target_id) → MergeResult
  .get_graph(session_id) → SessionGraph
  .recall(query) → List[Memory]
  .get_panel_data(view_type) → PanelData
```

### 存储设计

```text
Session 数据 → SQLite（本地持久化）
Session 索引 → 内存（快速查询）
Memory 索引 → SQLite + 向量索引（语义搜索）
```

## V1 → V2 架构简化

```text
V1 Bounded Contexts（9 个）：
  Interface → VoiceRuntime → IntentRouter → Planning → 
  Observation → Action → Safety → Trace → Personalization

V2 Bounded Contexts（7 个）：
  Interface → FullDuplexModel → RuleEngine →
  Observation → Action+Safety → Session → Personalization
```

V2 合并了 VoiceRuntime + IntentRouter + Planning 为 FullDuplexModel，新增了 Session Context，整体更简洁。

## 面向 AI 编码规范（GOVERN.CODE）

沿用 v1 的编码规范：

```text
1. 函数签名完整（参数类型 + 返回类型 + docstring）
2. 错误处理显式（不用裸 try-except）
3. 状态变化通过事件/消息传递
4. 测试覆盖核心路径
5. 避免魔数、隐式状态、全局变量
```

## 一句话总结

**AI 光标的代码架构面向 AI 可维护性设计：模块边界清晰、依赖单向、接口稳定、配置外置、命名自解释。v1.1 新增 Session Context 作为独立 bounded context，管理 Session 创建/fork/merge/查询的完整生命周期。V2 的两角色架构（全双工模型 + 记录引擎）将 9 个 bounded context 简化为 7 个，更利于 AI 理解和修改。**
