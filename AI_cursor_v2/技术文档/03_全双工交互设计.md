## 核心定位

v1 的 Voice Interaction Runtime 基于 VAD → ASR → LLM → TTS 四模块串联，延迟 1-3 秒，打断需要额外工程处理。

v2 改用**全双工语音模型**作为交互核心：

```text
用户音频 → 全双工模型 → 同时输出语音流 + 动作指令流
```

全双工模型能同时听和说，原生处理打断、纠正、附和，端到端延迟 < 200ms。

---

## V1 → V2 对比

| 维度 | V1 Pipeline | V2 全双工模型 |
|------|-------------|--------------|
| 架构 | VAD → ASR → LLM → TTS | 端到端单模型 |
| 延迟 | 1-3 秒 | < 200ms |
| 打断 | 外部 VAD 检测 + TTS 停止 | 模型原生处理 |
| 纠正 | 需要重新走完整 Pipeline | 模型内部实时调整 |
| 附和 | 不支持 | 原生支持（"嗯""好的"） |
| 轮替 | 明确的说-停-说 | 自然过渡，无需等待 |
| 部署 | 4 个独立模块 | 1 个模型 |

## 全双工模型的工作原理（DUPLEX.PRINCIPLE）

### 什么是"全双工"

传统语音交互是半双工：

```text
用户说 → AI 等待 → AI 说 → 用户等待 → 用户说 → ...
```

全双工模型同时处理输入和输出音频流：

```text
用户持续说话 ─────────────────────────→
AI 持续响应   ─────────────────────────→
              ↑ 同时进行，互不阻塞
```

### 核心能力

全双工模型必须具备以下能力：

```text
1. 平滑轮替（Smooth Turn-Taking）
   自然地接话，无需等待明确停顿

2. 用户打断（User Interruption）
   用户开口时立刻停止说话，切换为听

3. 暂停处理（Pause Handling）
   区分"用户在想"和"用户说完了"

4. 附和反馈（Backchanneling）
   在合适时机说"嗯""好的""我知道了"

5. 纠正处理（Correction Handling）
   理解"不对""换一个""右边那个"并调整

6. 实体追踪（Entity Tracking）
   记住对话中提到的对象，支持代词指代
```

## 全双工交互模式在 AI 光标中的体现（DUPLEX.INTERACT）

### 场景 1：正常任务指挥

```text
用户："帮我搜一下明天北京的天气"
AI（同时）："好的，正在搜"
AI 执行：打开浏览器 → 输入搜索 → 找到结果
AI："明天北京 25 度，晴天"
```

全双工模型在理解用户意图的同时就开始语音反馈，无需等理解完成。

### 场景 2：用户打断

```text
AI："我找到了 5 个结果，第一个是——"
用户："等下，先看第二个。"
AI（< 200ms 内停止）："好的，我打开第二个。"
```

模型原生检测到用户开口，立刻停止当前输出并理解新指令。

### 场景 3：用户纠正

```text
AI 正在点击左边的按钮
用户："不对，右边那个"
AI："好的，改点右边。"
记录引擎：fork 新分支，记录纠正事件
```

纠正在模型内部直接处理，不需要重走 Pipeline。Session Graph 同时 fork 新分支记录纠正历史。

### 场景 4：自然附和

```text
用户："帮我看看这个岗位……"
AI："嗯"（附和，继续听）
用户："……要求 3 年经验的那种"
AI："好的，我帮你筛选 3 年经验以上的。"
```

AI 在用户说话的间隙自然附和，让用户知道 AI 在听。

## 安全控制词的本地抢占（DUPLEX.RULE）

尽管全双工模型能处理打断，但安全控制词必须由**本地规则引擎抢占**，不经过模型：

```text
用户说"停" → 本地规则引擎 → 立即暂停执行器 → 延迟 < 50ms
```

必须本地抢占的词：

```text
停
暂停
取消
不要
停下来
我来
退回
```

原因：

- 模型推理有延迟，安全控制不能等
- 即使模型崩溃，安全控制仍然有效
- 安全控制是硬实时要求

规则引擎和全双工模型的协调：

```text
音频流 → 同时送入本地规则引擎和全双工模型
         │
         ├→ 规则引擎检测到安全词 → 立即中断执行 → 通知全双工模型
         │
         └→ 全双工模型正常处理 → 输出语音 + 动作
```

## 全双工模型的输出设计（DUPLEX.OUTPUT）

全双工模型需要同时输出两路数据：

### 语音流（Voice Stream）

```text
→ 直接送入耳机播放
→ 短句、确定、任务化
→ 支持流式播放
→ 被打断时立即停止
```

### 动作指令流（Action Command Stream）

```text
→ 结构化动作协议
→ 送入 Safety Policy 评估
→ 通过后送入 Execution Scheduler
→ 执行器执行真实操作
```

动作指令格式：

```json
{
  "action": "mouse.click",
  "target": {
    "type": "screen_element",
    "ref": "element_3",
    "description": "第一个搜索结果"
  },
  "safety": "safe",
  "expected_result": "打开搜索结果详情页"
}
```

## 与 Session 记录引擎的配合（DUPLEX.SESSION）

全双工模型的所有输入输出都被记录引擎捕获：

```text
记录引擎监听：
├── 用户音频/转写 → 记录为 user_voice chunk
├── 模型语音输出 → 记录为 ai_response chunk
├── 动作指令 → 记录为 action chunk
├── 执行结果 → 记录为 result chunk
├── 用户打断 → 记录为 interruption chunk
└── 用户纠正 → 触发 Session fork
```

当全双工模型需要历史上下文时，向记录引擎查询：

```text
全双工模型："用户之前查过什么天气？"
记录引擎：搜索 Session 历史 → "昨天查过北京后天天气"
全双工模型：利用上下文推断用户意图
```

## 评测基准（DUPLEX.BENCH）

采用 FullDuplexBench（v3.0）评测维度，定制 AI 光标专用基准：

| 评测维度 | 说明 | 目标 |
|---------|------|------|
| 打断响应延迟 | 用户说"停"后多久 AI 停止操作 | < 200ms |
| 纠正理解率 | "不对，右边那个"能否正确理解 | > 90% |
| 上下文召回率 | "再做一次昨天那个"能否找到历史 | > 85% |
| 误打断率 | AI 误以为用户打断（背景噪音） | < 5% |
| 平滑轮替成功率 | 轮流说话的过渡是否自然 | > 90% |
| 附和适当率 | AI 在合适时机说"嗯""好的" | > 80% |

## 候选全双工模型（MODEL.CANDIDATE）

| 模型 | 参数量 | 轮替延迟 | 打断延迟 | 中文能力 | 开源 | 适合场景 |
|------|--------|---------|---------|---------|------|---------|
| GLM-4-Voice | 9B | ~300ms | ~400ms | 原生中文 | 是 | 中文市场首选 |
| BayLing-Duplex | 9B | ~300ms | ~400ms | 原生中文 | 是 | 中文全双工微调版 |
| PersonaPlex | 7B | 170ms | 240ms | 待验证 | 是 | 延迟最低 |
| Moshi | 7B | ~200ms | ~300ms | 弱 | 是 | 英文市场 |

推荐选型策略：

```text
中文市场 → GLM-4-Voice + BayLing-Duplex（原生中文，全双工微调）
英文市场 → PersonaPlex（延迟最低）或 Moshi（社区成熟）
```

## 与 V1 的兼容：降级方案（DUPLEX.FALLBACK）

如果全双工模型不可用（硬件不足、模型不稳定），系统可以降级为 V1 Pipeline：

```text
降级模式：
VAD + 本地 ASR（SenseVoice/faster-whisper）
→ LLM 理解（本地小模型或云端）
→ TTS 播报（Supertonic 3/Kokoro）

触发条件：
- GPU 显存 < 8GB
- 全双工模型加载失败
- 用户手动选择"兼容模式"
```

降级模式下所有功能保持可用，只是延迟从 200ms 退回到 1-3 秒。

## 技术栈建议（DUPLEX.STACK）

### 全双工模型部署

```text
Runtime：vLLM / SGLang / llama.cpp
格式：GGUF（CPU+GPU 混合推理）或原生 PyTorch
量化：INT8 或 INT4（降低显存占用）
音频接口：本机系统音频 API（WASAPI / ALSA / CoreAudio）
```

### 本地规则引擎

```text
实现：有限状态机 + 关键词匹配
延迟目标：< 50ms
部署：纯 CPU，无额外依赖
```

---

## 一句话总结

**全双工模型替代了 V1 的 VAD → ASR → LLM → TTS 四步 Pipeline，实现同时听说、原生打断、实时纠正，交互延迟从秒级降至 200ms 以内。安全控制词由本地规则引擎抢占处理，保证即使模型故障也能立即喊停。全双工模型的所有输入输出被记录引擎捕获，沉淀为可回溯的 Session Graph。**
