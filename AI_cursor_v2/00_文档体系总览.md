## 产品一句话

**AI 光标是一个全双工耳机协作式 AI 桌面员工：用户通过耳机实时对话指挥 AI 在屏幕上可见地操作电脑，所有操作沉淀为可回溯的 Session，成功流程沉淀为可复用工作流。**

---

## 与 Agentic Web 项目群的版本关系

```text
Vlawd 产品演进：
AI_cursor_v1 → AI_cursor_v2 → 未来真正的 AI_cursor_v3

项目群蓝图演进：
Agentic_Web v1.0 → v1.1 → v1.2 → v2.0
```

`AI_cursor_v2/` 继续负责 Vlawd 的产品定位、交互、Runtime、Session、UI、MVP 和代码。`Agentic_Web/` 是与 V2 并行迭代的项目群蓝图，只管理长期愿景、项目关系、共享对象和跨项目阶段门。

项目群蓝图变化不会自动提高 Vlawd 主版本。只有 Vlawd 本身发生不可兼容的产品或架构替代时，才创建真正的 `AI_cursor_v3/`。

## V1 → V2 核心变化

```text
全双工模型替代 ASR → LLM → TTS Pipeline → 延迟从秒级降至 200ms
两角色架构替代九层架构 → 全双工模型（大脑）+ 记录引擎（笔记本）
Session Graph 替代聊天记录 → fork/merge 支持语音纠正和回溯
右下角任务面板 → 日志/树/拓扑三视图，用户和 AI 共享外部记忆
竞争策略调整 → 本地隐私 + 全双工实时 + Session 可回溯 vs Codex 云端异步
```

详见 [变更记录/V1_to_V2_变更说明](./变更记录/V1_to_V2_变更说明.md)

## 文档管理规则

### 版本号规则

```text
主版本号（v1 → v2）：架构级重写，整个模块逻辑推翻重来
次版本号（v2.0 → v2.1）：模块内局部调整
```

### 两层版本管理

```text
全局（本文件）：模块版本总表 → "系统整体是什么状态"
模块内（每个文件顶部）：变更日志 → "这个模块为什么变成了现在这样"
```

### 修改流程

```text
1. 查看本文件的版本表 → 定位要修改的模块
2. 打开目标模块 → 查看顶部变更记录 → 了解当前版本状态
3. 整段重写受影响的章节（不做增量补丁）
4. 更新模块顶部变更记录（新增一行）
5. 更新本文件的版本表（版本号 + 日期 + 摘要）
6. 如果涉及多个模块，检查交叉引用一致性
7. 如果新增/修改的功能涉及 MVP 范围变更，同步更新 02_MVP开发规划与阶段路线.md
```

### 编号规则

```text
格式：前缀.后缀（全大写英文，点号连接）

前缀 = 模块/子系统名称简称（≤ 8 字母）
后缀 = 功能点名称简称（≤ 8 字母）

约定：
  1. 编号一旦分配即固定，不随功能改名而变更
  2. 废弃的编号标记 deprecated，不回收复用
  3. 新增功能只需起不重复的名字，无顺序依赖
  4. 编号在章节标题后以（编号）格式标注
  5. 所有编号在 01_功能模块注册表.md 中注册
```

### 阶段管理原则

```text
完整设计文档描述最终目标状态，编号标记功能单元。
MVP 规划文档以产品 Cycle 定义交付顺序，再为当前能力配置风险门槛和最小工程范围。
阶段变更只修改 02_MVP开发规划与阶段路线.md，不修改设计文档。
设计文档更新不影响阶段归属，编号保持稳定。
历史代码 Phase / 执行清单只记录工程骨架，不决定产品路线，也不代表 Dogfood 已通过。

三个根级文件的职责：
  00_文档体系总览     → 规范（版本管理 + 编号规则 + 阶段管理原则）
  01_功能模块注册表   → 注册（编号 → 文档位置 → 功能描述 → 状态）
  02_MVP与阶段路线   → 规划（产品 Cycle → 风险门槛 → 工程范围 → 退出证据）

查询路径：
  02_MVP规划  → "当前要证明什么体验，必须先满足什么边界？"
  01_注册表   → "这些编号对应哪篇文档的哪个章节？"
  具体模块文档 → "这个功能的详细设计是什么？"
```

  使用路径和版本矩阵
