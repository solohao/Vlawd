# 07 · 桌面端优先与分场景产品路线

## 核心结论

这条产品路线需要进一步收束为：

> **第一阶段不是手机 App，不是小程序，也不是先做硬件，而是先做桌面端 AI 客户沟通工作台；手机与 IM 入口承担轻入口和远程触达；中后期再延展到 Agent Slate 与企业中枢工作站。**

这样做的原因不是桌面端更“高级”，而是它更适合个人创业者和早期团队先验证：

- 用户是否真的愿意为“沟通内容 → 结构化结果 → 下一步动作”付费
- 哪个垂直场景最先成立 PMF
- 哪种输入设备最值得深度支持
- 哪些输出工作流最有业务价值

---

## 为什么第一阶段应以桌面端为产品本体

### 1. 桌面端更适合做主工作台

桌面端天然更适合承载：

- 多输入导入
- 转写和结构化整理
- A2UI 卡片确认
- 客户、项目、任务、工单、访谈等资产沉淀
- 多工具同步
- 长时间连续使用
- 本地数据与隐私控制

相比之下，手机端更适合：

- 临时录音
- 消息提醒
- 轻确认
- 补充输入
- 移动触达

所以第一阶段更准确的产品定义不是“手机 / 桌面双端一起做”，而是：

```text
桌面端 = 主工作台
手机 / IM = 轻入口与远程触达
硬件 = 软件 PMF 之后的增强形态
```

### 2. 桌面端更适合个人创业者起步

早期如果先做手机 App、小程序或重 Web，会同时面对：

- 安卓 / iOS 双端适配
- 录音与后台权限限制
- 小程序权限和审核限制
- 支付与资质问题
- 更重的分发和合规成本

桌面端则更适合先验证：

- 输入导入是否高频
- 垂直 Skill 是否有明显价值
- A2UI 是否真的能降低 Agent 使用门槛
- 用户是否愿意沉淀长期数据资产

---

## 第一阶段产品本体：桌面端 AI 客户沟通工作台

第一阶段桌面端产品的本质不是“语音记录软件”，而是：

> **一个把客户沟通、售后反馈、用户访谈、产品讨论、创业想法转化为结构化工作资产的桌面工作台。**

它应重点完成以下能力：

- 接入多种输入来源
- 进行转写与场景识别
- 输出垂直 Skill 结果
- 通过 A2UI 让用户确认、修改和执行
- 沉淀长期数据资产
- 同步到 CRM、任务、工单、知识库等工具

### 第一阶段推荐输入

- 电脑麦克风
- 普通蓝牙耳机麦克风
- 手机导入录音
- 上传音频文件
- 粘贴聊天记录
- 导入会议纪要或转写文本

### 第一阶段推荐输出

- 客户跟进卡
- 工单卡
- 访谈洞察卡
- 假设验证卡
- 任务列表
- 日历提醒
- 邮件 / 微信回复草稿
- CRM / 知识库同步候选

## 第一阶段资产与页面设计

### 1. 先把卡片做成资产本体

第一阶段最重要的不是先做很多页面，而是先定义清楚：

> **系统最终沉淀的对象，应该先以卡片化资产存在。**

这些卡片不是一次性摘要，而是后续可确认、可关联、可再利用的资产原子。

第一阶段建议优先沉淀的资产类型包括：

- 输入来源卡：录音、转写、聊天、会议、访谈、讨论记录
- 洞察卡：需求、痛点、异议、风险、机会、证据
- 动作卡：跟进任务、回访提醒、行动项、待办
- 实体卡：客户、用户、项目、功能点、问题域
- 判断卡：假设、结论、决策、方案方向

### 2. A2UI 是资产入库前的确认层

第一阶段不应让 AI 直接把所有结果写入资产库，而应先经过：

```text
输入
→ Skill 结构化处理
→ A2UI 卡片确认
→ 进入资产库
→ 同步到外部工具
```

这意味着：

- 首页中的卡片首先是“待确认候选”
- 被确认后的卡片才进入长期资产层
- 驳回、修改、延后本身也是重要反馈信号

### 3. 资产视图要分层，而不是一开始全做

资产相关页面可以分成三层：

#### 第一层：基础工作台视图

第一阶段优先做：

- 待确认卡片列表
- 资产列表
- 项目、客户、访谈等基础聚合页
- 最近沉淀资产

这是最适合 MVP 的主界面。

#### 第二层：分析视图

中期适合增加：

- 需求功能矩阵
- 风险与证据矩阵
- 时间线视图
- 关系图谱

这些视图的本质不是新的资产类型，而是：

> **同一批卡片资产的不同观察方式。**

#### 第三层：自由分析画布

类似“白板 + 卡片”的分析画布是有价值的，尤其适合：

- 创业产品推演
- 用户访谈洞察整理
- 客户关系与问题链路分析
- 多实体、多事件的复杂归因

但它更适合作为中后期高级视图，而不是第一阶段首页主形态。

### 4. 第一阶段不宜把白板做成主入口

白板类界面虽然表达力强，但第一阶段风险较高：

- 交互复杂度高
- 自由度过大
- 用户学习成本高
- 很容易变成“看起来很强，但不够高频”

因此更合理的顺序是：

```text
先做卡片资产库
→ 再做矩阵与列表视图
→ 最后再做自由分析画布
```

### 5. 外部连接比高级视图更值得优先做

对第一阶段用户来说，很多时候真正重要的不是“又多一个新页面”，而是：

> **系统能不能把确认后的结果送到他已经在使用的地方。**

因此第一阶段更值得优先做的出口包括：

- 飞书文档生成
- 飞书表格追加或更新
- Notion 页面生成
- CRM 更新候选
- 邮件 / 微信回复草稿
- Webhook / API 单向推送

### 6. 第一阶段连接器优先级

建议按以下顺序推进：

#### P0：单向导出

- 一键生成文档
- 一键生成表格行
- 一键复制结构化结果

#### P1：确认后单向同步

- 确认后写入飞书文档
- 确认后追加飞书表格
- 确认后推送 CRM / 项目系统候选

#### P2：中后期再做的能力

- 双向同步
- 插件式复杂编排
- 外部系统改动反向回写
- 大规模自动更新多个系统

### 7. 对创业 / 访谈场景的特殊价值

如果第一阶段主要先用于产品访谈、创业讨论和需求验证，那么资产层尤其适合沉淀：

- 用户原话
- 痛点卡
- 需求卡
- 反对意见卡
- 假设卡
- 下一步验证任务

之后再把这些卡片投影成：

- 需求功能矩阵
- 假设与证据矩阵
- 项目白板
- 飞书文档 / 飞书表格输出

这会比一开始直接做成“很多独立页面”更稳定。

### 8. 一句话原则

> **先做资产协议，再做资产视图，最后做资产流转生态。**

也就是说：

- 卡片资产本体优先于白板画布
- 外部连接优先于复杂自由分析
- 真正的价值来自“确认后的结果是否进入真实工作流”，而不只是“页面是否丰富”

## 移动端不是缺席，而是采用“轻入口 + 远程桥接”策略

第一阶段不强做完整手机 App，并不意味着移动端缺席。

更合理的策略是：

> **桌面端负责复杂处理与资产沉淀，微信 / QQ / 飞书 / 钉钉 / QClaw 等承担轻入口、消息通道与远程触达。**

### 可以承担的移动能力

- 发送文字、语音、文件触发桌面端处理
- 查看结果摘要
- 做轻确认
- 收到任务提醒和同步结果
- 在外部场景下远程调用桌面端能力

### 不应夸大承诺的部分

- 不承诺完整替代原生手机 App 的所有能力
- 不承诺自动读取所有聊天上下文
- 不承诺直接控制所有手机录音能力
- 不承诺自动捕获所有通话音频

所以更准确的叙述是：

```text
移动端不是完整产品本体
而是桌面端工作台的远程入口、轻触达层和补充输入层
```

---

## 桌面端也需要“轻入口层”

### 为什么桌面主工作台之外还需要轻入口

即使第一阶段的产品本体是桌面端工作台，也不意味着用户每一次都愿意先打开完整主界面。

真实使用中，高频触发往往是碎片化的：

- 临时记录一句话
- 沟通刚结束时快速补充上下文
- 想知道系统现在是否在监听、转写、处理中
- 想快速结束本次记录并生成结果
- 想在不跳出当前软件的情况下查看待确认结果

因此更合理的产品结构是：

```text
桌面主工作台
= 复杂处理、A2UI 确认、资产沉淀、外部连接

桌面轻入口层
= 快速触发、状态感知、轻交互、快速跳转
```

### 轻入口层的目标

轻入口层不应替代主工作台，而应承担四件事：

- 降低启动成本
- 提高输入触发频率
- 持续向用户传达系统状态
- 在必要时把用户引导回主工作台

### 第一阶段适合的桌面轻入口形态

第一阶段更适合采用以下组合，而不是一开始做成复杂新界面：

- 全局快捷键
- 系统托盘常驻入口
- 小型悬浮窗 / 悬浮胶囊 / 状态球
- 任务完成后的轻量通知与待确认提醒

其中更值得优先考虑的是：

> **悬浮状态入口 + 全局快捷键 + 主工作台**

### 悬浮窗最适合承担什么角色

悬浮窗不应是缩小版首页，而更适合作为：

> **低干扰状态入口 + 快速动作入口 + 通往主工作台的桥。**

它更适合承载：

- 当前模式显示
- 当前状态显示
- 开始 / 暂停 / 结束记录
- 快速查看最近一次结果
- 提示有多少待确认卡片
- 一键打开主工作台

### 悬浮窗不适合承担什么

第一阶段不建议把以下能力堆进悬浮窗：

- 复杂 A2UI 编辑
- 大量卡片浏览
- 资产库深度管理
- 白板、矩阵、关系图谱
- 多层导航和复杂设置

这些能力都更适合留在桌面主工作台中。

### 状态动效为什么重要

这个产品不是静态工具，而是一个有明显处理阶段的系统：

- 待机
- 正在监听
- 正在转写
- 正在结构化处理
- 已生成待确认结果
- 正在同步或同步完成

如果没有清晰的状态反馈，用户会感觉系统是黑箱。

因此，动效在这里的价值不是装饰，而是：

- 降低不确定感
- 传达系统当前阶段
- 建立“系统在线且可控”的感觉

### 第一阶段适合表达的状态

建议第一阶段至少让轻入口表达这些状态：

- 待机
- 正在收音
- 正在处理
- 有待确认结果
- 同步完成
- 同步失败或需要用户处理

### 动效设计原则

轻入口上的动效应服务于状态沟通，而不是追求炫技。

更适合的方向是：

- 待机时轻微呼吸感
- 收音时有明确波形或能量反馈
- 处理中用流动或进度感表达
- 待确认时用数量提示或柔和高亮提醒
- 出错时用清晰但克制的警示状态

### 需要避免的风险

#### 1. 不要做成一直打扰用户的 AI 宠物

- 不要频繁跳动
- 不要过度吸睛
- 不要遮挡主要工作区域

#### 2. 不要把悬浮窗做成缩小版首页

- 轻入口负责触发和状态
- 主工作台负责闭环和资产

#### 3. 不要让轻入口绕开 A2UI 边界

- 轻入口可以做轻确认
- 但高风险动作仍应进入正式确认层

### 第一阶段建议优先级

#### P0

- 全局快捷键
- 托盘入口
- 小型悬浮状态窗
- 开始 / 结束记录
- 当前状态提示

#### P1

- 待确认数量提示
- 最近一次结果预览
- 快速打开待确认面板
- 当前模式切换

#### P2

- 更丰富的悬浮微交互
- 多任务并行状态
- 与 Agent Slate 或第二屏联动

### 与 Typeless 这类产品的关键差异

如果我们也做轻入口，不能让它只像一个“更聪明的语音输入悬浮球”。

更关键的是让用户感知到：

- 输入之后不是只得到一段文字
- 系统会进一步做结构化整理
- 会产生待确认卡片
- 结果会进入资产或外部工具

所以轻入口上真正应该被看见的不只是“正在转文字”，还包括：

- 正在提取需求
- 已生成 2 条待确认行动项
- 检测到 1 条风险
- 待同步到飞书 / CRM

## 分场景产品规划

### P0：销售 / 商务 / 客户经理

这是最适合优先验证 PMF 的场景。

#### 典型问题

- 客户沟通多，跟进动作密
- 沟通之后容易遗漏需求、异议和承诺事项
- CRM 填写成本高
- 会议纪要不是最终目标，真正目标是推进商机

#### 产品形态

```text
桌面端客户沟通工作台
+ 微信 / QClaw 远程入口
+ CRM / 日历 / 任务同步
```

#### 核心输出

- 客户摘要
- 需求点
- 异议点
- 跟进任务
- 回访提醒
- 微信回复话术
- CRM 更新候选

### P1：现场售后 / 实施 / 客户成功

这是第二优先级，但很强。

#### 典型问题

- 现场环境复杂
- 反馈多为口语化、碎片化
- 需要快速生成工单、问题描述、处理建议和回访动作
- 需要知识库沉淀

#### 产品形态

```text
桌面端工单 / 反馈处理工作台
+ 手机录音 / 领夹麦 / 录音豆输入
+ 企业微信 / 飞书 / 钉钉同步
```

#### 核心输出

- 工单卡
- 问题分类
- 严重程度判断
- 处理建议
- 回访任务
- 知识库条目

### P2：产品经理 / 用户研究 / 创业者

这个场景更适合形成高质量内容资产。

#### 典型问题

- 访谈很多，但结论分散
- 讨论和灵感容易丢失
- 真正需要的是需求、假设、证据链与行动项

#### 产品形态

```text
桌面端访谈与推演工作台
+ 音频导入 / 访谈记录 / 讨论文本导入
+ A2UI 洞察确认
+ 假设与需求资产沉淀
```

#### 核心输出

- 用户痛点
- 需求分类
- 机会点
- 风险与证据链
- 假设卡
- 行动项

## 分产品形态规划

### 1. 个人版：桌面端工作台

这是最早成立的产品本体。

#### 角色

- 个人客户沟通整理台
- 个人任务生成台
- 个人沟通资产沉淀台

#### 适合对象

- 销售
- 商务
- 咨询顾问
- 产品经理
- 创业者

### 2. 团队版：共享资产与协同工作台

当个人版跑通后，团队版自然出现。

#### 新增能力

- 共享客户库
- 共享项目库
- 团队任务协同
- 角色与权限
- 团队 Skill 模板
- 飞书 / 企微 / 钉钉 / CRM 更深集成

### 3. 企业版：企业 Agent 中枢

企业内部部署更适合单独定义为：

> **企业 Agent 中枢工作站 / 企业内网 Agent Hub**

#### 它不是个人设备，而是组织级基础设施

主要承担：

- 统一设备接入
- 统一权限
- 统一审计
- 统一工作流编排
- 统一企业系统同步
- 私有化部署
- 数据脱敏与安全边界控制

#### 部署形态

- 内网服务器
- 私有云实例
- 部门小主机
- 会议室中枢主机
- 门店 / 现场服务点中枢

### 4. Agent Slate：个人 / 专业用户的控制板

这是中后期的增强形态，不是第一阶段前提。

它更像：

- 低干扰工作流控制板
- A2UI 轻确认终端
- 输入设备管理面板
- 桌面端的轻量协同硬件

它适合：

- 外出拜访
- 售后现场
- 会议桌面
- 作为桌面端第二屏

### 5. 收音豆 / 录音增强设备

它的意义是提升输入质量，而不是替代桌面端。

角色应明确为：

- 高质量近场收音设备
- 断网缓存设备
- 一键触发输入设备
- 与桌面端 / 企业版协同的输入增强层

## 产品路线中的关键边界

### 1. 我们接入的是系统输入源，不是直接控制耳机底层协议

更准确的技术边界是：

```text
耳机 / 麦克风 / 录音设备
↓
操作系统或现成平台识别
↓
我们的产品接入系统暴露出的输入源或导入内容
↓
进行理解、确认、执行与沉淀
```

这意味着第一阶段不要承诺：

- 直接连接所有蓝牙耳机
- 接管厂商私有耳机能力
- 绕过系统直接抓取设备音频
- 自动控制所有蓝牙协议层行为

### 2. 护城河不在耳机控制权，而在工作流层

长期护城河是：

- 专业 Skill
- A2UI
- 长期数据资产
- 工具连接与工作流编排
- 企业权限与审计能力

而不是：

- 蓝牙协议控制权
- 单一硬件形态
- 通用聊天能力

## 分阶段路线更新

### 2026：桌面端优先的垂类 MVP

```text
桌面端 AI 客户沟通工作台
+ 多输入导入
+ 官方高价值 Skill
+ A2UI 卡片雏形
+ 轻量工具同步
```

目标：

- 先验证一个垂直场景
- 不做硬件
- 不强做完整手机 App
- 先让桌面端成为主工作台

### 2027：数据资产与 IM 远程桥接

```text
桌面端工作台
+ 微信 / QQ / 飞书 / 钉钉 / QClaw 轻入口
+ 个人 / 团队资产库
+ 更多官方 Skill
```

目标：

- 建立“桌面端 + 轻移动桥接”的产品结构
- 提升输入频率
- 让用户开始沉淀资产

### 2028：团队版与企业 Agent Hub 软件版

```text
Team 版
+ 权限与共享资产
+ 企业私有 Skill
+ 企业 Agent Hub 软件部署版
+ 有限的 Skill Builder
```

目标：

- 从个人工具走向团队工作流系统
- 开始形成企业版逻辑

### 2029：硬件增强与企业中枢工作站

```text
Agent Slate
+ 收音豆 / 输入增强设备
+ 企业中枢工作站硬件版
+ 本地模型 / 端侧能力增强
```

目标：

- 补足低摩擦输入和本地处理
- 建立更强企业闭环
- 强化隐私和离线场景能力

### 2030：平台化与生态化

```text
Skill / Agent 市场
+ A2UI 模板生态
+ 企业 API
+ 行业化产品线
+ 硬件合作生态
```

目标：

- 从产品走向平台
- 从官方能力走向生态供给
- 从单场景走向行业扩展
