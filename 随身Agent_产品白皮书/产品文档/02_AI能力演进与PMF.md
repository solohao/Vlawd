# 02 · AI 能力演进与 PMF 路径

## 核心结论

- **现在-2 年内**：能稳定支撑"普通耳机语音输入 + 专业 Skill 整理 + A2UI 确认 + 轻执行"的闭环
- **2-3 年内**：能稳定支撑"用户数据资产 + 个性化 Skill + 半自动 Agent 生成"的闭环
- **3-5 年内**：能稳定支撑"Agent 市场 + 多工具执行 + 长期记忆"的闭环
- **5-10 年内**：才可能稳定支撑"全天候随身 Agent"的完整愿景

但**不是所有维度同步进步**。有些会先到位（ASR / 成本），有些会卡很久（长期记忆 / 多 Agent）。

---

## 闭环 7 个环节的当前能力评估

| 环节 | 当前状态 | 2-3 年判断 |
|---|---|---|
| 普通耳机输入 | ✅ 基本可用 | ✅ 成熟 |
| ASR | ✅ 可用但成本敏感 | ✅ 近免费 |
| 场景识别 / 任务抽取 | ⚠️ 是最大不确定性 | ⚠️ 大幅提升但仍有边缘 case |
| 专业 Skill | ✅ 可工程化 | ✅ 可规模化 |
| A2UI 确认 UX | 🟡 需要产品打磨 | ✅ 可成为核心差异化 |
| 用户确认 UX | ✅ 工程问题 | ✅ 可控 |
| Agent 执行 | ⚠️ 60-80% 成功率 | ✅ 85%+ |
| 长期记忆 | ⚠️ 初级 | ⚠️ 进步缓慢 |
| 成本 | ⚠️ 必须漏斗 + BYO | ✅ 下降 10x |

## 能不能闭环？

### 现在能闭环的部分（短期可做）

- ✅ 语音捕捉 + 转写 + 整理
- ✅ 普通耳机 / 手机 / 桌面麦克风输入
- ✅ 专业 Skill 输出结构化结果
- ✅ A2UI 卡片 + 手动确认
- ✅ 轻量 Agent 执行（写邮件、发消息、生成草稿）
- ✅ 会议纪要 + 知识沉淀（已被 Plaud / 飞书 / viaim 验证）
- ✅ 销售/商务/售后/实施/访谈等垂类工作流

足够撑起一个 **PMF 子集**。

### 中期才能闭环的部分（1-2 年）

- ⚠️ 复杂跨软件工作流
- ⚠️ 长期记忆驱动的主动行动
- ⚠️ 根据用户持续数据半自动生成个人 Agent
- ⚠️ 多 Agent 协作
- ⚠️ "说一句就办完"的体验

### 暂时别碰

- ❌ 全天后台 Agent 自主运行
- ❌ AI 主动替你处理大量事务
- ❌ 实时陪伴式 AI

## 实时交流的 4 个能力 + 时间线

### 能力 1：实时语音对话 ✅ 已到位

- GPT-4o / Gemini Live / Claude Voice 已商业化
- 延迟 < 500ms

### 能力 2：边聊边理解任务 ⚠️ 2-3 年

- 流式 Agent 是研究前沿
- 增量任务建模、多任务并行追踪

### 能力 3：实时确认 + 动态调整 ⚠️ 2-3 年

- 指代消解、任务状态追踪、撤销回滚

### 能力 4：后台同步执行 ⚠️ 3-5 年

- 并行 Agent 调度、风险评估、错误恢复

### 三个时间画面

| 年份 | 体验 |
|---|---|
| 2027 | 多轮简单任务，AI 准确响应并确认 |
| 2029 | 对话过程中后台并行准备，"已发，要看一下吗？" |
| 2031 | 真人助理级体验，主动安排、提醒、执行、学习 |

## OpenClaw 算力成本问题

### 成本来源

- 语音成本（ASR / 长录音转写）
- 推理成本（Tool use / 多步规划 / 长上下文）
- 工具执行成本（爬网页 / 调接口 / 反复重试）
- 记忆和索引成本（向量化 / 检索 / 同步）

### 解决方案：漏斗式 AI 架构

```text
普通耳机 / 手机 / 桌面语音
↓ 按键 / 快捷入口 / 通话后补记
↓ 低成本 ASR
↓ 场景识别
↓ 专业 Skill
↓ 任务价值评分
↓ A2UI 结构化卡片
↓ 用户确认 / 规则判断 / 工具同步
↓ OpenClaw 或其他 Agent 执行（只处理最后 5-20%）
```

### 任务三分类

| 类型 | 描述 | 处理方式 | 成本 |
|---|---|---|---|
| A 类：只记录 | 灵感、备忘、观察 | 转写 + 入资产库 | 极低 |
| B 类：Skill 处理 | 销售摘要、售后工单、访谈整理、会议行动项 | 小模型 + 专业模板 | 中 |
| C 类：A2UI 确认 | CRM 更新、工单生成、邮件草稿、日历任务 | 卡片确认 + 工具同步 | 中 |
| D 类：真执行 | 写邮件、自动填表、跨工具 | 用户确认 + Agent 后端 | 高 |

### 预算系统

- 每个任务有工具调用次数上限
- 每个 Agent 有运行时长上限
- 默认先生成计划再执行，由用户确认
- 高风险/不可逆操作必须二次确认

### 商业模式

- **免费版**：少量记录 + 基础转写 + 几乎无 Agent
- **Pro 版**：固定 Agent 执行额度
- **Power 版**：高级模型 + 长任务 + 私有知识库
- **BYO Key / BYO OpenClaw**：把重度用户成本转移出去

## PMF 候选切入点

### 选项 A：销售/商务的"普通耳机 AI 跟进助手"

```text
客户电话 / 拜访沟通 → 普通耳机或手机补记 → 客户摘要 → 异议/需求/预算/决策链 → 跟进任务 → CRM / 日历 / 话术同步
```

PMF 信号：销售愿意付费 ¥200+/月，或团队愿意按席位采购。

### 选项 B：现场售后/实施的"语音工单与知识库助手"

```text
现场沟通 / 故障描述 / 客户反馈 → 普通耳机或手机记录 → 问题归类 → 工单字段 → 解决方案建议 → 知识库沉淀 → 回访任务
```

PMF 信号：团队能明显减少漏单、返工、回访遗漏和知识库维护成本。

### 选项 C：产品/创业者的"访谈与假设工作流"

```text
用户访谈 / 产品讨论 / 创业想法 → Skill 整理 → 需求/假设/风险/验证问题 → A2UI 卡片确认 → 项目资产库
```

PMF 信号：用户主动把访谈、讨论和灵感持续沉淀到系统里。

### 选项 D：知识工作者的"普通耳机语音收件箱"

```text
随口记录 → 场景识别 → 任务/想法/问题/资料请求分类 → 轻执行 → 长期资产沉淀
```

PMF 信号：用户每天 3 次以上，离不开。

### 不建议优先选择：通用会议纪要

```text
开会 → 录音 → 纪要 → 自动生成待办
```

该方向已被 Plaud、飞书、viaim、腾讯会议、钉钉等验证并快速卷入红海，除非绑定更深的垂直 Skill 和业务工具，否则不应作为第一 PMF。

更进一步说，转写、翻译、速记、会议纪要都更接近基础语音处理能力或卫生能力，而不是这个产品最核心的价值定义。

原因在于：

- 厂商如果深度适配自家耳机与 App，天然更容易在录音链路、转写速度和翻译体验上做专项优化
- 这些能力容易同质化，也容易被用户理解为一次性工具，而不是长期平台
- 它们能提升效率，但不足以单独支撑长期护城河

因此，我们更合理的产品方向不是去做"更强的会议纪要耳机"或"更快的翻译耳机"，而是：

- 让不同耳机和不同音频设备都能接入同一套语音工作流
- 让一次输入跨手机、桌面和业务系统持续同步
- 让每次语音交互沉淀为客户、项目、需求、假设、任务、复盘等长期资产
- 让 Skill、A2UI 和 Agent 编排把语音从记录层推进到行动层

## 早期需求验证与功能开发策略

### 不建议先做功能完备产品

对于第一阶段，更合理的策略不是先做一个大而全的平台，再等待市场来验证，而是：

> **先做一个创始人自己能高频使用、又能拿去做真实演示和白手套试用的窄 Alpha。**

原因是：

- 早期最大风险不是功能不够多，而是做错方向
- 用户首先判断的是“是否解决一个具体问题”，不是“是否足够完备”
- 在没有持续反馈回路前，功能越多，偏航成本越高

### 第一版应是什么

第一版更适合定义为：

> **一个围绕单一高价值场景的桌面端 Alpha 工作台，而不是面向所有普通用户的通用平台。**

它应满足：

- 创始人自己可以高频使用
- 可以处理真实输入，而不是只做静态 Demo
- 可以拿去给目标用户现场试用
- 可以在真实试用中快速迭代

### 第一版最小闭环

第一版只需要优先跑通：

```text
输入进入
→ Skill 结构化处理
→ A2UI 卡片确认
→ 导出 / 同步到真实工作流
```

典型最小能力包括：

- 实时记录、导入录音或粘贴文本中的 1-2 种输入方式
- 提取需求、风险、承诺、下一步、假设等高价值结果
- 通过 A2UI 做确认、修改、驳回、延后
- 输出到文档、待办、飞书、CRM 或项目记录中的至少一个真实出口

### 早期验证方式

最适合个人创业者的方式不是大规模公测，而是：

1. **创始人自用**
   - 先把产品用于自己的访谈、讨论、合作沟通、需求梳理

2. **设计伙伴验证**
   - 找到 3-5 个愿意持续交流、反复试用的种子用户

3. **白手套式试用**
   - 不是把链接发给对方让其自己探索，而是用真实材料现场跑一遍结果

4. **半人工补齐**
   - 在产品还不完整时，允许通过人工微调、手工补录、手工导出先交付结果

### 优先寻找什么样的验证对象

如果当前难以直接触达最终企业客户，更实际的方式是：

- 先找自己当前能够持续接触的人群
- 优先找问题结构相似、但更容易反复验证的近似用户
- 用自己的创业产品访谈、产品讨论和合作沟通作为第一批真实场景

这一步不是为了验证最终付费规模，而是为了验证：

- 结果是否真的值钱
- 哪个场景最值得继续做深
- 用户是否愿意把真实材料继续交给系统

### 早期最该看的强信号

早期不应优先看注册数、页面停留等泛指标，而更应看：

- 是否愿意提供真实录音、聊天、会议或访谈内容
- 是否在第一次之后愿意第二次继续使用
- A2UI 卡片的确认率、修改率和驳回率
- 结果是否真的被复制进真实工作流
- 是否主动要求导出、同步、分享、连接现有工具

### 需要警惕的弱信号

以下反馈很容易产生误导：

- “界面挺好看”
- “这个很酷”
- “以后也许会用”
- “功能挺多”

相比之下，更有价值的反馈通常是：

- “这个字段不对，我真正需要的是另一个结果”
- “这个动作如果能直接进入飞书/CRM 我就会持续用”
- “这一步太麻烦，所以我没有继续”
- “这个卡片我已经拿去做下一步了”

### 第二版如何收敛

第二版不应简单理解为“加更多功能”，而应理解为：

- 从多种可能场景中收敛出一个最强主场景
- 从很多输出里收敛出最值钱的 2-3 类卡片
- 从通用工作台收敛出更明确的默认模式
- 从 Alpha 试用收敛到更稳定的真实工作流出口

也就是说：

> **第一版负责证明闭环可以成立，第二版负责证明哪个场景最值得做深。**

## 现实建议

1. **先做普通耳机 AI 增强**：用现有耳机 + 手机/桌面 App 验证入口故事，不要先做硬件
2. **先切一个垂类 PMF**：优先销售/商务或现场售后/实施，不要泛做所有 Agent
3. **不要绑死 OpenClaw**：OpenClaw 适合极客 + BYO Key + 高级版，不适合默认引擎
4. **把 Agent 放在"最后一公里"**：前端的场景识别、Skill、A2UI、确认和同步要先扎实
5. **把数据资产当护城河**：客户、项目、需求、假设、工单、任务和复盘要持续沉淀，不做一次性总结工具
6. **Agent 市场不能太早开**：先做官方 Skill，再做用户自定义，最后开放市场
7. **设计成本可控的商业模式**：免费/Pro/团队/Agent 市场/BYO 五层

---

## 一句话

**AI 大概率会达到这个标准，问题不是"能不能做出来"，而是"你能不能在大厂统一普通耳机 + 手机 AI 入口之前，占据一个足够深的垂直语音工作流场景"。当下该做的不是硬件，而是先用普通耳机 AI 增强系统，跑通 Skill + A2UI + 数据资产的 PMF 子集。**
