# 01 · 架构设计

## 架构目标

本架构设计要同时满足两件事：

- **[当下可落地]**
  - 支撑 2026 年桌面端优先的垂类 MVP，快速跑通真实用户工作流。

- **[未来可扩展]**
  - 为更多输入来源、更多 Skill、更多连接器、团队版、企业版、Agent Slate、企业中枢工作站和平台化预留空间。

因此，本系统不适合一开始采用复杂微服务，而更适合：

> **DDD + 模块化单体 + 六边形架构 + 事件驱动集成**

---

## 核心架构结论

### 1. 推荐架构形态

推荐采用：

```text
Modular Monolith
+ Domain-Driven Design
+ Ports & Adapters
+ Domain Events
```

对应含义是：

- **模块化单体**
  - 先保持单部署单元，避免过早引入分布式复杂度。

- **领域驱动设计**
  - 按业务能力划分模块边界，而不是按页面或技术栈切分。

- **六边形架构**
  - 核心领域不直接依赖具体模型厂商、具体数据库或具体外部工具。

- **事件驱动集成**
  - 模块之间通过领域事件协作，降低耦合，为未来拆分服务预留空间。

### 2. 为什么不是微服务优先

现在不建议一开始就拆成微服务，原因包括：

- **[原因一] 领域边界仍在收敛**
  - 产品形态和 Skill 范围还在演进，过早物理拆分会固化错误边界。

- **[原因二] 当前核心矛盾不是吞吐量**
  - 现阶段最重要的是抽象正确、迭代够快，而不是分布式扩容。

- **[原因三] 基础设施成本过高**
  - 服务治理、链路追踪、分布式事务、幂等、补偿、部署编排都会过早压垮研发节奏。

正确策略应是：

> **先逻辑解耦，再物理解耦。**

当某些模块的边界、负载和团队职责成熟后，再独立拆分为服务。

## 总体系统视图

```text
桌面端客户端
↓
应用网关 / API / BFF
↓
模块化领域核心
  - Input
  - Capture
  - Understanding
  - Skill
  - Workflow
  - A2UI
  - Asset
  - Connector
  - Identity & Policy
↓
基础设施适配层
  - ASR / Realtime 模型
  - LLM Provider
  - OpenClaw / QClaw / Agent 后端
  - CRM / 日历 / 邮件 / Notion / 飞书
  - 数据库 / 对象存储 / 队列 / 检索系统
```

系统本质上是：

```text
多输入来源
→ 结构化理解
→ Skill / Workflow 处理
→ A2UI 确认
→ 资产沉淀
→ 外部工具同步
```

而不是：

```text
用户说一句
→ 通用 Agent 自主处理所有事情
```

## 分层设计

建议按四层来理解系统。

### 1. Interface Layer

面向用户或外部系统的交互层。

包括：

- 桌面端 UI
- 内部 API / BFF
- Webhook / 导入接口
- IM / QClaw / 文件导入入口

主要职责：

- 接收用户操作
- 展示 A2UI、工作流和资产
- 提供统一的前端调用入口

### 2. Application Layer

面向用例和流程编排。

包括：

- 创建输入记录
- 启动转写
- 触发场景识别
- 执行 Skill
- 生成 A2UI 卡片
- 处理用户确认
- 触发同步动作

主要职责：

- 管理事务边界
- 组装多个领域对象完成一次业务用例
- 发布领域事件

### 3. Domain Layer

系统核心业务层，也是最稳定、最重要的一层。

包括：

- 实体、值对象、聚合根
- 领域服务
- 领域规则
- 领域事件

主要职责：

- 定义业务边界
- 保证状态一致性
- 承载真正的产品逻辑

### 4. Infrastructure Layer

外部依赖实现层。

包括：

- PostgreSQL
- 对象存储
- 向量检索
- LLM / ASR 供应商
- CRM / 日历 / 邮件 / Notion / 飞书
- 队列、缓存、日志、监控

主要职责：

- 实现领域层定义的接口
- 隔离具体技术供应商

## 领域划分

以下边界上下文是当前最适合这款产品的划分方式。

### 1. Input Context

负责统一管理输入来源和输入记录。

- **[职责]**
  - 输入设备与来源抽象
  - 文件 / 文本 / IM 导入
  - 设备权限状态
  - 输入元数据维护

- **[核心对象]**
  - `InputSource`
  - `InputRecord`

- **[典型事件]**
  - `InputRecordCreated`
  - `InputSourceConnected`

### 2. Capture Context

负责把输入变成可处理的音频或文本素材。

- **[职责]**
  - 录音会话
  - 批量导入处理
  - 流式转写
  - 分段、说话人标记、时间轴

- **[核心对象]**
  - `CaptureSession`
  - `Transcript`
  - `TranscriptSegment`

- **[典型事件]**
  - `CaptureStarted`
  - `TranscriptCompleted`

### 3. Understanding Context

负责场景识别与结构化理解。

- **[职责]**
  - 场景识别
  - 任务、要点、实体抽取
  - 风险识别
  - Skill 推荐

- **[核心对象]**
  - `ScenarioMatch`
  - `UnderstandingResult`

- **[典型事件]**
  - `ScenarioRecognized`
  - `SkillRecommended`

### 4. Skill Context

负责官方 Skill、企业 Skill 和未来第三方 Skill 的定义与执行。

- **[职责]**
  - Skill 定义
  - 输入输出 Schema
  - Prompt / Rule 组织
  - 版本管理
  - 灰度与测试

- **[核心对象]**
  - `SkillDefinition`
  - `SkillVersion`
  - `SkillRun`

- **[设计要求]**
  - Skill 不能只是 prompt 文本。
  - Skill 应包含输入 Schema、输出 Schema、A2UI 映射和可执行动作声明。

### 5. Workflow Context

负责工作流编排与执行状态。

- **[职责]**
  - Workflow 状态机
  - 步骤编排
  - 重试与补偿
  - 成本与预算控制
  - 高风险动作前置校验

- **[核心对象]**
  - `WorkflowRun`
  - `TaskExecution`
  - `ExecutionPolicy`

- **[典型事件]**
  - `WorkflowStarted`
  - `StepCompleted`
  - `ExecutionRequested`
  - `WorkflowFinished`

### 6. A2UI Context

这是系统最关键的差异化域之一。

- **[职责]**
  - 将 AI 输出映射成卡片、表单、时间线和任务队列
  - 接收用户确认、编辑、拒绝、延期等动作
  - 控制高风险动作的执行边界

- **[核心对象]**
  - `A2UICard`
  - `ConfirmationDecision`
  - `UIActionProposal`

- **[关键判断]**
  - A2UI 不是普通前端展示层，而是 AI 结果进入真实业务动作前的确认协议层。

### 7. Asset Context

负责长期资产沉淀与关联。

- **[职责]**
  - 资产创建、更新、去重、合并、版本化
  - 资产关系和引用
  - 检索和沉淀历史

- **[核心对象]**
  - `Asset`
  - `AssetCollection`
  - `AssetLink`

- **[资产类型示例]**
  - 任务
  - 日程
  - 客户记录
  - 访谈洞察
  - 需求
  - 工单
  - 项目纪要
  - 创业假设
  - 知识条目

### 8. Connector Context

负责所有外部工具与执行系统接入。

- **[职责]**
  - 授权管理
  - 同步规则
  - 任务分发
  - 失败重试
  - 执行日志

- **[核心对象]**
  - `Connector`
  - `SyncJob`
  - `ConnectorCredential`

- **[外部目标示例]**
  - 日历
  - 邮件
  - CRM
  - Notion
  - 飞书
  - QClaw
  - OpenClaw
  - 企业内部系统

### 9. Identity & Policy Context

负责用户、组织、权限与合规规则。

- **[职责]**
  - 用户与工作区
  - 团队与组织权限
  - 操作审计
  - 录音提示策略
  - 模型使用策略
  - 工具执行限制

- **[核心对象]**
  - `User`
  - `Workspace`
  - `Policy`
  - `AuditLog`

## 模块通信方式

### 1. 首选模式：领域事件

模块之间优先通过事件协作，而不是直接互调复杂内部逻辑。

例如：

```text
InputRecordCreated
→ Capture Context 启动转写

TranscriptCompleted
→ Understanding Context 进行场景识别

ScenarioRecognized
→ Skill Context 匹配或执行 Skill

SkillRunCompleted
→ A2UI Context 生成确认卡片

CardConfirmed
→ Asset Context 写入资产
→ Connector Context 发起同步
```

### 2. 为什么适合本产品

- **[优点一] 解耦强**
  - 输入、理解、Skill、A2UI、资产和连接器之间不会形成强耦合调用链。

- **[优点二] 易插入新能力**
  - 后续加入 IM 导入、QClaw、Agent Slate、本地模型节点时更容易扩展。

- **[优点三] 方便未来拆服务**
  - 当模块边界成熟时，可以把事件流平滑迁移到消息总线。

## 核心业务流

### 1. 标准处理流

```text
用户输入
→ 创建 InputRecord
→ 创建 CaptureSession
→ 生成 Transcript
→ 场景识别
→ 匹配 Skill
→ 执行 Workflow
→ 生成 A2UI 卡片
→ 用户确认
→ 写入 Asset
→ 触发 Connector 同步
```

### 2. 实时处理流

```text
实时音频输入
→ 流式转写
→ 增量场景判断
→ 候选任务和要点生成
→ A2UI 预览
→ 用户确认后再执行后续动作
```

### 3. 高风险执行流

```text
A2UI 卡片生成动作候选
→ Policy 检查
→ 用户确认
→ Connector 执行
→ 审计日志记录
→ 回写执行状态
```

## 后端结构建议

建议的代码组织形态：

```text
src
 ├─ modules
 │   ├─ input
 │   ├─ capture
 │   ├─ understanding
 │   ├─ skill
 │   ├─ workflow
 │   ├─ a2ui
 │   ├─ asset
 │   ├─ connector
 │   └─ identity
 ├─ shared
 │   ├─ kernel
 │   ├─ events
 │   ├─ primitives
 │   └─ result
 └─ apps
     ├─ api
     ├─ worker
     └─ desktop-gateway
```

每个模块内部建议再分为：

```text
module
 ├─ domain
 ├─ application
 ├─ infrastructure
 └─ contracts
```

### 各子目录职责

- **`domain`**
  - 实体、值对象、聚合根、领域规则、领域事件。

- **`application`**
  - 用例、命令、查询、事务边界和事件发布。

- **`infrastructure`**
  - ORM、外部 API、模型适配器、消息队列、文件存储等实现。

- **`contracts`**
  - 模块公开 DTO、接口协议、事件契约、Schema 定义。

## 端口与适配器设计

所有外部能力都建议通过端口抽象：

- **[AI 相关端口]**
  - `TranscriptionProvider`
  - `RealtimeSpeechProvider`
  - `LLMProvider`
  - `EmbeddingProvider`

- **[执行相关端口]**
  - `AgentExecutor`
  - `WorkflowDispatcher`
  - `ToolInvoker`

- **[外部系统端口]**
  - `CRMConnector`
  - `CalendarConnector`
  - `KnowledgeBaseConnector`
  - `IMBridgeConnector`

- **[基础设施端口]**
  - `ObjectStorage`
  - `VectorStore`
  - `EventBus`
  - `JobQueue`

这样做的核心价值是：

- 更换模型供应商不影响核心领域
- 更换 QClaw / OpenClaw / 其他 Agent 后端不影响业务模型
- 企业私有部署可以替换基础设施实现

## 部署建议

### 1. MVP 阶段

建议：

```text
桌面端客户端
+ 单体后端 API
+ 异步 Worker
+ PostgreSQL
+ 对象存储
+ 可选 Redis / Queue
```

特点：

- 快速落地

- 结构清晰

可优先拆出的模块：

- 转写 / 音频处理服务

### 3. 企业版阶段

增加：

- 企业私有部署模式
- 本地模型或混合推理
- 企业中枢工作站节点
- 审计和合规增强模块

## 架构边界

### 我们要明确做什么

- **[做]**
  - 接入系统已经暴露的输入源或导入内容
  - 做语音后的理解、确认、执行与沉淀
  - 做桌面端工作台和 A2UI 边界
  - 做 Skill / Workflow / Asset / Connector 的组合能力

### 我们明确不做什么

- **[不做]**
  - 直接控制耳机底层协议
  - 依赖耳机私有按键或私有传感器
  - 把系统做成纯聊天机器人
  - 让 Agent 在无确认边界下大量自治执行
  - 一开始做泛开发者平台

## 最终架构定义

**随身Agent 的最合适架构，是一套以桌面端为主工作台、采用 DDD 驱动的模块化单体架构；系统以 Input、Capture、Understanding、Skill、Workflow、A2UI、Asset、Connector、Identity & Policy 等边界上下文组织核心业务，通过六边形架构隔离模型、设备与外部工具，并通过领域事件串联处理流程，从而在当前阶段保持高迭代效率，同时为未来团队版、企业版、Agent Slate 和平台化预留足够扩展空间。**
