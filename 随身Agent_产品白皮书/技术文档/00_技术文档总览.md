# 00 · 技术文档总览

> 本目录用于从工程实现角度整理“随身Agent”系统的技术结构、模块边界、数据流、集成方式与演进路线。

---

## 一句话定义

**随身Agent 的技术系统是一套以桌面端为主工作台、以多输入设备为入口、以实时语音理解与 Skill / Workflow 为处理核心、以 A2UI 为确认边界、以长期资产沉淀为护城河的 AI 语音工作流架构。**

它不是：

- AI 耳机固件系统
- 蓝牙协议控制系统
- 单一模型绑定产品
- 纯聊天式 Agent 应用
- 只有“录音转写”的工具软件

它本质上是：

> **一个围绕语音输入、结构化理解、确认执行与资产沉淀构建的桌面端工作流系统。**

---

## 技术目标

- **[目标一] 支撑桌面端优先的 MVP**
  - 先跑通“输入 → 转写 → 场景识别 → Skill → A2UI → 资产沉淀”的闭环。

- **[目标二] 为多输入设备预留扩展能力**
  - 普通耳机只是最低门槛入口，后续可以逐步接入手机麦克风、电脑麦克风、录音豆、会议麦、IM / QClaw 等入口。

- **[目标三] 为平台化保留边界**
  - 早期不做开放平台，但技术上要为 Skill、Connector、企业私有 Agent 和 A2UI 模板预留演进空间。

- **[目标四] 降低外部依赖绑定风险**
  - 不绑死某个实时模型、某个 Agent 后端或某个工具生态。

- **[目标五] 支撑企业化与合规化**
  - 从一开始就考虑权限、审计、录音提示、数据保留和高风险操作确认。

---

## 技术原则

- **[原则一] DDD 驱动**
  - 先定义领域边界、聚合根、领域事件和核心对象，再落数据库和接口。

- **[原则二] 模块化单体优先**
  - 早期优先采用 `Modular Monolith`，在一个部署单元内保持清晰模块边界，避免过早微服务化。

- **[原则三] 六边形架构**
  - 核心领域不直接依赖具体 ASR、LLM、CRM、QClaw、OpenClaw 或数据库实现。

- **[原则四] 事件驱动协作**
  - 模块之间优先通过领域事件解耦，例如“转写完成”“场景识别完成”“卡片确认完成”。

- **[原则五] A2UI 是系统边界**
  - AI 的结果不能直接等于执行动作，中间必须有可确认、可编辑、可执行的结构化确认层。

- **[原则六] 桌面端是主工作台**
  - 手机、IM、QClaw 主要作为轻入口、远程桥接和消息触达端。

- **[原则七] 云端优先，端侧预留**
  - MVP 以云端处理为主，后续可逐步引入端侧 ASR、本地缓存、企业私有部署和 Agent Slate / 企业中枢工作站能力。

## 核心模块总览

- **[Input 模块]**
  - 管理输入源、导入方式、权限状态和输入元数据。

- **[Capture 模块]**
  - 负责录音会话、流式音频、离线文件、转写和片段化处理。

- **[Understanding 模块]**
  - 负责场景识别、任务抽取、要点整理、风险识别和 Skill 推荐。

- **[Skill 模块]**
  - 负责官方 Skill、企业私有 Skill、输入输出 Schema、版本管理和执行规则。

- **[Workflow 模块]**
  - 负责工作流编排、步骤状态、重试、补偿、预算与工具调用边界。

- **[A2UI 模块]**
  - 负责将 AI 结果映射为卡片、表单、时间线和确认界面。

- **[Asset 模块]**
  - 负责客户、项目、任务、日程、访谈、需求、假设、知识等长期资产沉淀。

- **[Connector 模块]**
  - 负责 CRM、日历、邮件、Notion、飞书、QClaw、OpenClaw 等外部系统集成。

- **[Identity & Policy 模块]**
  - 负责用户、团队、组织、权限、审计、策略、录音提示和执行限制。

## 核心对象总览

技术实现上建议围绕以下对象建模：

- **`InputSource`**
  - 输入来源，例如普通耳机、系统麦克风、文件导入、IM 转发。

- **`InputRecord`**
  - 一次进入系统的原始输入记录。

- **`CaptureSession`**
  - 一次实时录音或批处理会话。

- **`Transcript` / `TranscriptSegment`**
  - 转写结果及时间轴片段。

- **`ScenarioMatch`**
  - 场景识别结果和推荐 Skill 候选。

- **`SkillDefinition` / `SkillRun`**
  - Skill 模板定义与一次执行记录。

- **`WorkflowRun`**
  - 一次完整工作流运行实例。

- **`A2UICard` / `ConfirmationDecision`**
  - 可确认界面与用户确认动作。

- **`Asset`**
  - 沉淀后的长期资产对象。

- **`Connector` / `SyncJob`**
  - 外部工具连接器与同步任务。

## MVP 技术边界

第一阶段建议明确只做以下内容：

- **[做]**
  - 桌面端主工作台
  - 多输入导入
  - 实时或批量转写
  - 1-3 个高价值官方 Skill
  - A2UI 卡片确认
  - 基础资产沉淀
  - 轻量工具同步

- **[暂不做]**
  - 完整移动端产品本体
  - 复杂多 Agent 协作
  - 大规模第三方 Skill 市场
  - 自研硬件依赖
  - 绑定单一 Agent 执行后端
  - 全自动后台自主执行系统

---

## 技术演进方向

- **[阶段一] 桌面端 MVP**
  - 跑通单用户、单工作台、官方 Skill 的完整闭环。

- **[阶段二] 多输入与资产增强**
  - 扩展更多输入来源、更多 Skill、更多资产模型和 IM / QClaw 桥接。

- **[阶段三] 团队与企业能力**
  - 增加权限、共享资产、审计、企业私有 Skill 和企业 Agent Hub 软件版。

- **[阶段四] 硬件与边缘节点**
  - 增加 Agent Slate、收音豆、企业中枢工作站、本地模型节点。

- **[阶段五] 平台与生态**
  - 开放 Skill、A2UI 模板、Connector 能力和企业 API。

---

## 一句话

**技术上，我们不是在做一个“耳机控制产品”，而是在做一个以桌面端为中心、以多输入为入口、以 Skill / Workflow / A2UI 为骨架、以资产沉淀为终点的可演进 AI 语音工作流系统。**
