## 项目群不是多个并列创业项目

Agentic Web 项目群的项目名称很多，但它们分属不同层级：

```text
产品入口：
  Vlawd
  IDE 插件
  Web 论坛
  二维画布

垂直项目：
  Module Spec
  高校知识项目
  未来服务项目

共同对象：
  Session
  Evidence
  Capsule
  Living Project

网络基础：
  Registry
  Post is Project
```

近期只允许维护一个核心代码主线和一个受控实验。其他名称是架构位置，不代表需要立即创建新仓库。

## Vlawd（PORT.VLAWD）

### 定位

> **Agentic Web 的全双工个人终端和本地 Agent Runtime。**

当前兼容版本：Vlawd v2.12。产品定位、交互、架构、MVP 和代码以 `AI_cursor_v2/` 为唯一权威来源。

### 核心职责

- 接收语音或文本目标；
- 感知用户当前环境；
- 在结构化接口不足时，通过状态作用域、语义优先的 Environment Interaction Adapter 执行；
- 可见地执行低风险动作；
- 显示多个任务的状态与确认点；
- 支持暂停、取消、接管和恢复；
- 记录 Session 与 Execution Report；
- 调用本地或远端 Capsule。

### 不负责

- 维护开放社区；
- 直接定义全网信誉；
- 承担所有模型推理；
- 保存平台全部公开对象；
- 自行发明 Agent 网络传输协议；
- 把外部 Computer Use 项目变成第二套 Runtime 或安全边界。

本项目群文档只定义 Vlawd 与其他模块的接口边界，不重新定义其 Provider、执行状态机、UI 或 Session 内部结构。

## Generative Module Spec（PORT.MSPEC）

### 定位

> **供 Coding Agent 针对具体项目上下文生成模块的、持续演化且可验证的方法规范。**

它位于普通文档、模板和代码库之间：

```text
Library：
直接复用实现

Template：
复制固定结构

Documentation：
解释如何实现

Generative Module Spec：
保存设计判断、上下文问题、约束、变体、失败经验和验收标准
```

### 核心职责

- 验证 Living Artifact 是否比 Prompt 更稳定；
- 用自动测试产生客观 Evidence；
- 用干净隔离环境验证构建、启动和运行边界；
- 试验 Version、Patch、Lineage 和 Profile；
- 为 Capsule Protocol 提供第一个机器可验证案例。

### 推荐 Profile

- Prototype；
- Internal；
- Production；
- High Assurance。

Profile 必须绑定 Conformance Suite 和 Evidence，不使用主观完成百分比。

## 可执行项目验证能力（ARCH.SANDBOX）

### 定位

> **为 Module Spec、Capsule 和 Post is Project 提供真实运行 Evidence 的共享 Runtime 能力。**

它不是新的创业项目，也不要求近期创建独立云平台：

```text
Project Revision
→ Execution Plan
→ 临时隔离环境
→ Agent 构建、启动和修复
→ 独立 Evaluator
→ Execution Report / Evidence
→ Project Patch 或 Module Spec Patch
```

职责边界：

- Module Spec 定义方法、Profile 和 Conformance Suite；
- Agent Runtime 规划、执行、暂停、恢复并申请确认；
- Environment Broker 创建、限制和销毁环境；
- Evaluator 独立判断结果；
- Post is Project 保存语义事件、报告和稳定发布；
- Capsule 在跨环境证据成立后保存可移植契约。

默认使用按需创建、到期销毁的环境，不为每个 Project 分配永久 VM。详细设计见 `共享架构与协议/03_可执行项目与部署验证.md`。

## 高校二维知识论坛（PORT.CAMPUS）

### 定位

> **班级或课程私有的全双工 AI 讨论与图谱式知识协作客户端。**

### 用户闭环

```text
学生提出问题
→ 与共享 AI 专家讨论
→ AI 引用课程知识
→ 对话转为结构化项目草稿
→ 用户审查后发布
→ 同学添加证据、方案或反例
→ 发起人接受、拒绝或保留分支
```

### 验证价值

- 全双工云端体验；
- 对话转项目；
- Graph / Canvas 可理解性；
- 人类贡献与项目演化；
- 私有空间、身份和权限；
- 共享推理服务的并发与成本。

### 不能替代的验证

- 本地蓝牙与音频路由；
- 本地桌面执行；
- 系统级抢占；
- 个人长期工作流。

## Capsule Protocol（PORT.CAPSULE）

### 定位

> **连接不同 Runtime、客户端和垂直项目的可执行能力对象规范。**

Capsule 接近：

```text
Skill
+ Workflow
+ Permission Manifest
+ Tests
+ Provenance
+ Runtime Metrics
```

它不是：

- 一段 Prompt；
- 一个不可检查的 Agent；
- 一个只适用于单机的脚本；
- 新的 HTTP / P2P 协议；
- 自动可信的第三方代码包。

### 启动条件

只有 Vlawd、Module Spec 或高校论坛中的至少两个场景需要交换同类对象时，才冻结 v0.1 Schema。

---

## Capsule Registry / Network（PORT.REGISTRY）

### 定位

> **Capsule 和 Living Project 的发布、检索、版本、来源与运行反馈层。**

### 核心信号

- 使用次数；
- 任务成功率；
- 失败类型；
- 适用环境；
- 所需权限；
- 被复用和 Fork 的次数；
- 改进是否被合并；
- 运行 Evidence；
- 节省时间和成本。

点赞和关注可以存在，但不能替代运行信号。

---

## Post is Project（PORT.PIP）

### 定位

> **让网络内容从一次性帖子升级为可版本化、可分支、可验证、可治理的项目。**

一个 Project 可以代表：

- 一个问题；
- 一种解决方法；
- 一份 Module Spec；
- 一个研究结论；
- 一个 Capsule；
- 一项服务能力。

### 协作事件

```text
comment
evidence
counterexample
experiment
branch
patch
review
merge
request_validation
attach_execution_report
promote_release
rollback_release
supersede
deprecate
```

Post is Project 不直接持有组织级云管理员凭据，也不负责底层 VM 调度。它通过受控 Runtime 请求验证，并将运行结果投影到 Project。

## 服务与设备生态

### 商家与服务（PORT.SERVICE）

长期可将商家能力表示为 Service Capsule：

- 输入、输出和地区；
- 价格、时效与退款规则；
- 权限要求；
- 真实履约 Evidence；
- 版本和接口。

交易必须优先兼容结构化 Agent Commerce 标准，截图只作为预览、证据或人工接管，不作为主要协议。

### 模块化设备（PORT.DEVICE）

近期先支持普通蓝牙耳机和电脑麦克风。只有真实使用证明硬件入口高频且现有设备限制严重，才定义设备接口或自有硬件。

可选设备事件：

- 硬件暂停；
- 长按私人指令；
- 明确确认；
- 录音状态；
- 任务完成触觉反馈。

## 项目依赖关系

```text
Vlawd
  └── 产生 Session / Evidence

Module Spec
  └── 产生机器可验证的 Project / Evidence
      └── 可选使用 Sandbox 产生真实部署 Evidence

高校论坛
  └── 产生人类协作的 Project / Evidence

两个垂直案例
  └── 抽取 Capsule Protocol

Capsule Protocol
  └── 支撑 Registry

Registry + Project Event
  └── 支撑 Post is Project

Post is Project + Agent Runtime + Environment Broker
  └── 支撑可执行项目验证和稳定发布

稳定入口 + 可信网络
  └── 才允许服务与设备生态
```

## 近期仓库策略

```text
Vlawd 代码：
继续使用 AI_cursor_v2/app 作为基线

Agentic Web 项目群文档：
统一管理项目群决策

Module Spec：
先在文档或小型夹具中验证
基础 Conformance 成立后才测试临时 Sandbox
优先复用 Dockerfile、Compose、Buildpacks 和现有 CI
不立即创建大型应用或通用云平台

高校论坛：
达到 Phase 2 启动条件后再决定是否创建独立应用

Capsule：
先在两个垂直对象中发现共同字段，不先开独立协议仓库

可执行项目验证：
先使用一个 Runner、一个隔离后端和无生产 Secret 的低风险项目
不默认创建永久 VM，不自动部署到生产环境
```

## 一句话总结

**Vlawd、Module Spec 和高校论坛负责产生真实使用证据；Capsule、Registry 和 Post is Project 负责把不同场景中的经验抽象成可迁移和可协作的网络能力。**
