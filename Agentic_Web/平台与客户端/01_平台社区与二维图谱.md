## 核心判断

“不做传统 Feed”不等于没有网络、没有界面或必须完全本地化。

推荐架构：

```text
中央项目网络
├── 对象与版本存储
├── Graph 关系索引
├── Event Log
├── 搜索与检索
├── 身份、签名与权限
├── Evidence 与信誉
└── Capsule 数据

客户端
├── 本地缓存
├── 任务上下文检索
├── 发布前隐私预览
├── Project View
├── Forum View
├── Canvas View
└── Diff / Review View
```

同一份数据可以投影成不同界面，不能为论坛、画布和 Feed 各建一套事实源。

## Graph 数据模型

底层是有向属性图，不强制是单向树。

### 推荐节点

- Problem；
- Goal；
- Constraint；
- Hypothesis；
- Solution；
- Evidence；
- Counterexample；
- Experiment；
- Patch；
- Decision；
- Result；
- Capsule；
- Environment Profile；
- Execution Report；
- Release；
- Person / Agent；
- Source。

### 推荐边

- supports；
- contradicts；
- depends_on；
- refines；
- forks_from；
- tested_by；
- derived_from；
- supersedes；
- merges_into；
- produced_by；
- runs_in；
- validated_by；
- promotes；
- cites。

所有节点和边必须支持：

- stable id；
- version；
- author / actor；
- created_at；
- privacy_scope；
- provenance；
- status。

## Project View（UX.PROJECT）

项目详情是默认权威视图，至少显示：

- 问题和目标；
- 当前稳定结论；
- 主要分支；
- 支持与反驳 Evidence；
- 最新版本和维护状态；
- Source Revision 和 Execution Profile；
- 最近一次验证、运行或发布状态；
- 运行或复用情况；
- Execution Report、Evaluator 结果、成本和环境过期时间；
- 待解决问题；
- 权限与隐私；
- 贡献入口。

---

## Run / Validate / Deploy View（UX.RUN）

运行视图是同一 Project Graph 的执行投影，不是独立云控制台。

不是每个 Project 都需要服务器：

- 文档显示构建和链接检查；
- Library 显示包和兼容性测试；
- Web / API 显示临时 Preview；
- Workflow 显示隔离运行；
- Infrastructure 显示 Plan 和策略结果。

推荐状态：

```text
not_configured
→ planned
→ waiting_approval
→ provisioning
→ running
→ passed / failed / blocked
→ expired
→ promoted / rolled_back
```

视图至少显示：

- 当前 Project Revision；
- 使用的 Module Spec / Capsule 版本；
- Execution Plan 和 Profile；
- 所需权限、Secrets 引用、网络和资源预算；
- Sandbox 类型和到期时间；
- Build、Start、Health 和 Golden Test 结果；
- Execution Report、Evidence 和失败分类；
- 候选 Project Patch；
- Preview 提升为 Stable Release 的确认入口。

边界：

- 平台通过受控 Agent Runtime 请求执行，不直接持有组织级云凭据；
- 原始日志、Secret 和私有配置不自动公开到 Graph；
- 公网入口、付费资源、生产数据、数据库迁移、DNS 和长期环境必须显式确认；
- 环境创建和探针轮询留在 Runtime，只有语义里程碑写入 Project Event。

详细设计见 `共享架构与协议/03_可执行项目与部署验证.md`。

## Forum View（UX.FORUM）

论坛视图按人类容易阅读的时间和主题顺序显示 Project Event：

```text
用户提出问题
AI 生成结构化草稿
同学添加证据
另一位同学提出反例
维护者创建分支
实验结果更新
分支合并为新版本
```

论坛中的自由文本不直接改变稳定对象。它可以：

- 保持为评论；
- 被 AI 转成候选节点；
- 转成 Patch；
- 转成实验建议；
- 被维护者拒绝。

## Canvas View（UX.CANVAS）

二维画布是 Graph Renderer，不是独立平台。

### 交互能力

- 选择中心节点；
- 局部子图加载；
- Cluster 折叠；
- Semantic Zoom；
- 按节点类型过滤；
- 按状态、可信度和环境过滤；
- 展示冲突和依赖；
- 在 Canvas、列表、详情和 Diff 间切换。

### 布局

径向树可以作为问题探索视图：

```text
中心问题
→ 约束
→ 方案
→ 证据
→ 结果
```

但数据模型不能限制为树，因为一个 Evidence 可以支持多个方案，一个方案也可能依赖多个项目。

### 规模控制

- 默认只加载当前节点附近子图；
- 远端节点折叠为 Cluster；
- 使用摘要节点降低视觉密度；
- 不同时展示全部历史事件；
- 复杂关系优先在列表和 Diff 中审查。

## Explore、Search 与 Feed

### Explore / Topic / Search（UX.EXPLORE）

发现应优先依据：

- 任务目标匹配；
- 适用环境；
- 权限风险；
- 真实成功率；
- Evidence 新鲜度；
- 维护状态；
- 与当前 Project 的依赖关系。

### Activity / Inbox（UX.INBOX）

用户更需要知道：

- 哪个任务等待确认；
- 哪个 Project 收到 Patch；
- 哪个依赖发布了新版本；
- 哪个 Capsule 在当前环境失败；
- 哪个 Project 的验证环境失败、过期或等待批准；
- 哪个分支等待合并。

### 传统 Feed（UX.FEED）

Feed 可以作为派生视图展示订阅对象的更新，但不能成为内容和关系的底层模型，不能用停留时长取代任务成功和贡献质量。

## 高校论坛最小产品

```text
一个班级
+ 一个课程主题
+ 一个共享 AI 专家
+ 一个私有项目空间
+ 一种 Graph Schema
+ 一项他人贡献
```

### MVP 页面

1. AI 专家私聊；
2. 对话转图谱草稿；
3. 草稿审查；
4. 班级项目列表；
5. Project / Forum / Canvas 三视图；
6. 添加证据、方案或评论；
7. 接受、拒绝或保留分支。

### 隐私

- 原始录音和 Transcript 默认私密；
- 用户明确确认后才发布结构化摘要；
- 发布前显示将公开的节点和来源；
- 班级空间与公开空间严格分离；
- AI 不把私有讨论自动加入公共知识库。

### AI 权威边界

AI 回答必须区分：

- 已验证资料；
- 当前社区共识；
- 有争议分支；
- AI 推断；
- 尚未验证假设。

## 云端推理边界

第一版不建设通用共享 GPU Pool，只需要：

```text
一个模型
+ 一个 GPU 推理服务
+ 一个 Voice Gateway
+ Session 隔离
+ 并发上限和排队
+ 用户额度
+ 延迟、成本和失败监控
```

真实并发超过单 Worker 后，才增加调度、多 Worker、容错和弹性。

---

## 一句话总结

**论坛、画布、运行视图、项目页和 Feed 都是同一 Living Project Graph 的不同投影；平台的价值来自结构化贡献、真实运行 Evidence 和版本演化，而不是重新创造一个内容流或云控制台。**
