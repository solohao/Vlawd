## 核心原则

```text
先验证真实任务，不先建设平台。
先使用现有协议，不先发明网络。
先产生两个垂直案例，再抽取共同 Capsule Schema。
先修最大阻塞，不追求界面和模型的局部最优。
每个阶段必须有进入条件、退出证据和 Not Now 清单。
```

---

## 路径总览

Agentic Web 项目群存在三条垂直验证路径，但不是三条同时开发的主线。

| 路径 | 验证对象 | 优势 | 主要风险 |
|------|---------|------|---------|
| Vlawd Dogfood | 本地 Runtime、全双工、Session、可见执行 | 符合创始人真实需求和热爱 | 技术集成复杂 |
| Module Spec | Living Project、Evidence、Patch、自动回放 | 判断标准客观、验证速度快 | 容易变成新项目并偏离 Vlawd |
| 高校论坛 | 人类协作、画布、AI 专家、云端全双工 | 有真实封闭用户和外部证据 | 社区运营与隐私成本 |

当前执行顺序：

```text
Vlawd 自我 Dogfood
→ Module Spec 小型方法实验
→ 高校私有论坛试点
→ 从至少两个垂直案例抽取 Capsule v0.1
→ 最小 Registry
→ Post is Project 平台
→ 服务、商业与设备生态
```

Module Spec 与高校论坛可以在后续交换顺序，但任何时候同时进行中的核心功能只能有一个。

---

## Phase 0：范围冻结与代码基线

### 目标

建立 Agentic Web 项目群文档基线，不重写 V2 工程。

### 交付

- Agentic Web 项目群文档；
- 单一 Golden Path；
- Not Now 清单；
- V2 代码安全缺陷和真实模型缺口继续作为实现输入；
- 明确模型和成本 Baseline。

### 退出标准

- 所有当前想法能够归入项目、层级或 Idea Vault；
- 下一阶段只有一个验收场景；
- 不再因新想法新增并行项目。

---

## Phase 1：Vlawd 自我 Dogfood

> 目标：证明用户可以通过全双工语音监督一次真实研究任务，并在第二天继续使用其结果。

本阶段的产品设计、功能编号、实现批次和验收细节以 `AI_cursor_v2/02_MVP开发规划与阶段路线.md` 为准；下列内容只定义跨项目阶段门。

### Cycle 1：真实全双工入口

```text
用户与一个固定模型实时交流
→ 用户可以自然插话
→ AI 在目标延迟内停止
→ 正确接受新的约束
```

阶段边界：

- 一个模型；
- 一个音色；
- 现有耳机或电脑麦克风；
- 不建设 GPU 池；
- 不比较大量候选模型。

### Cycle 2：一个只读研究任务

```text
语音提出研究目标
→ 浏览器可见搜索与阅读
→ 用户途中纠正
→ 输出带来源的整理结果
```

阶段边界：

- 只开放低风险浏览器动作；
- 表单提交、支付、文件写入和系统级操作关闭；
- 先修安全策略优先级、目标隔离和确认状态机。

### Cycle 3：Session 延续

```text
保存目标、来源、动作、纠正、结论与未解决问题
→ 第二天重新打开
→ 从一个分支继续研究
```

### Phase 1 退出标准

- Golden Path 连续成功至少三次；
- 用户在真实工作中主动重复使用；
- 无未确认高风险动作；
- 打断、任务耗时、成本和纠正次数可记录；
- 三个周期各有一段 30–90 秒真实演示；
- UI 达到可理解和不易误操作，不要求精致。

## Phase 2：第二垂直案例

Phase 2 只选择一个垂直实验，完成后再考虑另一个。

### 路径 A：Module Spec Lab

```text
选择一个低风险模块
→ 建立 MODULE.md
→ 在三个项目上下文中生成
→ 运行 Conformance Tests
→ 根据失败更新 v2
→ 与无规范 Baseline 比较
```

退出证据：

- 首次通过率提高，或人工修正显著减少；
- 失败能够转化为新测试或规范 Patch；
- 新版本能改善多个项目，而不是只修一个特例。

### 路径 B：高校二维知识论坛

采用三个受控周期：

```text
Cycle C1：能自然讨论
  一个模型、一个音色、一个课程主题
  支持插话、暂停和引用小型知识库

Cycle C2：讨论变成项目
  生成 Problem、Constraint、Solution、Evidence、Open Question
  用户可以修改、删除和确认

Cycle C3：第二个人能够贡献
  发布到班级私有空间
  同学添加评论、证据或方案
  发起人接受、拒绝或保留分支
```

```text
一个班级 / 课程主题
→ 与共享全双工 AI 专家讨论
→ 对话转可编辑图谱草稿
→ 发布到私有班级空间
→ 第二个人贡献证据或方案
```

退出证据：

- 真实用户无需大量解释即可完成体验；
- 至少一个项目获得第二个人的有效贡献；
- 原始私聊默认私密；
- AI 回答能区分来源、共识、争议和推断。

## Phase 3：Capsule Protocol v0.1（VALID.CAPSULE）

> 只有至少两个垂直场景出现共同对象需求后才启动。

### 最小范围

```text
Artifact Manifest
Project Event
Execution Report
Permission Manifest
Lineage
Evaluator Reference
```

### 明确不做

- 新传输层；
- P2P 网络；
- 区块链和 Token；
- 通用 Agent 身份标准；
- 全网共识；
- 涌现式调度协议。

### 退出标准

- 同一 Capsule 可以从环境 A 导出并在环境 B 导入；
- 权限和前置条件可以预览；
- 结果可复现，或明确报告不兼容；
- Evidence 能关联到精确版本和环境；
- 两种不同客户端可以读取同一 Manifest。

---

## Phase 4：最小 Registry（VALID.REGISTRY）

### 目标

让 Agent 在任务发生时找到正确的 Capsule，而不是建设完整社区。

### 最小能力

- 发布签名对象；
- 按目标、环境、风险和版本查询；
- 下载指定版本；
- 提交执行成功或失败报告；
- 查看来源和版本历史；
- 私有、班级和公开三种可见范围中的至少两种。

### 退出标准

- Agent 能在真实任务中检索并使用一个远端对象；
- 远端对象执行前经过权限检查；
- 运行结果回传并改变对象的兼容性或可信信息；
- 不依赖传统 Feed 完成发现。

---

## Phase 5：Post is Project 平台（VALID.PIP）

### 目标

让项目不再是静态帖子，而是可持续演化的网络对象。

### 最小能力

- Problem / Method / Capsule 项目；
- typed comment；
- branch、patch、evidence、merge；
- 当前稳定结论；
- 项目成熟状态；
- 论坛、列表、项目和画布视图共享同一 Graph 数据。

### 退出标准

- 陌生贡献者能够理解项目并提交一次有效贡献；
- 贡献能通过自动验证或明确的人类审核进入新版本；
- 高风险修改不能只因文本可合并而自动发布；
- 画布在节点增多时仍能通过局部加载、折叠和过滤使用。

---

## Phase 6：生态扩展

只有真实使用证明入口、状态和网络三层已经成立后，才评估：

- 服务 Capsule 与商家能力；
- Agent Commerce 标准接入；
- 多 GPU Worker 和成本调度；
- 模块化输入设备协议；
- 自有硬件；
- 开放社区与推荐；
- 可选动态 Feed。

---

## 阶段升级决策

```text
Phase 1 未成立：
  不建设 Capsule 平台
  先缩小 Golden Path 或保留文本降级

Phase 2 只有一个垂直成立：
  继续积累，不抽象通用协议

Phase 3 无法跨两个环境运行：
  说明只是本地配置，不是 Capsule Protocol

Phase 4 检索没有真实使用：
  不建设社区和推荐

Phase 5 没有有效第三方贡献：
  不扩张为开放平台
```

---

## 当前 Not Now

- 传统社交 Feed；
- Agent 人设和自动点赞；
- 自建支付系统；
- 自研通用大模型；
- 多模型自由切换 UI；
- 自定义蓝牙协议栈；
- 一开始支持所有桌面软件；
- 为视觉完美重写前端；
- 在没有运行证据时设计完整 Capsule 标准；
- 同时推进 Vlawd、Module Spec 和高校论坛三个产品。

---

## 一句话总结

**Agentic Web 项目群的开发路线不是从 Runtime 一次跳到平台，而是依次证明：个人任务能完成、经验能积累、对象能移植、网络能发现、陌生人能协作，最后才允许生态扩展。**
