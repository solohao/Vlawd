## 核心体验

Agentic Web 项目群的第一体验不是“AI 自动完成一切”，而是：

> **AI 一边说、一边做；用户可以自然插话；AI 能立即停下、保留上下文，并按照纠正继续。**

```text
用户表达目标
→ 任务进入独立工作区
→ AI 说明当前计划并开始执行
→ 用户随时暂停、纠正或接管
→ 高风险动作等待显式确认
→ 任务完成并留下来源、动作和结果
→ 用户以后可以继续或复用
```

核心体验指标：

- 插话和暂停延迟；
- 任务完成率；
- 单位任务纠正次数；
- 高风险动作误执行次数；
- 失败恢复率；
- 从目标到有用结果的时间；
- 第二天继续任务的比例。

这些指标来自 Vlawd 当前产品验证，但全双工、Task Workspace、安全状态和界面实现以 `AI_cursor_v2/` 为权威。本模块只保留它们对整个项目群提出的体验约束。

## 多任务不是多鼠标

Agentic Web 项目群不把“多个光标同时移动”作为主要体验。更稳定的表现是 Task Workspace：

```text
任务 A：比较三款耳机
状态：等待用户选择

任务 B：整理课程讨论
状态：正在生成知识图谱

任务 C：研究全双工模型
状态：已暂停，可继续
```

每个任务拥有：

- 独立 Session；
- 当前状态和下一步；
- 当前工具或页面；
- Evidence；
- 暂停、取消、接管和回滚；
- 等待确认的动作；
- 成本和运行时间。

AI Cursor 只用于指示当前活跃执行位置，不承担全部任务管理。

Task Workspace 的卡片、状态和控制规则由 `AI_cursor_v2/产品文档/05_桌面端UI与运行时界面.md` 与 `AI_cursor_v2/技术文档/05_任务工作区与光标调度.md` 维护。

## 产品边界

### Agentic Web 项目群是

- 人与 Agent 的连续协作入口；
- 可监督、可打断的任务执行层；
- 个人任务状态和工作流记忆；
- 可验证能力与项目的网络；
- 可执行项目的隔离验证和环境兼容性记录；
- 本地和云端 Agent 的中立连接层；
- 多种客户端共享的项目对象模型。

### Agentic Web 项目群不是

- 通用操作系统；
- 普通聊天机器人；
- 纯语音输入法；
- 固定流程 RPA 的完全替代品；
- 无责任边界的自主 Agent 网络；
- 以社交热度为核心的平台；
- 立即覆盖所有设备、网站和商家；
- 为每个项目默认分配永久服务器的通用 PaaS；
- 依赖单一模型厂商的产品壳。

## 护城河结构

单个功能难以形成长期防御力。Agentic Web 项目群的护城河必须由入口、状态和网络组合形成。

### 高频入口（MOAT.ENTRY）

```text
现有耳机
+ 全双工交互
+ 桌面悬浮入口
+ 可见任务工作区
```

它负责让用户高频表达真实目标，但本身容易被复制。

### 个人状态（MOAT.STATE）

```text
历史 Session
+ 用户纠正和偏好
+ 信任过的 Capsule 与服务
+ 进行中的任务
+ 权限和风险偏好
+ 失败恢复经验
```

它让产品越用越懂用户，降低单位任务监督成本，并形成正常的迁移成本。用户必须能够导出数据，防御力来自持续价值而不是恶意锁定。

### 可验证网络（MOAT.NETWORK）

```text
真实任务
→ Execution Report
→ 在干净环境中复现
→ Capsule / Project
→ 复用、失败和改进
→ 更准确的适用范围和风险判断
→ 更高任务成功率
```

竞争者可以复制语音按钮和页面布局，但不能瞬间复制长期真实运行产生的 Evidence、跨环境兼容性矩阵、失败模式和可信贡献网络。

### 可靠执行（MOAT.TRUST）

信任来自长期工程，而不是模型承诺：

- 确定性安全规则；
- 最小权限；
- 显式确认；
- 低延迟抢占；
- 可见执行；
- 可恢复状态；
- 完整审计；
- 真实失败数据。

### 服务与设备生态（MOAT.ECOSYSTEM）

长期可能形成双边网络效应，但不属于早期壁垒：

```text
更多真实用户任务
→ 更多服务愿意提供结构化能力
→ Vlawd 可完成更多任务
→ 用户更依赖这一入口
```

它伴随商家拓展、支付、欺诈、售后和监管，不应提前建设。

## 防御力判断

| 能力 | 单独防御力 | 长期作用 |
|------|-----------:|---------|
| 全双工语音 | 低 | 高频入口 |
| 普通蓝牙耳机支持 | 低 | 使用便利 |
| AI Cursor 界面 | 中低 | 可见执行和品牌识别 |
| 低延迟安全抢占 | 中高 | 信任工程 |
| 个人 Session 和任务连续性 | 高 | 状态积累 |
| 私有真实任务 Eval | 高 | 持续优化 |
| 干净环境部署与回放 Evidence | 高 | 复现、兼容性和规范演化 |
| Capsule / Project Network | 很高 | 数据与网络效应 |
| 服务生态 | 高 | 双边供给 |
| 自研硬件 | 中 | 只有规模成立后才可能有效 |

## 耳机和设备路径

### 第一阶段

支持现有蓝牙耳机、USB 耳机和电脑麦克风。由 Vlawd Host 处理：

- 音频设备选择；
- VAD 和抢占；
- 隐私提示；
- 本地安全词；
- 云端模型连接。

### 第二阶段

真实使用出现稳定需求后，定义 Vlawd Device Interface，而不是直接制造硬件。

### 第三阶段

只有现有设备在延迟、隐私、佩戴或硬件确认方面持续阻碍核心体验时，才评估自有设备。

## 商家与服务边界

长期购买流程可以是：

```text
用户表达购买目标
→ Agent 检索结构化服务
→ 比较价格、时效、退款和权限
→ 展示商品与服务摘要
→ 用户修改条件
→ 显式最终确认
→ 安全支付令牌完成交易
→ 订单进入独立任务工作区
```

优先级：

1. 商家结构化 API 或 Agent Commerce 标准；
2. 经授权的浏览器 DOM；
3. 截图作为预览、Evidence 和人工接管。

可参考的行业协议：

- Agentic Commerce Protocol：https://github.com/agentic-commerce-protocol/agentic-commerce-protocol
- Google AP2：https://github.com/google-agentic-commerce/AP2

Agentic Web 项目群不自行保存原始支付凭据，不允许模型绕过最终确认。

---

## 最大竞争威胁

操作系统和浏览器厂商控制设备、账号、桌面和支付入口。Agentic Web 项目群不能只做“语音调用模型”，而应坚持：

- 跨平台；
- 本地优先和数据可导出；
- 模型中立；
- 可见、可停和可恢复；
- 开放 Capsule；
- 真实运行 Evidence；
- 服务一个巨头不愿深度优化的窄场景。

---

## 一句话总结

**全双工和耳机负责入口，可靠 Runtime 建立信任，个人 Session 形成持续状态，可验证 Capsule / Project 网络形成长期防御力；任何单独界面或模型都不是护城河。**
