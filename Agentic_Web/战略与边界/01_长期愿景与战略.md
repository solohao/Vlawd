## 类别定义（VISION.AGENTIC）

Agentic Web 项目群使用 **Agentic Web** 作为长期生态类别。

它指向一种正在形成的互联网形态：

```text
网络中的内容、软件和服务
不再只供人阅读、点击和复制
而是向 Agent 暴露可理解的能力、权限、输入输出和验证方式
Agent 可以代表用户发现、组合、执行和持续改进这些能力
```

Agentic Web 项目群对这一类别的独特限定是：

> **Agentic Web 必须可执行、可验证、可演化。**

- 可执行：能力有明确输入、输出、权限和运行入口；
- 可验证：结果关联环境、版本、Evidence 和 Evaluator；
- 可演化：对象支持版本、Patch、Fork、Merge 和 Lineage。

相关概念参考：

- Agentic Web 研究：https://arxiv.org/abs/2507.21206
- Agent Web 白皮书：https://github.com/openagentprotocol-OAP/oap-spec/blob/main/papers/agent-web-whitepaper.md
- Internet of Agents：https://arxiv.org/abs/2407.07061

这些概念仍在形成，并非已经存在唯一标准。Agentic Web 项目群使用行业语汇描述类别，不把任何外部协议视为本项目必须遵循的最终标准。

## 可执行互联网主张（VISION.EXEC）

“可执行互联网”是 Agentic Web 项目群的战略主张，而不是另起一套传输协议。

```text
信息互联网：
页面 → 阅读 → 人工判断 → 人工操作

可执行互联网：
目标 → Agent 发现对象 → 权限确认 → 执行
→ Evidence → 版本更新 → 后续复用
```

一句话传播表达：

> **从信息互联网走向可执行互联网。**

完整表达：

> **将今天供人阅读和点击的互联网，升级为智能体能够发现、理解、执行、验证并共同改进的网络。**

“可执行”不代表自动执行一切。高风险行为仍必须经过人类明确授权，网络对象必须可预览、可拒绝、可撤销和可审计。

## 可执行项目原则（VISION.EXEC_PROJECT）

Post is Project 的长期目标不止是保存讨论、Patch 和版本，还要允许适合执行的 Project 在受控环境中被真实验证：

```text
Project Revision
→ 选择 Execution Profile
→ 创建短生命周期隔离环境
→ Agent 配置、构建、启动和修复
→ 独立 Evaluator 验证
→ Execution Report / Evidence
→ Project Patch 或 Module Spec Patch
→ 跨项目回放
→ Stable Release
```

这不代表每个 Project 都必须拥有云服务器：

- 文档可以验证构建和链接；
- Library 可以验证包和多 Runtime 兼容；
- Web / API 可以创建临时 Preview；
- Workflow 可以在隔离任务沙箱中运行；
- Infrastructure 可以执行 Plan、Dry Run 和策略检查；
- 硬件项目可以提交模拟器或结构化实验 Evidence。

默认形态是按需创建、到期销毁的 **Executable Validation Environment**。只有真实项目持续需要长期运行，且成本、权限、监控、备份和责任边界已经明确后，才允许将临时环境提升为稳定部署。

详细边界见 `共享架构与协议/03_可执行项目与部署验证.md`。

## 可验证环境交互原则（VISION.INTERACTION）

可执行互联网不能假设所有软件都会提供 Agent API。Vlawd 等 Runtime 需要在必要时操作桌面应用和网页界面，但 Computer Use 必须是结构化接口的补充，而不是首选捷径：

```text
API / MCP / CLI
→ DOM / CDP
→ Accessibility / UI Automation
→ Semantic + Visual
→ 受控 Raw Input
```

每次 UI 动作都应回答：

- 观察来自哪个应用、窗口、页面和状态；
- 元素引用是否仍属于当前状态；
- 执行是否改变前台焦点或用户输入；
- 预期结果如何独立验证；
- 失败、未知和动作前已成立如何区分；
- 哪些截图、状态、确认和结果可以进入 Evidence。

外部项目可以提供架构参考、原生 helper 研究对象或比较基准，但不能自动成为 Agentic Web 的协议或 Vlawd 的核心依赖。详细原则见 `共享架构与协议/04_可验证环境交互与ComputerUse.md`。

## 人类监督原则（VISION.HUMAN）

Agentic Web 项目群不是“所有事情交给 Agent 自治”的项目。人的角色不能被降为异常处理器。

```text
人负责：
目标、价值判断、权限、风险偏好、最终责任

Agent 负责：
检索、规划、执行、比较、记录、提出改进

协议负责：
明确边界、版本、权限、来源和 Evidence
```

不可自动绕过的控制点：

- 身份和隐私授权；
- 支付、提交、删除和公开发布；
- 权限扩大；
- 安全策略改变；
- 高风险 Capsule 的首次运行；
- 公开网络入口、付费资源和长期运行环境；
- 生产数据、数据库迁移、DNS 和域名变更；
- 项目稳定版本的发布。

## 战略飞轮

Agentic Web 项目群的长期价值飞轮是：

```text
自然表达目标
→ Vlawd / Agent Runtime 完成真实任务
→ Session 记录动作、纠正和结果
→ Evaluation 识别成功路径和失败模式
→ 成功经验沉淀为 Capsule / Project
→ 可执行 Project 在干净环境中复现并生成运行 Evidence
→ 个人或网络复用后提高任务成功率
→ 用户更愿意交付真实任务
→ 产生更多高质量 Evidence
```

全双工语音不是独立飞轮，而是让目标、纠正和监督更自然的入口。

## 项目群战略

### Vlawd：个人入口

```text
用户说出目标
→ 看到多个任务状态
→ 随时插话、暂停、取消和接管
→ 结果保存为 Session
```

Vlawd 是 Agentic Web 的个人 Runtime 和监督终端。

### Capsule：窄腰协议

Capsule 将 Runtime、模型和平台解耦。它不保存某个模型的全部思考，而保存可执行能力所需的稳定契约、权限、测试和来源。

### Post is Project：协作层

网络中的核心对象不是帖子热度，而是问题、方法和能力的版本演化。评论只是 Project Event 的一种；真实部署、验证、发布和回滚也应成为可审查的项目事件，但底层环境生命周期由 Runtime 管理。

### 垂直项目：证据来源

Module Spec 和高校知识论坛不是先建设平台后的示例，而是用来证明对象模型是否有价值的真实场景。

## 长期范围边界（VISION.SCOPE）

### Agentic Web 项目群要做

- 可监督的个人 Agent Runtime；
- 状态作用域、语义优先且可验证的环境交互；
- Session 与 Evidence；
- 可移植、可授权、可测试的能力对象；
- 可持续演化的项目协作；
- 可执行项目的隔离验证、运行 Evidence 和稳定发布；
- 多种客户端共享同一 Graph 数据；
- 可验证的第三方知识、能力和服务。

### Agentic Web 项目群不做

- 替代 Windows、macOS 或 Linux；
- 自研通用基础模型；
- 自研互联网传输协议；
- 以区块链或 Token 作为系统成立前提；
- 以传统社交 Feed 作为底层数据模型；
- 无人监督的高风险自动化；
- 把外部 Computer Use Runtime 直接变成系统核心或安全边界；
- 在已有 API、MCP、CLI 或 CDP 时优先使用屏幕坐标；
- 一开始建设开放世界的多 Agent 自治网络；
- 把“一项目一台永久 VM”作为系统成立前提；
- 在缺少真实部署证据时自建通用多云平台；
- 为证明概念而同时维护多个完整产品。

## 一句话总结

**Agentic Web 项目群的长期战略不是打造另一个 AI 应用集合，而是逐步建立 Agentic Web 的个人入口、事实层、能力协议和协作网络，让真实任务产生的成功经验能够被验证、迁移和持续改进。**
