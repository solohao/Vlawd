## 验证原则

本文件只比较三条项目级验证路径。Vlawd 当前 Golden Path 的实现周期、功能编号和验收标准以 `AI_cursor_v2/02_MVP开发规划与阶段路线.md` 为准。

三条路径不是为了分别证明三个产品值得创业，而是从不同角度验证同一个长期系统。

```text
Vlawd：
人和 Agent 是否能自然完成真实任务

Module Spec：
方法是否能被版本化、测试和持续改进

高校论坛：
人类是否愿意围绕 Living Project 发现、贡献和协作
```

任何路径都必须有明确停止条件，不能因已经投入而自动升级为完整产品。

## 路径 A：Vlawd Dogfood（VALID.VLAWD）

### 核心假设

> 用户通过全双工语音监督 AI 完成多变任务，比亲手完成更省，并且 Session 积累会继续降低监督成本。

### 首个场景

```text
真实研究目标
→ 全双工讨论
→ 只读网页研究
→ 用户途中纠正
→ 输出带来源的结果
→ 第二天继续 Session
```

### 指标

- 每周主动使用次数；
- 任务完成率；
- 从目标到有用结果的时间；
- 每任务插话和纠正次数；
- 中断后恢复率；
- 高风险误动作；
- 每任务成本；
- 第二天继续比例；
- 同类任务第 N 次监督成本是否下降。

### 成立信号

- 用户无需强迫自己就会主动使用；
- 至少三个真实任务持续比手工更省；
- 相似任务所需解释逐步减少；
- 失败时能够继续而不是全部重来。

### 停止或缩小信号

- 监督时间长期高于手工操作；
- 全双工延迟和成本无法进入可用范围；
- 用户更愿意直接使用文本 Agent；
- Session 没有产生后续价值；
- 真实执行长期不可靠。

如果语音不成立，可以保留文本入口验证 Runtime 和 Session，不因模态失败否定整个系统。

## 路径 B：Generative Module Spec（VALID.MSPEC）

### 核心假设

> 保存设计判断、约束、失败经验和验收测试的方法规范，可以让 Coding Agent 在不同项目中生成更可靠的适配实现。

### 最小实验

选择一个低风险且边界清晰的模块，例如：

- 结构化日志；
- 健康检查；
- 配置加载；
- 重试与退避；
- 只读审计日志。

分别比较：

```text
Baseline：
只给普通自然语言需求

Treatment：
提供 MODULE.md + Conformance Suite
```

在至少三个不同上下文生成实现。

### 指标

- 首次测试通过率；
- 人工修改次数；
- 安全和边界缺陷数；
- 跨技术栈适配率；
- 规范更新后多个案例是否同时改善；
- 失败能否转化为通用规则或测试。

### 成立信号

- Treatment 稳定优于 Baseline；
- 规范不是为单个测试硬编码；
- 新 Evidence 能推动新版本；
- 其他 Agent 无额外解释即可使用。

### 停止或缩小信号

- MODULE.md 退化为超长 Prompt；
- 维护成本高于重复生成成本；
- 改进只对一个项目有效；
- 自动测试无法表达真正质量；
- 普通文档或库已经足够。

## 路径 C：高校二维知识论坛（VALID.CAMPUS）

### 核心假设

> 学生愿意先与 AI 专家自然讨论，再将结果发布为可协作项目；第二个人的贡献能够让项目产生可见改进。

### 最小实验

```text
一个班级
+ 一个课程主题
+ 一个共享 AI 专家
+ 一个私有空间
+ 对话转项目草稿
+ 一次第二人贡献
```

### 指标

- 对话生成项目草稿的比例；
- 用户审查草稿所需时间；
- 草稿修改比例；
- 发布项目中得到第二人贡献的比例；
- 贡献被接受、保留分支或拒绝的比例；
- 一周后主动查找已有项目的用户数；
- 引用准确率；
- 每十分钟有效讨论的推理成本。

北极星指标：

> **每周产生多少个被第二个人复用或改进的知识项目。**

### 成立信号

- 学生愿意发布经过审查的结构化结果；
- 同学无需完整阅读原始对话即可贡献；
- 图谱比普通帖子更容易表达分支、证据和冲突；
- 社区积累使相似问题更容易解决。

### 停止或缩小信号

- 用户只体验 AI，不愿发布或回来；
- 图谱增加理解负担；
- 第二人贡献非常稀少；
- AI 生成内容大量污染知识库；
- 隐私和运营成本超过验证价值。

## 三条路径对比

| 能力 | Vlawd | Module Spec | 高校论坛 |
|------|------:|------------:|---------:|
| 全双工体验 | 强 | 无 | 强 |
| 本地执行 | 强 | 弱 | 无 |
| Session | 强 | 中 | 中 |
| 自动 Evaluation | 中 | 强 | 弱 |
| Capsule 对象 | 中 | 强 | 中 |
| 画布 / Graph | 中 | 弱 | 强 |
| 人类协作 | 弱 | 中 | 强 |
| 网络效应 | 弱 | 中 | 强 |
| 独立开发适配 | 强 | 强 | 中低 |

## 选择规则

当前默认：

1. 先完成 Vlawd 自我 Dogfood；
2. 再选择一个第二垂直案例；
3. 只有两个案例都需要交换对象，才启动 Capsule v0.1。

选择第二路径时：

```text
如果希望继续独立、低社交验证：
  选 Module Spec

如果 Vlawd 全双工体验已经稳定，需要验证人类协作：
  选高校论坛

如果 Vlawd 自用都未成立：
  不启动任何新路径
```

---

## 外部验证边界

第一阶段允许创始人完成绝大多数工作，但不能用个人偏好证明市场成立。

外部反馈只在固定 Gate 进入：

- Golden Path 已连续稳定；
- 至少三段周期演示；
- 已知限制明确；
- 用户能独立完成同一个任务；
- 只观察事实，不收集无限功能建议。

推荐少量、异步、结构化测试，而不是持续运营式交流。

---

## 一句话总结

**Vlawd 验证任务闭环，Module Spec 验证可执行方法对象，高校论坛验证群体协作；三者按证据顺序串行推进，而不是同时开发。**
