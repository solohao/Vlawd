## 模块定位（ARCH.SANDBOX）

可执行项目验证是 Agentic Web 的跨层能力，不是新的并列产品，也不是要求近期建设通用云平台。

它负责回答：

- 一个 Project 的代码、方法或能力能否在干净环境中真实运行；
- AI 是否能根据仓库和 Module Spec 完成配置、构建、启动和验证；
- 失败来自项目、规范、环境、平台还是外部依赖；
- 运行结果如何形成 Execution Report、Evidence 和 Project Patch；
- 哪些可复现经验达到 Capsule 的可移植标准。

长期闭环：

```text
Project Revision
→ 选择 Execution Profile
→ 创建短生命周期隔离环境
→ Agent 配置、构建、启动和修复
→ 独立 Evaluator 验证
→ Execution Report / Evidence
→ Project Patch 或 Module Spec Patch
→ 跨项目回放
→ Stable Release
```

核心原则：

> **先用临时干净环境证明可复现，再考虑为项目保留长期运行环境。**

---

## 不是所有 Project 都等于服务器

Project 应声明自己的可执行验证方式，不强制拥有云服务器。

| Project 类型 | 推荐验证方式 |
|--------------|-------------|
| 文档或研究 | 构建、链接检查、渲染 Preview |
| Library / SDK | 包构建和多 Runtime 测试矩阵 |
| Web / API | 临时 Preview Environment |
| Worker / Agent Workflow | 隔离任务沙箱和输入输出验证 |
| Stateful Application | 临时服务、临时数据库和迁移演练 |
| Infrastructure | Plan、Dry Run、策略和权限检查 |
| 硬件或线下实验 | 模拟器、测试夹具或结构化实验 Evidence |

通用抽象是 **Executable Validation Environment**，不是“一项目一台永久 VM”。

## 模块边界

### Post is Project 负责

- 保存执行意图、Project Revision 和稳定发布；
- 记录 review、patch、promotion 和 rollback 等语义事件；
- 展示 Execution Report、Evaluator 结果和兼容性；
- 将部署失败转为候选 Project Patch；
- 维护哪个版本在什么环境中被验证。

### Agent Runtime 负责

- 检查仓库和现有部署元数据；
- 生成可审查的 Execution Plan；
- 申请权限、Secrets、网络和预算；
- 调用隔离环境完成配置、构建和启动；
- 支持暂停、取消、重试、接管和恢复；
- 写入 Session 与 Execution Report。

### Environment Broker 负责

- 创建和销毁隔离环境；
- 注入短期最小权限凭据；
- 实施 CPU、内存、磁盘、网络、时长和成本限制；
- 提供日志、制品和健康检查入口；
- 保证不同项目和运行之间的隔离。

### Evaluator 负责

- 独立检查 Build、Start、Health、Golden Test 和回滚结果；
- 不接受执行 Agent 的自然语言自评作为唯一成功依据；
- 生成可关联到 Revision、环境和规范版本的 Evidence。

### Module Spec 负责

- 描述某类模块的上下文问题、约束、变体和失败模式；
- 为不同 Profile 绑定 Conformance Suite；
- 根据多个 Project 的同类失败形成规范 Patch；
- 不吸收只对单个仓库有效的部署特例。

## 逻辑架构

```text
Post is Project / Project View
  ├── Source Revision
  ├── Module Spec Version
  ├── Execution Profile
  ├── Project Events
  ├── Patch / Review / Merge
  └── Evidence / Stable Release
              │
              ▼
Agent Runtime / Deployment Runner
  ├── Repository Inspector
  ├── Plan Generator
  ├── Policy & Approval Gate
  ├── Environment Broker
  ├── Agent Runner
  ├── Independent Evaluator
  └── Evidence Collector
              │
              ▼
Ephemeral Sandbox
  ├── Clean checkout
  ├── Short-lived credentials
  ├── Resource and time limits
  ├── Controlled network
  ├── Build and runtime logs
  ├── Health and Golden Tests
  └── Automatic destruction
```

长期逻辑模块不代表近期必须拆成多个服务。首个实验可以由一个进程、一个 Runner 和一个容器后端完成。

## Execution Environment（ART.ENVIRONMENT）

Execution Environment 分为两类：

- **Environment Declaration**：可复现的环境需求；
- **Environment Instance**：一次具体运行创建的短生命周期实例。

建议最小字段：

```yaml
id:
project_id:
source_revision:
profile:
provider:
isolation:
runtime:
network_policy:
resource_limits:
persistence:
secret_refs: []
created_at:
expires_at:
status:
```

规则：

- `secret_refs` 只保存引用，不保存 Secret 值；
- 默认无持久存储；
- 默认不公开网络端点；
- Instance 到期后必须销毁；
- Evidence 只公开必要的环境指纹，不自动公开私有日志和配置；
- 只有可复现的 Declaration 才可能进入 Capsule Manifest。

## Execution Profile

沿用 Module Spec 的 Profile，但为部署验证补充客观边界。

### Prototype

- 干净环境安装；
- Build；
- Unit / Conformance Tests；
- 不暴露公网；
- 不要求长期运行。

### Internal

- 服务启动；
- Health Check 和 Golden Smoke Test；
- 进程重启后恢复；
- 临时内部访问地址；
- 环境到期自动销毁。

### Production

- 临时公开 Preview；
- HTTPS 和基础监控；
- 数据迁移、备份或回滚演练；
- 权限、资源和成本报告；
- 稳定版本仍需人工批准。

### High Assurance

- 更强隔离；
- 依赖和制品检查；
- 网络白名单；
- 可重复构建和来源证明；
- 灾难恢复测试；
- 独立人工或组织审批。

当前阶段只记录这些长期 Profile。Module Spec Lab 首次部署实验只允许 Prototype，稳定后才进入 Internal。

## Execution Report 扩展

部署验证复用 `ART.REPORT`，不新建一套部署日志对象。建议增加：

```yaml
project_id:
source_revision:
execution_profile:
environment_ref:
resource_usage:
evaluator_results: []
```

完整运行仍应记录：

- Runtime、Agent 和模型；
- 已授予权限；
- 开始、结束和状态；
- 输出和制品；
- 失败类型；
- 延迟和成本；
- Evidence 引用。

原始日志属于 Session 或私有运行记录。Project 只保存用户允许发布的结构化报告和必要 Evidence。

---

## Project Event 扩展

Project Event 只记录对项目语义有影响的里程碑，不复制全部沙箱日志。

建议事件：

- `request_validation`；
- `approve_execution`；
- `attach_execution_report`；
- `propose_runtime_patch`；
- `promote_release`；
- `rollback_release`。

环境创建、进程输出和探针轮询保留在 Runtime / Session，不进入公共 Graph 的主事件流。

## 失败分类

部署失败必须先分类，再决定修改哪个对象：

| failure_type | 含义 | 默认动作 |
|--------------|------|---------|
| `spec_gap` | Module Spec 缺少可复用规则 | 候选规范 Patch，并跨项目回放 |
| `project_defect` | 当前仓库自身代码或配置错误 | Project Patch |
| `environment_incompatible` | OS、架构、Runtime 或依赖不兼容 | Compatibility Evidence |
| `secret_required` | 缺少必须由用户提供的凭证 | 阻塞并请求最小权限 Secret |
| `platform_limit` | 当前 Sandbox 不支持所需能力 | 记录平台限制，不修改规范 |
| `external_failure` | 第三方服务、下载源或网络失败 | 重试或记录外部失败 |
| `policy_blocked` | 权限、网络或操作违反策略 | 请求确认或拒绝执行 |
| `non_reproducible` | 结果依赖隐含状态或不稳定 | 清理环境并重新验证 |

只有 `spec_gap` 可以直接推动 MODULE.md 演化。单项目特例不能自动进入通用规范。

## Module Spec 部署 Conformance（VALID.DEPLOY）

部署成功本身不能证明 MODULE.md 有效，必须保留 Baseline / Treatment 对照：

```text
Baseline：
Agent 只得到普通需求和仓库

Treatment：
Agent 得到同一仓库、MODULE.md 和 Conformance Suite
```

比较：

- 首次构建成功率；
- 首次启动成功率；
- 首次 Golden Test 通过率；
- 部署总时间、计算和 Token 成本；
- 人工确认与修改次数；
- 安全缺陷和未声明权限；
- 干净环境重复运行成功率；
- 回滚成功率。

防止过拟合：

- 使用至少三个不同项目上下文；
- 保留至少一个未参与规范编写的 Holdout Project；
- Evaluator 与执行 Agent 分离；
- 规范更新后回放旧项目；
- 只有改善多个项目且没有引入回归时才升级规范版本。

## 安全、权限与成本边界

第一版至少满足：

- 每次运行使用独立环境；
- 默认无持久磁盘和公网入口；
- 到期或超时自动销毁；
- 限制 CPU、内存、磁盘、运行时长和费用；
- 使用短期、最小权限凭据；
- 禁止访问宿主机、其他项目和云元数据服务；
- 网络默认关闭或使用明确白名单；
- Secrets 只在运行时注入，不写入日志、镜像或 Project；
- 仓库修改进入临时分支或 Patch，不直接写稳定分支；
- 所有操作可审计；
- Evaluator 不与执行 Agent 共享成功判断权。

以下操作必须显式确认：

- 公开网络访问；
- 创建付费或长期资源；
- 使用生产数据；
- 数据库迁移和不可逆写入；
- DNS、域名和证书变更；
- 注入新的 Secret；
- 提升云权限；
- 将临时环境升级为稳定部署。

组织级云管理员凭据不能直接交给 Agent。

## 优先复用现有标准

Repository Inspector 应先识别并使用：

- `Dockerfile`；
- `docker-compose.yml`；
- `.devcontainer/devcontainer.json`；
- `Procfile`；
- Buildpacks / Nixpacks；
- 包管理器脚本；
- 项目已有测试和 Health Check；
- 现有云平台或 CI Preview 配置。

处理顺序：

```text
仓库已有部署配置
→ 直接使用并验证

仓库只有 README 或 scripts
→ Agent 生成候选配置并提交 Project Patch

仓库信息不足
→ 返回结构化阻塞原因，不猜测生产配置
```

只有至少三个真实项目出现同一表达缺口后，才考虑定义新的 Deployment Manifest。

## 分阶段路线

### Gate D0：基础 Conformance 已成立

- MODULE.md + Tests 稳定优于 Baseline；
- 失败可以形成跨项目 Patch；
- 当前模块具有真实运行或部署语义。

### Gate D1：Prototype Sandbox

- 三个低风险、无生产 Secret 的项目；
- 一个隔离执行后端；
- Build、Start、Health 和一个 Golden Test；
- 环境 30–60 分钟内自动销毁；
- 默认不开放公网。

退出证据：

- 同一 Revision 在全新环境可重复运行；
- Treatment 在成功率、修正次数或耗时上优于 Baseline；
- 至少一个 `spec_gap` Patch 改善两个项目；
- 无未声明权限和 Secret 泄露；
- 资源和成本可记录。

### Gate D2：Internal Preview

- 临时访问地址；
- 受控网络；
- 重启和恢复；
- 可选临时数据库；
- 失败和环境过期在 Project View 中可见。

### Gate D3：跨环境可移植

- 同一对象在两个不同执行后端运行；
- 环境要求可预览；
- 不兼容原因结构化；
- Evidence 关联到精确 Revision、Profile 和环境。

该 Gate 为 Capsule Deployment Contract 提供证据。

### Gate D4：稳定部署

- 真实项目持续需要长期运行；
- 成本、监控、备份、恢复和责任边界明确；
- 用户明确将 Preview 提升为 Stable Release；
- 高风险变更仍经过 review 和批准。

只有达到 D4 才考虑长期环境，不把永久 VM 作为每个 Project 的默认资源。

---

## 当前 Not Now

- 为每个 Project 默认创建永久 VM；
- 自建通用多云控制面；
- 开放执行不可信第三方仓库；
- 自动使用组织级云管理员凭据；
- 无确认公开端口、域名或 Preview；
- 自动修改生产数据库；
- 在基本 Module Spec 对照实验成立前建设部署平台；
- 为尚未出现的需求发明完整 Deployment Protocol。

---

## 一句话总结

**Post is Project 保存可执行项目的版本、决策与 Evidence，Agent Runtime 在短生命周期隔离环境中完成真实部署，独立 Evaluator 验证结果；失败通过 Project Patch、Module Spec Patch 和跨项目回放推动对象持续演化。**
